# Custom MCP extensions in Go: what the official SDK lets you do, and what it doesn't

**The official `github.com/modelcontextprotocol/go-sdk` (v1.3.0, January 2026) lets you *declare* arbitrary experimental capabilities, but as of v1.3.0 it does not expose a public API for *sending or receiving* non-spec JSON-RPC methods or notifications.** If your extension — e.g. Claude Code's `notifications/claude/channel` — only requires advertising `capabilities.experimental["your/ext"] = {}` and then riding on top of standard MCP requests, you are well-served. If it requires emitting or handling a bespoke JSON-RPC method name, the SDK has no first-class seam today, and you must either work through a custom `Transport` that wraps the `Connection` or adopt one of several in-flight patches that add a `SendNotification` method. This matters because the SDK is explicitly conservative (v1.x is API-frozen) and deliberately omits the generic `server.notification({method, params})` escape hatch that the TypeScript SDK uses to ship experimental protocols. **However, there is active upstream work on exactly this gap, with a maintainer-designed convention in PR #844 that is likely the fastest path to a stable solution.**

The SDK separates three layers — `Client`/`Server` configuration, `ClientSession`/`ServerSession` per-connection RPC methods, and the underlying `Transport` → `Connection` → `jsonrpc.Message` plumbing — and custom behavior is possible at each, but with very different ergonomics. Middleware (`AddSendingMiddleware`/`AddReceivingMiddleware`) looks superficially like the right tool, but it operates on typed `MethodHandler`s keyed off methods the SDK already knows, not on arbitrary JSON-RPC payloads. The cleanest supported extension points are: (1) capability declaration via `ClientCapabilities.Experimental` / `ServerCapabilities.Experimental` (both `map[string]any`), (2) per-request `_meta` via the `Meta` field, and (3) a custom `Transport` implementation that reads and writes raw messages.

## Capability negotiation: fully supported, and this is the one thing that "just works"

Both `ClientCapabilities` and `ServerCapabilities` in `github.com/modelcontextprotocol/go-sdk/mcp` expose **`Experimental map[string]any`** explicitly intended for "non-standard capabilities" — the doc comment reads *"Experimental reports non-standard capabilities that the client supports. The caller should not modify the map after assigning it."* Since v1.2.0 there is also an **`Extensions map[string]any`** field on `ClientCapabilities`, keyed as `"{vendor-prefix}/{extension-name}"`, with a helper `ClientCapabilities.AddExtension` to normalize nil settings to empty objects.

As of v1.2.0 (released December 22, 2025, per issue #706 "Proposal: Add {Client,Server}Options.Capabilities"), you populate these at construction time via `ServerOptions.Capabilities` and `ClientOptions.Capabilities`. The SDK augments whatever you set with capabilities it infers from registered features — so setting `Experimental: map[string]any{"claude/channel": map[string]any{}}` on a `ServerOptions.Capabilities` will produce an `initialize` result with `capabilities.experimental["claude/channel"] = {}`, which is exactly what Claude Code looks for. The older `ServerOptions.Has{Tools,Prompts,Resources}` booleans are deprecated in favor of this unified `Capabilities` field. After negotiation, the peer's declared capabilities are available via `ClientSession.InitializeResult()` and `ServerSession.InitializeParams()`, so your code can gate behavior on whether the other side opted into your extension.

**This path requires no workarounds.** If your extension is purely a handshake + piggybacks on spec methods, stop here.

## The missing piece: there is no public `Session.Notify(method, params)` on the Go SDK

The `ServerSession` and `ClientSession` types expose only the RPC methods the SDK was written to understand. On `ServerSession` (per v1.3.0 pkg.go.dev) that means **`CreateMessage`, `Elicit`, `ListRoots`, `Log`, `NotifyProgress`, `Ping`, `Close`, `ID`, `InitializeParams`, `Wait`**, plus the server-level `Server.ResourceUpdated`. On `ClientSession` the set is analogous: `CallTool`, `ListTools`, `ReadResource`, `GetPrompt`, `NotifyProgress`, `SetLoggingLevel`, `Subscribe`, `Unsubscribe`, `Complete`, `Ping`, etc. **None of these accept a caller-supplied method name**, and there is no exported `Notify(ctx, method, params)` or `Call(ctx, method, params)` on either session type.

This is a deliberate design choice. The design doc (discussion #364) states plainly: *"Servers here do not have methods for sending notifications or calls, because they are logically distinct from the ServerSession… The Transport interface here is lower-level than that of mcp-go."* Contrast this with the TypeScript SDK, where the Claude Code channels reference code calls `mcp.notification({ method: 'notifications/claude/channel', params: {...} })` directly — that generic method simply does not exist in the Go SDK's public surface. An analogous complaint has been filed for spec notifications (python-sdk #741, "Enable Server-Initiated Notifications to All Connected Clients"), confirming that server-initiated, arbitrary notifications remain a gap across the official SDKs, and the Go SDK inherits the same constraint.

**Receiving custom notifications is equally closed off.** The Go SDK dispatches incoming messages through a `methodInfo` registry (internal `mcp/shared.go`) that maps known method strings to typed unmarshalers and handlers. An arbitrary inbound `notifications/claude/channel` will not find a registered handler and will not reach user code — there's no `ServerOptions.UnknownNotificationHandler` or similar.

## Middleware is not the answer, even though it looks like it

`Server.AddSendingMiddleware`, `Server.AddReceivingMiddleware`, and the matching `Client` methods wrap a `MethodHandler` defined in `mcp/shared.go` as `func(ctx, method, req) (result, error)`. This is the abstraction used for tracing, OTel instrumentation (e.g. `curioswitch/mcp-go-sdk-otel`, `olgasafonova/mcp-otel-go`), and auth. It runs *after* the SDK has dispatched a known method to its typed handler, and the middleware operates on typed `Params` and `Result` values via `Middleware = func(MethodHandler) MethodHandler`. **Receiving middleware will never be invoked for an unknown method name**, because the message never enters the dispatch path in the first place. **Sending middleware only runs when user code calls one of the typed session methods** — you can't "fake" a custom send by invoking middleware, because there is no SDK call site that produces `notifications/claude/channel`.

Middleware is still the right place for cross-cutting concerns on *spec* methods: injecting a `progressToken` into outgoing `CallTool` requests, logging every inbound request, validating OAuth tokens. It is not an extension mechanism for new JSON-RPC methods.

## The real escape hatch: a custom `Transport` that wraps `Connection`

The supported way to inject or intercept arbitrary JSON-RPC traffic is the `Transport` interface. `Transport.Connect(ctx)` returns a `Connection` with **`Read(ctx) (jsonrpc.Message, error)`** and **`Write(ctx, jsonrpc.Message) error`**. The `github.com/modelcontextprotocol/go-sdk/jsonrpc` package is explicitly public for exactly this purpose; the README states *"The github.com/modelcontextprotocol/go-sdk/jsonrpc package is for users implementing their own transports."* `jsonrpc.Message` is an interface implemented by the concrete request/response/notification types, so you can construct a notification with any method name and push it onto the stream.

The pattern is a decorator: take a standard transport (e.g. `&StdioTransport{}` or `&StreamableServerTransport{...}`), call its `Connect`, and return a wrapper `Connection` whose `Read` and `Write` forward to the inner connection except when they see your extension's method strings. There's a precedent for this in the SDK itself — `LoggingTransport` ships as a wrapper that logs every `jsonrpc.Message` it sees. For outbound custom notifications, you expose a sideband channel (a Go `chan jsonrpc.Message` or an API object passed to your business logic) whose messages your wrapper's `Write` loop drains alongside SDK traffic. For inbound custom notifications, your wrapper's `Read` inspects the method, routes `notifications/claude/channel` (or whatever) to your handler, and returns only spec-compliant messages to the SDK core. This works because both `Read` and `Write` are documented as concurrent-safe, and the SDK's JSON-RPC core will simply never see the methods you intercept.

The costs are real. **You sacrifice the type-safety and automatic schema handling the SDK is built around**, you reimplement concerns like request/response ID correlation if your extension needs request/response semantics (not just one-way notifications), and you couple your code to the `jsonrpc.Message` interface shape. But this is the only path the SDK authors have left open for genuinely new protocol methods, and it is consistent with their stated philosophy of keeping the transport "lower-level" than mcp-go's higher-level send-a-notification abstraction.

## A practical recipe for a Claude-Code-channel-style extension in Go

Combine the three mechanisms above in this order. First, **declare the capability**: `mcp.NewServer(impl, &mcp.ServerOptions{Capabilities: &mcp.ServerCapabilities{Experimental: map[string]any{"claude/channel": map[string]any{}}}})`. Second, **implement a wrapping Transport** that embeds `*mcp.StdioTransport` (or whatever base you need), whose `Connect` returns a custom `Connection`. Third, in that connection's `Write` path, read from an app-owned `chan *jsonrpc.Notification` and forward those onto the underlying connection interleaved with SDK writes; in the `Read` path, peek at incoming messages and short-circuit your extension's method names to application-side handlers before returning the rest to the SDK. Construct your custom notification as a `jsonrpc.Message` using the `jsonrpc` package types — the exact concrete type names are `jsonrpc.Request` with a nil ID for notifications, per the WebSocket-transport proposal (#652) and the `Connection` interface signature (`Read(context.Context) (jsonrpc.Message, error)`).

For **two-way channels that also need a reply tool** (the pattern Claude Code uses for Telegram/Discord), the standard `mcp.AddTool` path still works — you just register a normal MCP tool alongside the capability declaration, and the tool runs through the ordinary dispatch. Only the push direction needs the transport wrapper.

## What to watch and where the SDK may grow

Several signals suggest the SDK will get better extension affordances over time but hasn't yet. The v1.0.0 release notes committed to API stability, and subsequent v1.1–v1.3 releases have focused on spec coverage (2025-11-25 support), icons, standardized `jsonrpc.Error` codes, session hijacking prevention, and streamable-HTTP robustness — not extension hooks. Issue #741 on the Python SDK (and analogous Go behavior) shows that even **broadcasting spec-defined notifications to all connected sessions** is unsupported — `Server.Sessions()` returns an `iter.Seq[*ServerSession]` you can range over manually, but there's no `Server.NotifyAll`. If you need broadcast for a custom method, you'll iterate sessions yourself and push into each session's transport-wrapper channel.

The design team has been open about the trade-off: from discussion #364, *"Other SDKs define higher-level transports, with, for example, methods to send a notification or make a call. Those are jsonrpc2 operations on top of the logical stream, and the lower-level interface is easier to implement in most cases."* The consequence for extension authors is that "lower-level" is where you will live, and the transport wrapper is the officially-sanctioned location for it.

## Conclusion

The go-sdk gives you **clean support for declaring** experimental/extension capabilities, **zero first-class support for sending or receiving** arbitrary JSON-RPC methods, and a **well-defined but low-level escape hatch** via custom `Transport`/`Connection` implementations over the public `jsonrpc` package. Middleware is a distractor — it won't intercept unknown methods. Anyone porting Claude Code channel plugins, custom telemetry protocols, or other `capabilities.experimental.*` extensions from TypeScript should budget for writing a connection-wrapping transport and treat `jsonrpc.Message` as part of their API surface. The good news is this pattern is stable, isolated from future spec changes, and composes with all the SDK's built-in transports (stdio, streamable HTTP, SSE, in-memory). The bad news is you are on your own for type safety, message correlation, and cross-cutting concerns like tracing on the extension path. If your extension is important and broadly useful, filing a proposal issue (the SDK uses "Proposal" / "Proposal-Accepted" labels and has an established review process) is the route to getting a higher-level API — but don't wait for it.
