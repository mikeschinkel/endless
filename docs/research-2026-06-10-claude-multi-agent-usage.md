# Single-Session Multi-Agent Orchestration in Claude Code: Coordinating Eight Related Tasks from One Conversation

*(v2 — revised after determining that prompt-level CWD discipline is non-viable; only gates work.)*

## TL;DR

- **One orchestrator session driving sub-agents for all eight tasks is realistic for the *planning* phase, but not for implementation — the binding constraints are (a) sub-agents cannot ask you questions or accept your mid-task input, only the orchestrator can, (b) the orchestrator's context window fills with sub-agent return payloads, and (c) write-capable sub-agents have no per-agent CWD and rely on prompt-level path discipline, which is non-viable.** For your specific shape, a hybrid wins: orchestrate planning and dependency sequencing in one session, but run parallel implementation through Claude Code's *background/agent-view sessions*, each launched with its CWD physically inside its tracker worktree — never eight tmux windows you manually switch between, and never write-capable sub-agents relying on path discipline.
- **On the worktree conflict: do NOT use Claude's `isolation: "worktree"` as-is** — it creates its own ephemeral worktree under `.claude/worktrees/`, branched from your default branch (not your current HEAD), invisible to your tracker, with documented bugs around stale-branch reuse and wrong base refs. **Recommended near-term approach keeps Option 3's core principle** (the tracker creates and owns each worktree via `<tracker> task claim <id>` before any agent touches it), but the worker is a **background session launched with its CWD inside the resulting `.endless/worktrees/e-NNNN/` path** rather than a non-isolated sub-agent prompted to use absolute paths. A **migration path toward the `WorktreeCreate` hook** (which fully replaces Claude's git logic and lets the tracker own worktree creation) remains the future-proof bridge that does not preclude eventual tracker/Claude worktree sharing.
- **The decisive question is your token budget and how planning-heavy the work is.** Anthropic's own data: "agents typically use about 4× more tokens than chat interactions, and multi-agent systems use about 15× more tokens than chats," and they explicitly warn that "most coding tasks involve fewer truly parallelizable tasks than research, and LLM agents are not yet great at coordinating and delegating to other agents in real time."

## Key Findings

1. **The Agent tool (renamed from "Task" in v2.1.63) spawns isolated sub-agents, up to ~10 concurrent, with a hard architectural limit: sub-agents cannot spawn their own sub-agents, and the only channel from orchestrator to sub-agent is the prompt string.** Each sub-agent gets a fresh context window, its own tool allowlist, and its own model; the parent receives only the sub-agent's final message verbatim.
2. **Sub-agents cannot interact with you.** `AskUserQuestion`, `EnterPlanMode`, and `ExitPlanMode` are on the documented list of tools unavailable to sub-agents. A sub-agent that hits ambiguity either guesses, or (in background mode) has the asking tool-call fail silently while it continues. Only the orchestrator (main thread) can pause and ask you questions.
3. **Sub-agents have no per-agent working directory.** A sub-agent inherits the main conversation's CWD; `cd` does not persist between its Bash calls; the only documented alternative working directory is `isolation: "worktree"`. Making a non-isolated sub-agent operate on another path depends entirely on prompt-level discipline (absolute paths, `git -C`) — which, per your own prior experience, is non-viable without a hard gate.
4. **Context economics are the real ceiling, not the concurrency cap.** Per Anthropic Engineering, multi-agent systems use ~15× the tokens of a chat and single agents ~4×; community write-ups cite ~7× for subagent-heavy coding work. Each sub-agent's *return payload* lands in the orchestrator's context.
5. **`isolation: "worktree"` is real but rough, and conflicts structurally with your tracker.** It creates an ephemeral worktree under `.claude/worktrees/<slug>` on a branch like `worktree-agent-<hex>`, branched from your default branch unless `worktree.baseRef: "head"` is set. Open bugs: stale-branch reuse on agentId-prefix collision, forking from `main` instead of the caller's HEAD, and leftover-worktree accumulation.
6. **The `WorktreeCreate` hook is the escape hatch.** It *fully replaces* Claude's default `git worktree add` logic — it receives a JSON payload on stdin (name, cwd, session_id) and must print the worktree's absolute path on stdout. This is the supported mechanism for making Claude's worktree handling defer to an external system.

## Details

### 1. Capabilities summary — what the Agent tool can and cannot do

**Spawning and concurrency.** The `Task` tool was renamed to `Agent` in v2.1.63; `Task(...)` still works as an alias. Each sub-agent is "a named, isolated Claude instance with its own system prompt, its own context window, its own tool access list, and its own permission mode." The practical concurrency ceiling is about 10 simultaneous sub-agents; the orchestrator issues multiple Agent calls in a single message to fan them out. Beyond the parallel slots, additional tasks queue — and in practice Claude runs them in fixed batches (e.g., 4 at a time) rather than true streaming refill. There is no hard `maxParallelAgents` setting today; it is a frequently-requested feature, and CLAUDE.md instructions like "run at most 2 subagents" are reportedly ignored by the model.

**Built-in sub-agents.** Three matter: **Explore** (Haiku, read-only, fast codebase search), **Plan** (read-only, used in plan mode), and **general-purpose** (all tools, for tasks needing both exploration and modification). Explore and Plan skip CLAUDE.md and git status to stay cheap; every other sub-agent loads them.

**Hard limits to design around:**
- **Sub-agents cannot spawn sub-agents.** No nested delegation. A tree deeper than orchestrator→worker must be chained from the main conversation or built with Skills.
- **The prompt is the only inbound channel.** A non-fork sub-agent starts fresh: it does not see your conversation history, the files Claude already read, or prior decisions. Every file path, error message, and constraint must be in the delegation prompt.
- **The final message is the only outbound channel.** The parent receives the sub-agent's final message verbatim (then may summarize it). For larger work products, the established "artifact" pattern has sub-agents write findings/diffs to files and return a lightweight reference, so the orchestrator does not re-absorb everything through chat-style returns.
- **Sub-agents cannot ask you questions.** `AskUserQuestion`, `EnterPlanMode`, and `ExitPlanMode` are explicitly unavailable to sub-agents.

**Working directory.** A sub-agent starts in the main conversation's current working directory; per the official docs, "Within a subagent, `cd` commands do not persist between Bash or PowerShell tool calls and do not affect the main conversation's working directory." The *only* officially documented way to give a sub-agent a different working directory is `isolation: "worktree"`. There is **no per-sub-agent `cwd` parameter** in the Agent tool frontmatter (documented fields: `description`, `prompt`, `tools`, `disallowedTools`, `model`, `permissionMode`, `mcpServers`, `hooks`, `maxTurns`, `skills`, `initialPrompt`, `memory`, `effort`, `background`, `isolation`, `color`). To make a non-isolated sub-agent operate against another absolute path, you must instruct it in the prompt to use absolute paths and `git -C <path>` — pure path discipline, with nothing at the filesystem level preventing writes to the main checkout or another task's worktree. **This is the structural reason write-capable sub-agents are unsuitable for implementation in your setup: the failure mode (a confused sub-agent writing into the sacred main checkout) is exactly the class of error that prior experience shows guidelines do not prevent — only gates do.**

**Failure visibility.** A foreground sub-agent is watchable and its permission prompts pass through to you. In the background, it runs with already-granted permissions and auto-denies anything that would prompt — so a sub-agent needing an un-granted permission "fails silently" and you discover partial results later. The orchestrator notices a stuck/wrong sub-agent at the *return* boundary, not mid-flight; the recommended mitigation is to require each sub-agent to return a structured report (files changed, tests run/results, confidence, unresolved risks) so the orchestrator can detect gaps and re-dispatch.

### 2. Orchestration patterns — named and described

Four recognized patterns apply:

- **Orchestrator–worker (a.k.a. split-and-merge / fan-out-fan-in).** One lead agent decomposes the task, fans out Agent calls to workers in parallel isolated contexts, then merges/synthesizes. This is Anthropic's published multi-agent research architecture: "a multi-agent system with Claude Opus 4 as the lead agent and Claude Sonnet 4 subagents outperformed single-agent Claude Opus 4 by 90.2% on our internal research eval" — at the ~15× token cost noted above, where token usage "explains 80% of the performance variance." The merge step needs a capable orchestrator because it must reconcile conflicting/overlapping outputs.
- **Chained / sequential sub-agents.** "Use the code-reviewer subagent to find issues, then use the optimizer to fix them." The orchestrator passes relevant context from one to the next. This is the pattern for your dependent tasks (docs after CLI).
- **Reviewer-as-fresh-instance (verification pattern).** Anthropic's best-practices doc recommends a sub-agent that reviews a diff against the plan and reports only gaps that affect correctness — "the implementing session receives the gaps directly and can fix them and re-review without you copying findings between windows." This maps directly to your step (4) verification. Read-only reviewers are immune to the CWD-discipline problem: no write capability means no gate needed.
- **Agent Teams (experimental; `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`).** Multiple *independent* sessions coordinating via a shared task list with peer-to-peer messaging, managed by a lead. Heavier (3–4× tokens of a single session per practitioner accounts), recommended only when workers must communicate/cross-check. For your eight loosely-coupled tasks, teams are overkill. Anthropic's own caveat sets the boundary: "some domains that require all agents to share the same context or involve many dependencies between agents are not a good fit for multi-agent systems today."

A fifth parallel primitive — and the load-bearing one in the revised recommendation — is **Agent View / background sessions** (research preview, v2.1.139+; `/bg`, `claude --bg`). These are full independent Claude Code processes, supervised by a per-user daemon that survives terminal close, all surfaced on *one screen* with state rows (Working / Needs input / Ready for review / Completed). This delivers the "one place to coordinate, no tmux window-switching" UX you want — *without* forcing all eight returns into a single orchestrator's context window, *and* with each session launched in its own real CWD, eliminating the path-discipline problem at the process level. Each background session independently consumes quota (ten in parallel burns quota ~10× as fast).

Performance context for the fan-out gains: Anthropic attributes much of the speedup to the fact that "the lead agent spins up 3-5 subagents in parallel rather than serially; (2) the subagents use 3+ tools in parallel," changes that "cut research time by up to 90%."

### 3. Worktree reconciliation — recommended approach

**Confirming/correcting your understanding of Claude's model:**
- ✅ Correct: `isolation: "worktree"` creates a temporary worktree for the sub-agent's lifetime, agent-managed, under `.claude/worktrees/`.
- ⚠️ Lifecycle correction: cleanup is "auto-removed **only if the sub-agent made no changes**; if it made changes, the worktree and its branch are **kept** (returned for review)." A sub-agent that writes code leaves its worktree behind — and these accumulate; users "discover them weeks later" (open issue on leftover visibility). Background/sub-agent worktrees are swept only when older than `cleanupPeriodDays` AND clean.
- ⚠️ Base-ref correction: by default the worktree branches from your **default branch (`origin/HEAD`), not your current HEAD** — there's an open bug where dispatches from a feature branch silently forked from `main`. Setting `worktree.baseRef: "head"` fixes that, but it is a *global* setting (accepts only `"fresh"` or `"head"`, not arbitrary refs).
- ⚠️ Additional risk: a documented bug where `isolation: "worktree"` **silently reuses a stale branch** when an 8-hex agentId prefix collides with a prior session's branch, inheriting that branch's uncommitted files and stash stack.

These behaviors are exactly why `isolation: "worktree"` collides with your tracker: the path is `.claude/worktrees/`-relative not `.endless/worktrees/e-NNNN/`; the branch/base-ref logic is Claude's not the tracker's; the lifecycle (keep-on-change, sweep-on-`cleanupPeriodDays`) is Claude's not the tracker's two-week grace; and your `.endless/worktree.json`/`.lock` metadata never gets created.

**Evaluating your four options:**

- **Option 1 (no isolation; sub-agents work in tracker worktrees via prompted paths).** **Not viable for write work in your setup.** Because there is no per-sub-agent cwd, correctness rests entirely on prompt-level path discipline — absolute paths, `git -C`, "do not touch the main checkout." Prior experience has established that this class of discipline fails in practice; only filesystem-level or hook-level gates work. Option 1 without a mandatory PreToolUse path-enforcement gate is therefore ruled out, and building that gate is real plumbing (per-dispatch task→worktree binding lookup) that exists only to compensate for the missing per-sub-agent CWD.
- **Option 2 (Claude's worktree symlinked into the tracker path).** Fragile. `git -C <symlink>` behavior, keep-on-change lifecycle, and Claude's branch naming all fight the tracker binding; there's a documented bug where atomic writes through a symlinked file *replace the symlink with a regular file*. Not recommended near-term.
- **Option 3 (orchestrator runs `<tracker> task claim <id>` first; tracker creates and owns the worktree; orchestrator handles `worktree land`).** **The tracker-side half of this is recommended and retained.** The tracker creates and owns the worktree (correct path, branch, `.endless/worktree.json`/`.lock` metadata, two-week grace), Claude never touches worktree creation/teardown, `main` stays sacred, and the orchestrator runs the post-work `<tracker> worktree land <id>`. **The agent-side half is revised:** instead of pointing a non-isolated sub-agent at the path (which reduces to Option 1's discipline problem), the worker is a **background session launched with its CWD inside the tracker worktree**. CWD-as-gate replaces prompt-as-guideline: relative paths just work, and writing outside the worktree requires the session to actively construct foreign paths rather than merely drift.
- **Option 4 (serial reuse of one worktree).** Correctly rejected; defeats parallelism.

**The future-proof bridge.** The revised Option 3 does not preclude — and naturally leads to — the **`WorktreeCreate`/`WorktreeRemove` hook** path. That hook *fully replaces* Claude's default `git worktree add`: Claude hands your script a name on stdin and consumes whatever absolute path you print on stdout. You can therefore make `claude --worktree` / `isolation: "worktree"` call `<tracker> task claim <id>` under the hood and return `.endless/worktrees/e-NNNN/`. That is exactly the "both sides see what they expect" shared-worktree future you described, optionally pairing `worktree.symlinkDirectories` to share heavy build/vendor dirs. Community caveat: using `WorktreeCreate` for anything beyond VCS replacement means reproducing Claude's internal branch/base-ref conventions, which are undocumented and can drift — so pin your Claude Code version and re-test the hook on upgrades.

### 4. Recommended setup for your 8-task case

Given the shape (1–5 parallel-safe; 6,7 depend on others; 8 independent; each hours-to-a-day; some shared Go files likely across 1/2/4/5):

**Phase A — Planning, in the single orchestrator session (this is where single-session shines).**
1. Drive all eight design plans from one conversation. Use **plan mode** + `AskUserQuestion` here — both are *orchestrator-only* tools, and your "review the plan" gate (step 2) lives naturally in the main thread. Spawn **Explore** sub-agents (read-only, Haiku, cheap) in parallel to gather per-task context so exploration noise stays out of your context. Read-only sub-agents are safe regardless of CWD: no write capability, no discipline problem.
2. For each task, the orchestrator presents a plan; you approve/revise inline. This is the cleanest part of your target workflow — do it in one session. Planning is read-heavy and parallelizes well at low token cost.

**Phase B — Implementation: background/Agent-View sessions in tracker worktrees (the default, not the fallback).**

For each implementation task, the orchestrator (or you, scripted) runs `<tracker> task claim <id>`, then dispatches a **background session** (`claude --bg` / Agent View) **launched with its CWD inside the resulting `.endless/worktrees/e-NNNN/`**. Each session's prompt contains the approved plan, the file-ownership scope ("work only in this worktree; shared schema/migrations/root config are out of scope unless assigned"), and the required structured completion report (files changed, test command + results, confidence, unresolved risks). Set `worktree.bgIsolation: "none"` (v2.1.143+) so Agent View does not spawn its own `.claude/worktrees/` isolation on top of the tracker's.

- Run tasks 1, 2, 4, 5 in parallel sessions; run **task 3 (PreToolUse hook)** and **task 8 (bug investigation)** alongside since they're independent. Hold **6, 7 (docs)** until 1–5 land.
- Monitor everything from the **one Agent-View screen** — answer "Needs input" rows in place, no tmux switching. This restores the per-task ability to answer questions (which Agent-tool sub-agents structurally lack) while keeping single-pane coordination — and it makes correct paths the *physical default* of each process rather than a behavior you hope the model maintains.
- **Critical scoping note:** tasks 1, 2, 4, 5 are all "pure Go" on the write path and may touch overlapping files (shared schema/event pipeline). Worktrees prevent disk collisions, not logical ones. Treat any shared file as a **single-writer** resource — never let two parallel sessions edit it. If two tasks genuinely need the same file, sequence them. File-level races and logically-incompatible-but-cleanly-merging changes are the #1 source of parallel-coding failures.

**Why not single-session sub-agents (the former B1)?** Write-capable, non-isolated sub-agents inherit the main checkout's CWD and rely on prompt-level path discipline to stay inside their assigned worktree. That discipline has been established as non-viable — only gates work — and the Agent tool offers no per-sub-agent CWD gate. B1 only re-enters the picture if you build a **mandatory PreToolUse path-enforcement hook** that rejects any Edit/Write/Bash targeting a path outside the dispatching sub-agent's bound worktree (lookup via `.endless/worktree.json` or equivalent binding). That is a prerequisite, not a hardening step — and given the tracker already enforces intent pre-decision via hooks, it may be infrastructure worth building anyway; until it exists, B1 is off the table.

**Phase C — Verification, back in the orchestrator session.** For each landed task, spawn a fresh **reviewer sub-agent** (read-only: Read, Grep, Glob, Bash) that diffs the worktree against the plan and reports only correctness/requirement gaps. Read-only means no gate required. Then run `<tracker> worktree land <id>` from the orchestrator.

**Working directories, summarized:** every implementation session's CWD *is* its tracker-managed `.endless/worktrees/e-NNNN/`; the orchestrator session runs from the main checkout (read/coordinate only); Claude's own `.claude/worktrees/` is **not used**.

### 5. Likely failure modes — what to watch

- **Orchestrator context exhaustion (Phase A/C).** Eight plans plus reviewer returns can push the orchestrator past the ~2/3-full point where quality degrades, triggering auto-compaction (~95% by default). Mitigation: artifact pattern (plans and review findings written to files, references returned); keep returns to structured summaries.
- **Silent failure on permissions in background sessions.** A background session needing an un-granted permission auto-denies; pre-grant the worker toolset, and treat an incomplete completion report as a signal to attach to the session and inspect.
- **File-level races / logical merge conflicts** among tasks 1, 2, 4, 5. Worktrees prevent *disk* collisions but not *logical* ones. Mitigation: single-writer rule on shared files; sequence genuinely-coupled tasks; final integration pass before docs (6, 7).
- **`isolation: "worktree"` stale-branch reuse / wrong base ref** — avoided entirely by not using it. If you ever enable it (e.g., post-`WorktreeCreate`-hook), set `worktree.baseRef: "head"` and watch for leftover worktrees.
- **Path drift in any write-capable sub-agent** — the failure mode that rules out B1. If a future need forces write-capable sub-agents, the PreToolUse path gate is mandatory before first dispatch, not after first incident.
- **Background-session sprawl.** Eight-plus parallel sessions burn quota proportionally and each is a process to supervise. If the initiative's interactive load is genuinely low after planning, you can run fewer sessions at a time in dependency order — Agent View makes the queue visible either way.

### 7. One-paragraph recommendation

**Do this:** Run *planning and review* for all eight tasks in one orchestrator session — that part of your target workflow is fully realistic and is where single-session coordination clearly beats eight tmux windows; use plan mode + `AskUserQuestion` to front-load every design decision while you still control the main thread, with cheap parallel Explore sub-agents for context. For *implementation*, dispatch each task as a **background session (Agent View)** launched with its CWD inside its tracker-managed worktree — the orchestrator runs `task claim` first, the session starts in `.endless/worktrees/e-NNNN/`, and correct paths are the physical default rather than a behavior you hope Claude maintains. **Don't** use non-isolated sub-agents for implementation: they inherit the main checkout's CWD and rely on prompt-level path discipline, which prior experience has established is non-viable — only gates work, and the Agent tool has no gate unless you build a PreToolUse path-enforcement hook as a hard prerequisite. **Don't** use Claude's `isolation: "worktree"` either; it fights your tracker on path, branch, base-ref, lifecycle, and metadata. Keep *verification* in the orchestrator session via read-only reviewer sub-agents (Read/Grep/Glob only — no write capability means no discipline problem), then land each worktree from the orchestrator. You still get one screen and zero tmux-window-switching via Agent View's "Needs input" rows, and each implementation task retains the ability to ask you questions — which Agent-tool sub-agents structurally cannot. Adopt the `WorktreeCreate` hook later as the bridge to true tracker/Claude worktree sharing.

## Recommendations

1. **Now:** Single orchestrator session for Phase A (plan + your review) and Phase C (read-only reviewer sub-agents). For Phase B, `task claim` each child, then launch a background session with CWD inside the tracker worktree; set `worktree.bgIsolation: "none"`. Monitor from Agent View.
2. **If you later want write-capable sub-agents in the orchestrator session:** build the PreToolUse path-enforcement hook first (reject Edit/Write/Bash outside the dispatching sub-agent's bound worktree, binding resolved via `.endless/worktree.json`). It is a prerequisite, not a hardening step.
3. **Never:** rely on `isolation: "worktree"` with this tracker until you've wrapped it with a `WorktreeCreate` hook; never let two parallel workers edit the same Go file (single-writer rule); never assume a sub-agent will ask you something — it can't.
4. **v2 / future:** Implement the `WorktreeCreate`/`WorktreeRemove` hook so `<tracker> task claim`/`worktree land` back Claude's native worktree commands; pin Claude Code version and re-test the hook on each upgrade.

## Caveats

- **Agent View / background agents and Agent Teams are research-preview/experimental** features (v2.1.139+ and `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` respectively); interfaces and exact behaviors may change. Background sessions are local-only and stop on machine sleep/shutdown (recoverable via respawn).
- **The token multipliers (4×/15×) are Anthropic's research-system figures for breadth-first research**, not coding-specific measurements; Anthropic explicitly notes coding has fewer parallelizable tasks. Treat them as order-of-magnitude, not precise predictions for your eight Go tasks. The ~7× figure is a community estimate for subagent-heavy coding.
- **The "no per-sub-agent cwd" finding is established from documented constraints + practitioner reports, not a single official "do it this way" doc.** The `WorktreeCreate` hook is documented but officially framed as a non-git-VCS adapter; using it as a tracker bridge is a supported-but-off-label use the community has validated.
- **Several `isolation: "worktree"` bugs cited are open GitHub issues**, not guaranteed-permanent behavior — they may be fixed in versions after this writing (June 2026). Re-verify against your installed version.
- **Concurrency specifics** (the ~10 cap, batch-of-4 refill behavior, soft "thread limit reached" around 5–7 on consumer plans) are community-observed and version-dependent, not a published hard contract.
