# Custom MCP extensions in Go: what the official SDK lets you do, what it doesn't, and what's in flight upstream

**The official `github.com/modelcontextprotocol/go-sdk` (v1.3.0, January 2026) lets you *declare* arbitrary experimental capabilities, but as of v1.3.0 it does not expose a public API for *sending or receiving* non-spec JSON-RPC methods or notifications.** If your extension — e.g. Claude Code's `notifications/claude/channel` — only requires advertising `capabilities.experimental["your/ext"] = {}` and riding on top of standard MCP requests, you are well-served by the released API. If it requires emitting or handling a bespoke JSON-RPC method name, the released SDK has no first-class seam today.

However, the gap is actively being worked on upstream, and there is already a PR (#844) that implements the convention the maintainer wants. The practical situation is less "the SDK makes this impossible" and more "the feature is mid-flight and the community needs one more push to land it." The recommendation at the end of this document reflects that reality: the right move for an Endless-style channel plugin is probably to help the in-flight work across the finish line while using a fork short-term, not to write a parallel implementation from scratch.

The SDK separates three layers — `Client`/`Server` configuration, `ClientSession`/`ServerSession` per-connection RPC methods, and the underlying `Transport` → `Connection` → `jsonrpc.Message` plumbing — and custom behavior is possible at each, but with very different ergonomics. Middleware (`AddSendingMiddleware`/`AddReceivingMiddleware`) looks superficially like the right tool, but it operates on typed `MethodHandler`s keyed off methods the SDK already knows, not on arbitrary JSON-RPC payloads. The cleanest extension points that exist in the *released* SDK are: (1) capability declaration via `ClientCapabilities.Experimental` / `ServerCapabilities.Experimental` (both `map[string]any`), (2) per-request `_meta` via the `Meta` field, and (3) a custom `Transport` implementation that reads and writes raw messages. The in-flight work (#745 design, #844 and #898 implementations) adds a fourth: a public `SendNotification(ctx, method, params)` method on `ClientSession` and `ServerSession` that reuses the existing middleware pipeline.

## Capability negotiation: fully supported today

Both `ClientCapabilities` and `ServerCapabilities` in `github.com/modelcontextprotocol/go-sdk/mcp` expose **`Experimental map[string]any`** explicitly intended for "non-standard capabilities." The doc comment on the client side reads: *"Experimental reports non-standard capabilities that the client supports. The caller should not modify the map after assigning it."* Since v1.2.0 there is also an **`Extensions map[string]any`** field on `ClientCapabilities`, keyed as `"{vendor-prefix}/{extension-name}"`, with a helper `ClientCapabilities.AddExtension` that normalizes nil settings to empty objects.

As of v1.2.0 (released December 22, 2025, per issue #706), you populate these at construction time via `ServerOptions.Capabilities` and `ClientOptions.Capabilities`. The SDK augments whatever you set with capabilities it infers from registered features — so setting `Experimental: map[string]any{"claude/channel": map[string]any{}}` on `ServerOptions.Capabilities` produces an `initialize` result with `capabilities.experimental["claude/channel"] = {}`, which is exactly what Claude Code looks for. The older `ServerOptions.Has{Tools,Prompts,Resources}` booleans are deprecated in favor of this unified `Capabilities` field. After negotiation, the peer's declared capabilities are available via `ClientSession.InitializeResult()` and `ServerSession.InitializeParams()`, so your code can gate behavior on whether the other side opted into your extension.

**This path requires no workarounds and no fork.** If your extension is purely a handshake plus piggybacking on spec methods (tools, prompts, resources), stop here — the released SDK already does what you need.

## The gap: no public method for sending or receiving custom JSON-RPC notifications

The `ServerSession` and `ClientSession` types in the released SDK expose only the RPC methods the SDK was written to understand. On `ServerSession` (per v1.3.0 pkg.go.dev) that means `CreateMessage`, `Elicit`, `ListRoots`, `Log`, `NotifyProgress`, `Ping`, `Close`, `ID`, `InitializeParams`, `Wait`, plus the server-level `Server.ResourceUpdated`. On `ClientSession` the set is analogous: `CallTool`, `ListTools`, `ReadResource`, `GetPrompt`, `NotifyProgress`, `SetLoggingLevel`, `Subscribe`, `Unsubscribe`, `Complete`, `Ping`, and similar. **None of these accept a caller-supplied method name**, and there is no exported `Notify(ctx, method, params)` or `Call(ctx, method, params)` on either session type in the released code.

This is a deliberate design choice. The design doc (discussion #364) states plainly: *"Servers here do not have methods for sending notifications or calls, because they are logically distinct from the ServerSession… The Transport interface here is lower-level than that of mcp-go."* Contrast this with the TypeScript SDK, where the Claude Code channels reference code calls `mcp.notification({ method: 'notifications/claude/channel', params: {...} })` directly — that generic method simply does not exist in the released Go SDK's public surface. A parallel gap exists on the Python side (python-sdk #741, "Enable Server-Initiated Notifications to All Connected Clients"), confirming that server-initiated arbitrary notifications are a cross-SDK shortcoming, and the Go SDK inherits the same constraint.

**Receiving custom notifications is equally closed off.** The Go SDK dispatches incoming messages through a `methodInfo` registry (internal `mcp/shared.go`) that maps known method strings to typed unmarshalers and handlers. An arbitrary inbound `notifications/claude/channel` will not find a registered handler and will not reach user code — there's no `ServerOptions.UnknownNotificationHandler` or equivalent.

In issue #745, the reporter (working on Gemini CLI's IDE companion integration, which needs `ide/diffAccepted`, `ide/diffRejected`, and `ide/contextUpdate` notifications) describes the concrete blocker: the `conn` field on `ServerSession` is unexported, and the `mcp.Connection` interface's `Write` method requires `*mcp.JSONRPCMessage`, which is itself unexported. The reporter resorted to `unsafe` and `reflect` to reach the unexported `conn` field. That's the clearest statement of how tightly the released SDK closes off custom message traffic.

## Issue #745 and the maintainer-designed convention

Issue #745 ("Expose generic `SendNotification` on `ServerSession` for custom protocol extensions," opened January 2026, labeled "needs investigation") is the canonical tracking issue for this gap. The discussion went further than a simple "add a method" proposal and effectively ratified a specific design convention that any acceptable implementation needs to follow.

The maintainer jba (Go team member, author of much of the underlying jsonrpc2 machinery inherited from gopls) confirmed the use case is valid: *"While the MCP spec does not explicitly allow custom notifications, it doesn't explicitly exclude them either… So I recommend we do this."* But he flagged three complications that make this more than a one-liner.

First, all other notifications in the SDK flow through sending middleware, which performs accounting, logging, PII scrubbing, and other cross-cutting work. Custom notifications should do the same — skipping middleware for one class of messages creates observability gaps. Second, the SDK's internal sending mechanism recognizes something as a notification by checking whether its method name starts with `notifications/`, and it expects every method name it sees to be known. A custom method like `ide/diffAccepted` or `notifications/claude/channel/permission` doesn't reliably hit that path. Third, a server may support many sessions, so the question of whether a custom notification targets one session or broadcasts to all needs a general answer even if early use cases only have one session.

After some back-and-forth with findleyr (the other primary maintainer), who argued for simply exposing a method without a registration mechanism, jba proposed a concrete convention that resolves the routing question without a registry: *"if `method` begins with the segment `x-notifications`, that segment is removed and the remainder is sent as a notification. We would skip everything else in the function until L107."* The `x-` prefix signals "custom extension" to the middleware, the stripping rule means the wire-format method name is preserved exactly as the spec or extension defines it, and the whole thing stays within the middleware pipeline.

This is the design that any merge-bound PR needs to implement. It is not a finalized "ready for work" label on the issue yet — maciej-kisiel explicitly noted that in reply to a contributor who tried to implement it without ratification — but it is the specific technical direction the maintainers have agreed on in public, and there is no competing alternative on the table.

Multi-session broadcast (jba's third point) is the one genuinely unresolved design question. Neither of the existing PRs addresses it, and the maintainers haven't rendered a verdict on whether it should be solved in the first implementation or scoped to a follow-up.

## The two in-flight PRs

Two contributors have attempted implementations. The architectural gap between them is significant, and understanding it matters for anyone trying to push this forward.

**PR #844** ("feat: implement generic SendNotification method for Issue #745," submitted by ajuijas on March 15, 2026, open) implements exactly the `x-notifications/` convention jba proposed. It adds `SendNotification(ctx context.Context, method string, params any) error` symmetrically on both `ClientSession` and `ServerSession`. The implementation wraps arbitrary payloads in a new `customNotificationParams` type that satisfies the SDK's internal `Params` interface (it has `GetMeta`, `SetMeta`, `isParams`, plus a `MarshalJSON` that delegates to the wrapped value), prepends `x-notifications/` to the method name, and routes through the existing `handleNotify` so the call goes through middleware. A short-circuit in `defaultSendingMethodHandler` detects the `x-notifications/` prefix, strips it, and calls `getConn().Notify()` with the real method name. Tests cover both directions — a client sending `custom/myNotif` and a server sending `ide/diffAccepted`. This is the PR that matches jba's design.

Interestingly, ajuijas stated upfront: *"I don't have a personal use case for this. I was just bored and looking for open source issues to help with… Please let me know if you decide to move forward with the implementation!"* That is an explicit invitation to hand the PR off, which substantially affects the social calculus for anyone with a real use case.

**PR #898** ("feat(mcp): add ServerSession.Notify for custom notification methods," submitted by andylbrummer on April 15, 2026, open) predates gh output gathering by three days as of this writing. It is motivated directly by Claude Code channels (the PR description names `notifications/claude/channel`). It adds a single `Notify(ctx, method, params)` method on `ServerSession` that checks for the `notifications/` prefix and then calls `ss.getConn().Notify()` directly, bypassing the middleware pipeline. It is server-only — no symmetric client-side method. jba's terse review — *"Please look at #745 and make sure you're solving the problem there. Also refer to that issue in the commit message."* followed by *"(No, you're not.)"* — is not a rejection of the concept; it's a specific complaint that the implementation skips middleware and doesn't implement the `x-notifications/` convention. The author has been silent for three days. The PR is structurally behind #844 and would need a substantial rewrite to align with the design.

One other related-but-closed PR is worth knowing about: **#364** ("make mcp.handleNotify public," by slimslenderslacks, closed August 2025) tried to solve the same problem by simply exporting `handleNotify`. It was closed without merge — presumably because exporting an internal function with opaque semantics isn't the right abstraction — but its existence shows this gap has been pressing on contributors since early in the SDK's life.

## Middleware is not a standalone answer

`Server.AddSendingMiddleware`, `Server.AddReceivingMiddleware`, and the matching `Client` methods wrap a `MethodHandler` defined in `mcp/shared.go` as `func(ctx, method, req) (result, error)`. This is the abstraction used for tracing, OTel instrumentation, and auth. It runs *after* the SDK has dispatched a known method to its typed handler, and operates on typed `Params` and `Result` values via `Middleware = func(MethodHandler) MethodHandler`. **Receiving middleware will never be invoked for an unknown method name**, because the message never enters the dispatch path in the first place. **Sending middleware only runs when user code calls one of the typed session methods** — you can't "fake" a custom send by invoking middleware, because there is no SDK call site that produces `notifications/claude/channel` today.

What #844 shows is that middleware *can* carry arbitrary notifications if the sending side cooperates — `customNotificationParams` satisfies the typed interface by wrapping the untyped payload, and the `x-notifications/` prefix routes past the method-registry check. Middleware remains the right place for cross-cutting concerns on spec methods (injecting a `progressToken` into outgoing `CallTool` requests, logging every inbound request, validating OAuth tokens), and with #844's design it also carries custom notifications. It is still not an independent extension mechanism: until #844 or an equivalent lands, middleware alone cannot inject a new method.

## The custom Transport escape hatch: possible today, with caveats

The supported way to inject or intercept arbitrary JSON-RPC traffic in the released SDK is the `Transport` interface. `Transport.Connect(ctx)` returns a `Connection` with `Read(ctx) (jsonrpc.Message, error)` and `Write(ctx, jsonrpc.Message) error`. The `github.com/modelcontextprotocol/go-sdk/jsonrpc` package is explicitly public for exactly this purpose; the README states *"The github.com/modelcontextprotocol/go-sdk/jsonrpc package is for users implementing their own transports."*

The caveat I missed in the first draft of this report: `jsonrpc.Message` is the interface you can work with, but the concrete `mcp.JSONRPCMessage` value that the SDK constructs internally for its own messages is unexported. You can construct `jsonrpc.Request` and `jsonrpc.Notification` via the public `jsonrpc` package — that's what the #745 reporter initially tried — but weaving those into the SDK's own write path without going through a transport wrapper requires either the unexported API or `unsafe`/`reflect`. The pattern that cleanly works today is a decorator transport: take a standard transport (e.g. `&StdioTransport{}`), call its `Connect`, and return a wrapper `Connection` whose `Read` and `Write` forward to the inner connection except when they see your extension's method strings. The `LoggingTransport` that ships with the SDK is an existing in-repo example of this decorator pattern. For outbound custom notifications, you expose a sideband channel (a Go `chan jsonrpc.Message` or an API object passed to your business logic) whose messages your wrapper's `Write` loop drains alongside SDK traffic. For inbound custom notifications, your wrapper's `Read` inspects the method, routes `notifications/claude/channel` (or whatever) to your handler, and returns only spec-compliant messages to the SDK core.

The costs are real. You sacrifice the type safety the SDK is built around for your custom messages, you reimplement request/response ID correlation if your extension needs request/response semantics (not just one-way notifications), and you couple your code to the `jsonrpc.Message` interface shape. But this works today on an unmodified SDK. It is consistent with the maintainers' stated philosophy of keeping the transport "lower-level" than mcp-go's higher-level send-a-notification abstraction, and it is also what you would fall back to if #844 stalled indefinitely or the maintainers decided not to add `SendNotification` at all.

## A practical recipe for a Claude Code channel plugin in Go

The fastest working implementation today combines three mechanisms. First, **declare the capability** using the fully-supported released API: `mcp.NewServer(impl, &mcp.ServerOptions{Capabilities: &mcp.ServerCapabilities{Experimental: map[string]any{"claude/channel": map[string]any{}}}})`. Second, **register reply tools normally** via `mcp.AddTool` — this path is standard MCP and uses the released, type-safe API. Third, for the push direction (sending `notifications/claude/channel`) and for permission relay (receiving `notifications/claude/channel/permission_request`, sending `notifications/claude/channel/permission`), choose one of the approaches below based on how much scaffolding you want to maintain long-term.

For custom notifications, there are three viable paths. The first is to **run against a fork that has #844 applied**, using a `go.mod` replace directive pointing at the fork. This gives you the clean `ss.SendNotification(ctx, "notifications/claude/channel", params)` API, routes through middleware correctly, and is forward-compatible with whatever lands upstream. The swap cost when upstream merges is one `go.mod` edit.

The second is the **transport-wrapper pattern** described above. This works on stock v1.3.0 without any fork, but you'll write more code — a decorator `Transport` and `Connection`, a sideband channel for outbound custom notifications, and explicit dispatch logic for inbound ones. This is the right answer if you can't tolerate a fork dependency for policy reasons.

The third is the **hybrid approach**: use the SDK's standard APIs for tools and the capability declaration, and drop to raw JSON-RPC via the `jsonrpc` package only for the channel-specific messages, threading them through a minimal custom transport that passes everything else through untouched. This is slightly less work than a full custom transport because you're only handling the custom method names, but conceptually it's the same pattern.

For all three paths, the capability declaration and the reply tools are unchanged. Only the notification layer differs.

## Recommendation

The original draft of this report framed this as a choice between six options (use the SDK as-is, hybrid, fork with PR, fork standalone, raw JSON-RPC, or TypeScript). With the #745 design discussion and #844 in view, the decision tree collapses substantially.

**The short-term recommendation is to fork `modelcontextprotocol/go-sdk` directly off `main`, cherry-pick the #844 commit, and use a `go.mod` replace directive in the Endless channel plugin to depend on your fork.** This gives you a working implementation today, on the design the maintainer has publicly endorsed, with a clean exit when upstream merges. Forking upstream (rather than forking `standardbeagle/go-sdk` or any other downstream fork) keeps you with one parent to track and one authorship story. Cherry-picking #844 (rather than #898 or a hand-rewrite) aligns you with jba's middleware and `x-notifications/` convention, so when upstream merges something it will almost certainly be close to what you're already running.

**The medium-term recommendation is to help drive #844 across the finish line.** This is where it pays off to be a user with a concrete need, because the maintainers have a design but the PR is unmanned — ajuijas, the author, has publicly stated they don't have a use case and would welcome a handoff. A comment on #844 offering to collaborate or take over, combined with a direct question to jba and findleyr about what would move #745 from "needs investigation" to "ready for work" (the multi-session broadcast question is the concrete design point still outstanding), is cheap to write and high-leverage. The Claude Code channels use case strengthens the case for landing this; jba has already acknowledged Gemini's IDE companion as a legitimate use case, and a second real use case from the Anthropic ecosystem adds weight.

If #844 stalls despite reasonable community effort — a month or two of silence from maintainers after clear engagement — the long-term fallback is Option B from the original report: the pure transport-wrapper approach on stock SDK, permanently. This is more code than using a fork, but it avoids indefinite fork maintenance and keeps you on released versions forever. It is not the preferred path because it duplicates work that upstream is already doing, but it is a safe landing spot if upstream proves unwilling to merge.

**TypeScript is not necessary.** Every path above keeps you in Go. The claim in the original report that the Go SDK "deliberately omits" a generic notification method is still accurate for released code, but the claim that forking is a significant undertaking is wrong in light of #844 — the code is written, the maintainer's design is public, and the effort ahead is social (coordinating with ajuijas, getting maintainer ratification) rather than architectural.

## Revised scope estimates for the fork-plus-PR path

With #844 as the reference diff, the concrete work ahead is much smaller than my original report estimated. The PR touches four files: `mcp/client.go` (adds 8 lines), `mcp/server.go` (adds 8 lines plus a `strings` import), `mcp/shared.go` (adds a 6-line short-circuit in `defaultSendingMethodHandler` plus the roughly 20-line `customNotificationParams` type), and two test files for coverage. That's on the order of 100 lines including tests. There is no architectural rewrite, no new abstraction layer — it's additive exposure of functionality that already exists internally, routed through a convention the maintainer designed.

The remaining work to get this merged is almost entirely non-code: write a PR description that references #745 and explains the Claude Code channels use case, address the multi-session broadcast question one way or another (either solve it or explicitly scope it out with maintainer acknowledgement), add any test coverage the maintainers ask for, and be responsive during review. A contributor picking up ajuijas's work can reasonably expect this to land inside a week or two of active engagement, assuming maintainer bandwidth.

If the multi-session broadcast question proves controversial, one conservative approach is to implement `SendNotification` on `ServerSession` as per-session only, and add a separate helper like `(*Server).NotifyAll(ctx, method, params)` that iterates over `Server.Sessions()` and calls each session's `SendNotification`. This matches how other SDKs structure broadcast (the Python SDK's #741 discussion points in the same direction) and lets per-session and broadcast semantics be argued about separately. But that's a design judgment for the PR discussion, not a blocker.

## Conclusion

The Go SDK cleanly supports declaring experimental capabilities, has no released support for sending or receiving arbitrary JSON-RPC notifications, and has a maintainer-endorsed design for closing that gap sitting in PR #844 with a willing handoff from the author. Anyone building a Claude Code channel plugin in Go today should fork upstream with #844 cherry-picked, use a `go.mod` replace directive, and put modest effort into helping #844 land — that is the highest-leverage combination of short-term working code and long-term upstream alignment. Middleware is a distractor for extension authors until #844 or equivalent lands; the custom-transport escape hatch is a permanent Plan B if upstream proves unwilling. TypeScript is not needed at any point.

The broader lesson for anyone porting Claude Code channel plugins, custom telemetry protocols, or other `capabilities.experimental.*` extensions from TypeScript: the Go SDK's current shape reflects a deliberate philosophy (*"the Transport interface here is lower-level than that of mcp-go, but serves a similar purpose"*), and the maintainers are engaged but cautious about adding higher-level generic APIs. The route to change is through the community process — file or join an issue, converge on a design, submit a PR that matches it, and be prepared to iterate. That process is alive and well on #745, and your use case strengthens it.
