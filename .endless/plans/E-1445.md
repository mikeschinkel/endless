# E-1445: Auto-create worktree on `task update --text` only when task is claimed

## Context

`endless task update --text` today auto-creates a worktree (and a per-worktree sandbox under `~/.cache/endless/sandboxes/`) whenever no worktree exists for the task. The trigger is `_worktree_for_task(task_id) is None` — purely a "does a worktree exist yet?" check, with no consideration of whether the caller intends to start working.

Surfaced during E-1444's plan attach: filing a ticket for triage and attaching its long-form plan provisioned an unwanted worktree + sandbox the user had no intention of working in. The `--no-create-worktree` flag exists as an escape hatch but defaults backwards — the unsafe behavior is the default; the safe behavior requires explicit opt-out.

The cost asymmetry favors flipping:
- **Cost of forgetting `--create-worktree` when intending to work:** zero — `endless task claim E-NNNN` is the natural next step and it creates the worktree.
- **Cost of forgetting `--no-create-worktree` when just filing:** an orphaned worktree + sandbox per filed task. Cleanup requires `endless worktree drop --force` and possibly `endless-sandbox destroy`.

## Approach (decided 2026-05-21 with Mike)

Change the auto-create trigger in `write_plan_to_worktree` (or wherever `task update --text` decides to create) from "no worktree exists" to a **smart heuristic** that infers caller intent from task state.

A task is "claimed" — and therefore a worktree is wanted — iff **`tasks.status == 'in_progress'`** at the moment `task update --text` runs. Use `status` as the single primary signal, not `sessions.active_task_id` binding:
- Active session binding implies `in_progress` already (claim flips status and binds atomically).
- `status='in_progress'` survives a session ending — a task you started, then closed the pane on, is still "in flight" and a worktree is still wanted on the next plan attach.
- `--force` claims without a session binding still flip status to `in_progress`, so the no-binding case is covered without a separate check.

### Behavior matrix

| `tasks.status` | Worktree exists? | Result |
|----------------|------------------|--------|
| `needs_plan`, `ready`, `blocked`, `revisit`, `later`, `maybe` | no  | **DO NOT auto-create.** Write plan to main checkout's `.endless/plans/E-NNNN.md`; sync to DB; leave worktree unprovisioned. |
| `needs_plan`, `ready`, … | yes (manual `git worktree add` or stale) | Use existing — `_worktree_for_task` resolves it. |
| `in_progress`            | no  | Auto-create (today's behavior). User signaled intent via earlier `task claim`. |
| `in_progress`            | yes | Use existing. |
| `verify`, `confirmed`, `assumed`, `completed`, `declined`, `obsolete` | no  | **DO NOT auto-create.** Done-ish task; if user re-attaches a plan it's for record-keeping, not new work. Write to main only. |

Override flags remain:
- `--create-worktree` forces creation regardless of status (for the "I'm about to claim, save me the round-trip" case).
- `--no-create-worktree` forces skip regardless of status (today's escape hatch; behavior unchanged, but rarely needed once the default is smart).

### Main-checkout plan storage

When no worktree exists and the heuristic says "don't auto-create," the plan file goes to `<main>/.endless/plans/E-NNNN.md`. This is a known path the next `endless task claim E-NNNN` already consults when creating the worktree — claim's `create_task_worktree` should copy from main into the new worktree if a plan file exists there. This is partially the existing behavior; verify it during implementation.

### Considered alternatives (rejected)

- **Flip default globally (`--no-create-worktree` becomes the default).** Rejected: makes the `claim` → `update --text` flow require an extra flag for the common case, defeating the convenience the auto-create was supposed to provide.
- **Per-project config toggle (`task_update_create_worktree: bool`).** Rejected: overkill for a single binary question; adds surface area for marginal flexibility.

## Implementation sites

- `src/endless/task_cmd.py` — `write_plan_to_worktree` (around line 425, near `_worktree_for_task` lookup). Replace the `wt_path is None → create_task_worktree` branch with a status check; bypass creation when status is not `in_progress`.
- `src/endless/task_cmd.py` — wherever `--text` is plumbed in (search `allow_create_worktree`). The flag semantics gain a third state implicitly (default = heuristic; `--create-worktree` = force-on; `--no-create-worktree` = force-off).
- `src/endless/cli.py` — `task update` option help text: rephrase `--no-create-worktree` to reflect it's overriding a heuristic, and add `--create-worktree` if it doesn't already exist.
- `src/endless/worktree_cmd.py` — `create_task_worktree` (used by claim path) should also copy `<main>/.endless/plans/E-NNNN.md` into the newly-created worktree's plans dir if present. Verify whether this already happens; if not, add it as part of this PR.

## Verification

1. **Triage filing (the surprise case):**
   ```sh
   # As a session that hasn't claimed E-1445:
   endless task update E-1445 --text /tmp/plan.md
   ```
   Expect: plan synced to `<main>/.endless/plans/E-1445.md`, DB text set, **no worktree created**, **no sandbox provisioned**. `endless worktree list` does not show e-1445.

2. **Working filing (the today-case):**
   ```sh
   endless task claim E-NNN     # status flips to in_progress, worktree created
   endless task update E-NNN --text /tmp/plan.md
   ```
   Expect: plan synced to the worktree's `.endless/plans/E-NNN.md` and DB. Same as today.

3. **Override on:**
   ```sh
   # On a needs_plan task:
   endless task update E-NNN --text /tmp/plan.md --create-worktree
   ```
   Expect: worktree force-created despite status.

4. **Override off:**
   ```sh
   endless task claim E-NNN     # in_progress
   endless task update E-NNN --text /tmp/plan.md --no-create-worktree
   ```
   Expect: heuristic says auto-create, override says no, override wins. Plan written to main only.

5. **Claim-after-triage copies plan:**
   ```sh
   # Earlier: triage filing wrote to <main>/.endless/plans/E-NNN.md
   endless task claim E-NNN
   ls .endless/worktrees/e-NNN/.endless/plans/E-NNN.md
   ```
   Expect: the file exists, content matches the earlier triage attach.

6. **Done-ish task:** attempt `task update --text` on a `confirmed` task → no worktree created; warning that the task is in a done-ish status (defensive; orthogonal to this change, but worth a courtesy log line).
