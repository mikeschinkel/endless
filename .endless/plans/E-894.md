# E-894 — Move task display reads from Python to Go

## Context

After E-805 moved task **writes** to Go via the events-authoritative pipeline, four task-display commands still query SQLite directly from Python: `task list`, `task show`, `task next`, `task search` — plus secondary read paths in `active_tasks`, `recent_tasks`, `list_decisions`, relation queries (`get_all_relations`, `_related_task_ids`, `show_relations`), `_project_root_for_task`, and a single `tasks` SELECT in `session_cmd.py:1300`. Per house rule (memory `feedback_no_python_sqlite_reads`), no new Python SQLite reads are written; legacy reads should migrate.

E-1116 surfaced a concrete cost of the split status quo: Python's `db.py:_migrate` ALTERs `tasks` without checking the table exists, crashing on a truly empty DB. Empty DBs were impossible historically, but under E-1074 sandboxes (E-1078 fixture in particular) they become routine. The exploration also confirmed a deeper divergence: **Python's migration chain runs through V6 (V5 task_deps active-voice rebuild + V6 `outcome` column); Go's runs only through V4**. Whichever side creates the DB first determines whether V5/V6 land. The split DB-ownership model is not just stylistically inconsistent — it has bug surface.

E-1053 already established the pilot for the read-side migration: `focus_cmd.py` reads via `query_bridge.py` shelling out to `endless-event` Go subcommands (`get-focus`, `get-session`, `project-by-path`, `focus-ancestors`, `focus-subtree`), with read helpers in `internal/events/query.go`. E-894 mirrors that pattern for tasks.

## Decision: Scope

**Option B — reads + schema ownership, in phases.**

Phases 1-2 deliver the read migration (E-894's nominal scope). Phases 3-4 move schema/migration ownership into Go (subsumes E-1116). Phase 5 deletes Python's SQLite footprint entirely.

Rationale:
- Option A (reads only) leaves the fragility E-1116 surfaced. The two Python-only migrations (V5 task_deps active-voice, V6 `outcome` column) remain Python's responsibility, contradicting E-894's direction "Python has zero SQLite knowledge for task operations".
- Costs (rough): Option A ~4-6 hr; Option B adds ~7-10 hr for full schema ownership. The full ownership is a one-time cost; the alternative is paying schema-divergence tax indefinitely.
- E-1116's narrow fix is "patch Python's `_migrate` to tolerate empty DB". The structural fix is "stop running Python migrations at all" — and E-894 already does most of the work to get there.
- Mike's prior signals (memories: `feedback_no_python_sqlite_reads`, `feedback_no_legacy`, `feedback_decouple_before_ship`) favor full decoupling over partial.

Phasing lets each phase land independently. If Mike wants to stop after Phase 2, the read migration is complete and useful on its own; Phases 3-5 can still happen later without rework.

## Architecture

**Extend `cmd/endless-event/`** rather than create a new binary. Precedent: `get-focus`, `focus-ancestors`, etc. are already there per E-1053. Adding task subcommands keeps the Python bridge layer's binary discovery (`shutil.which("endless-event")`) unchanged.

**Read helpers** go in `internal/events/query.go` alongside the focus helpers. Each helper returns a typed struct with JSON tags; the main.go runner marshals to stdout.

**Python bridge** extends the existing `src/endless/query_bridge.py` with task functions (mirrors how `get_focus`, `get_session` already live there). If `query_bridge.py` grows unwieldy, split into `query_bridge_focus.py` / `query_bridge_task.py` at implementation time — judgment call, default is to extend.

**JSON output convention** (mirrors focus pilot exactly):
- Single-row queries: one JSON object on stdout, or literal `null` if not found.
- List queries: JSON array on stdout, or `[]` if empty.
- Errors: stderr message prefixed `endless-event: error:`, exit 1 (or exit 2 for missing required flags).

**Subcommand surface (Phase 1):**

| Subcommand                       | Replaces                                          | Output      |
|----------------------------------|---------------------------------------------------|-------------|
| `get-task --id N`                | `task show` core query (`detail_item`)            | object/null |
| `list-tasks [filters]`           | `task list` core query (`show_plan`)              | array       |
| `next-tasks [filters --limit N]` | `task next` core query (`next_tasks`)             | array       |
| `search-tasks --query S [...]`   | `task search` core query (`search_tasks`)         | array       |
| `task-children --id N`           | `detail_item --children` query                    | array       |
| `task-relations --id N [--rel-type T]` | `show_relations`, `_related_task_ids`       | array       |
| `active-tasks`                   | `active_tasks` (in_progress + verify)             | array       |
| `recent-tasks [--limit N]`       | `recent_tasks`                                    | array       |
| `list-decisions [filters]`       | `list_decisions` (type='decision')                | array       |
| `task-by-source-file --path P`   | `_project_root_for_task` SQL                      | object/null |

Filters mirror the existing Click flags: `--phase`, `--status`, `--tier`, `--parent`, `--related-to`, `--rel-type`, `--all`, `--sort`, `--limit`. Flag syntax follows pilot precedent (raw `flag.FlagSet`).

Output JSON shapes must preserve every field Python's `--json`/`--llm` formatters consume today (full set captured per command in the agent inventory). Python continues to render Rich tables, markdown, and tree views — only the data-fetch step moves.

**Conventions (house rules):** new Go code applies ClearPath (`go-clearpath-auditor`), doterr (`go-error-handling-author`), and go-dt for paths. The pilot pre-dates strict adherence in some places; new code is held to the house standard. Migrating the pilot itself to house conventions is out of scope (separate task).

## Implementation phases

Each phase becomes a child task filed via `endless task add --parent E-894` after plan approval. Phases gate on the prior phase landing on main; each gets its own worktree per `feedback_one_task_per_worktree`.

**Phase 1 — Go read helpers + subcommands**
- Add task structs (`TaskRow`, `TaskChildRow`, `TaskRelationRow`, etc.) and helper functions to `internal/events/query.go`.
- Wire each helper to a runner in `cmd/endless-event/main.go` (or split runners into per-file modules following sandbox precedent if main.go grows too large).
- Reuse SQL from `internal/web/queries.go` where it already exists for tasks (DRY); lift shared queries into `query.go` rather than duplicate.
- Unit tests for each helper against a fixture DB.

**Phase 2 — Python bridge + cutover**
- Extend `src/endless/query_bridge.py` with `get_task`, `list_tasks`, `next_tasks`, `search_tasks`, `task_children`, `task_relations`, `active_tasks`, `recent_tasks`, `list_decisions`, `task_by_source_file`.
- Replace every `db.query(...)` call against the `tasks` / `task_deps` tables in `task_cmd.py`, `session_cmd.py`, and any other read-side caller with the bridge.
- Delete Python's read-side SQL bodies (`active_tasks`, `recent_tasks`, `list_decisions`, `get_all_relations`, `_related_task_ids`, `_project_root_for_task` SQL, `detail_item` SQL, `show_plan` SQL, `next_tasks` SQL, `search_tasks` SQL — keep formatting code, drop SQL).
- Run all tests; smoke-test each CLI command against golden output.

**Phase 3 — Move V5 + V6 migrations to Go**
- Port `_migrate_v5` (task_deps active-voice rebuild) into `migrateV5` in `internal/monitor/db.go`. Largest piece — disables FKs, rebuilds the table, swaps source/target on subset of rows. Validate with `endless-event validate-db` against pre-V5 DB snapshots.
- Port `_migrate_v6` (add `outcome` column to tasks) into `migrateV6` (trivial ALTER).
- Wire both into the existing `migrate()` orchestrator at `internal/monitor/db.go:140`.

**Phase 4 — Go owns DB initialization (subsumes E-1116)**
- Verify `monitor.DB()` is the only DB-creation path. Audit Python for any remaining `_init_schema`/`_migrate` invocations and remove them.
- Subsumes E-1116: Python no longer touches the DB at all on startup. Mark E-1116 `assumed` once Phase 4 lands. The empty-DB crash disappears because Python no longer runs ALTERs.

**Phase 5 — Delete Python db.py reads/migrations**
- Remove `_init_schema`, `_migrate*`, `query`, `scalar`, `exists` from `src/endless/db.py`. Delete `db.py` entirely if nothing else imports it; otherwise reduce to a stub.
- Remove `import sqlite3` from any Python module that still has it.
- Final sweep: grep `src/endless/` for `db.query(`, `sqlite3`, `.execute(`. Should be zero outside test fixtures.
