# E-1164 + E-1182 — Worktree-routing shell helpers + auto-inject shell-init

Bundled: E-1164 (routing logic in shell-init) and E-1182 (auto-inject shell-init in `endless-sandbox enter`) ship together because each has limited user-visible value alone. Routing without auto-inject only helps users who manually `eval $(endless shell-init)`. Auto-inject without routing means the wrong endless is called inside a sandbox.

E-1183 (fish support) is filed as `maybe` and out of scope here.

## Problem

Two coupled defects discovered while verifying E-1159:

1. `esu`/`esp`/`esf` call bare `endless`, which resolves to whatever's first on `$PATH` (typically the global `/usr/local/bin/endless`). New subcommands added in a worktree (e.g. `session forget` on `task/1159-…`) are unreachable from helpers until the worktree merges and `just install` runs from main. Surfaced when `esf` errored with `Error: No such command 'forget'.` despite the subcommand existing in the active worktree.

2. Shell helpers are only present if the user has run `eval "$(endless shell-init)"` in their rc. Inside a sandbox subshell launched by `endless-sandbox enter`, this depends on the user having added it themselves. Per Mike (2026-05-04): "every time you require the user to configure something you move farther away from 'it just works'."

## Approach

### Piece A — `_SHELL_INIT_SNIPPET` in `src/endless/cli.py` (E-1164)

Each helper (`esu`/`esp`/`esf`) gains:
- A precondition check: `[ -n "$ENDLESS_SESSION_ID" ]`, with a clear "no active session, run `esu` first" error for `esp`/`esf`. (`esu` is the entry point, so its check is different — it requires a project arg.)
- A worktree-routing check: when `ENDLESS_SESSION_ID` is set, run `endless session cd --target worktree "$ENDLESS_SESSION_ID"` to resolve the worktree path. If non-empty and a directory, set `cmd="uv run --directory $worktree_path endless"`. Otherwise fall back to bare `endless` with no warning (sessions without worktrees are normal).

No new marker env var. `ENDLESS_WORKTREE_PATH` (set by `esu`) already signals "session active in a worktree" for callers that care about state; `command -v esu` (or `declare -f esf` in tests) directly answers "are helpers defined" without needing a marker.

### Piece B — Auto-inject in `cmd/endless-sandbox/enter.go` (E-1182)

Before launching the subshell:
1. Detect shell from `filepath.Base(shellPath)`.
2. For `zsh`: create tempdir; symlink `~/.zshenv` (if exists) into it; write `<tmp>/.zshrc` containing `[ -f "$HOME/.zshrc" ] && source "$HOME/.zshrc"; eval "$(endless shell-init)"`; set `ZDOTDIR=<tmp>` in subshell env; defer `os.RemoveAll(tmp)`.
3. For `bash`: create tempfile containing `[ -f "$HOME/.bashrc" ] && source "$HOME/.bashrc"; eval "$(endless shell-init)"`; pass `--rcfile <tmpfile>` as args; defer cleanup.
4. For other shells (fish etc.): launch as today, no injection. (Filed E-1183.)

`endless` is on PATH (symlinked by `just install`); subshell can resolve `endless shell-init` without a hardcoded path.

The Supervisor.Run() returns after the subshell exits, so cleanup defers fire after the user types `exit`.

### Piece C — Proactive checks (per Mike: "if we can add a check elsewhere we should")

| Where | Check | Action on fail |
| --- | --- | --- |
| Inside auto-inject rcfile (zsh + bash), after the eval | `command -v esu >/dev/null 2>&1` | Print `endless: shell-init failed to load helpers (esu missing)` to stderr. Don't abort. |
| Inside `esp`/`esf` helper bodies | `[ -n "$ENDLESS_SESSION_ID" ]` | Print `<helper>: no active session, run 'esu <project>' first` to stderr; return non-zero. |
| Inside helper routing logic | `[ -d "$worktree_path" ]` after the lookup | Fall back to bare `endless`; no warning (legit case for sessions without worktrees). |
| Inside test fixture | `declare -f esf > /dev/null` after eval'ing shell-init | Test fails with clear diff. |

`functions <name>` (zsh) and `declare -f <name>` / `command -v <name>` (bash, portable) are the right primitives — they query the running shell's symbol table directly, so we never have to guess where the user defined the helper.

   ```sh
   ./bin/endless-sandbox destroy probe-1164 --if-exists
   ```
