# E-1552 — Decide endless task spawn behavior when invoked on various task types

## 1. Context

`endless task spawn` today claims a task, creates its worktree, opens a tmux
window, and pastes a Python `string.Template`-rendered handoff. The flow is
**type-blind**: spawn reads only `id`, `title`, `status`, `project_id`. The
handoff is a single document assuming the spawned session implements code and
ends in `verify`.

E-1538 lands `task_types` FK seeding (`task`, `bug`, `research`, `epic`).
E-1537 adopts `epic` as a coordinator-only type that never implements code
directly. That breaks today's spawn assumption: spawning an epic with the
current handoff produces a session that tries to implement a container.

The wider problem is tmux-window proliferation. Today's spawn opens a fresh
window per task. For an epic dispatching N children, that's 1 + N windows the
user must task-switch between. The May 2026 release of Claude Code's Agent
View (`claude --bg`, `claude agents`, `claude attach`) gives us a load-bearing
alternative: child sessions run as supervisor-hosted bg agents with no tmux
window of their own, viewable in-place from the coordinator's pane, promotable
to a dedicated window only when focus is needed.

This task delivers a single coherent design covering:
- All 24 cells of {`task`, `bug`, `research`, `epic`} × children-state.
- Bg-agent dispatch as the v1 default for epic children.
- Type-aware handoff templates rendered via Go `text/template`.
- DB and CLI changes needed to support coordinator + bg-agent operation.

Inputs:
- `docs/research-2026-06-10-claude-multi-agent-usage.md` (Model 1/2 framing).
- `docs/research-2026-06-12-claude-background-agents.md` (Agent View mechanics,
  `claude --bg`, `claude attach`, `worktree.bgIsolation`, recovery).
- E-1537 §6.5 — coordinator session semantics.
- E-1538 — `task_types` FK seeding (must land first).

## 2. The matrix

Design space: {`task`, `bug`, `research`, `epic`} × {zero, all `needs_plan`,
all `ready`, all `in_progress`, all terminal, mixed children}.

**Key reduction.** For types `task`, `bug`, `research`, the spawned session
works on **the parent task itself** — children are follow-ups (verification
matrices, drive-by discoveries, sub-research). Children-state doesn't change
what spawn does at the CLI level; it surfaces in the handoff as context.

For type=`epic`, the spawned session is a coordinator (E-1537 §6.5); children
ARE the work. Children-state varies what the coordinator **does inside its
session**, not what spawn does at the CLI level.

Conclusion: **spawn's CLI flow stays uniform across all 24 cells; the handoff
template branches.** Same pattern endless already uses for type-conditional
behavior (verb gates, completable-status gates).

## 3. Per-type spawn behavior

| Type | Spawn behavior | Handoff variant | End-state |
|---|---|---|---|
| `task` | Today's foreground tmux + claim | `handoff/task.md.tmpl` | `verify` → confirmed |
| `bug` | Same as task; reproduce-first framing | `handoff/bug.md.tmpl` | `verify` → confirmed |
| `research` | Same as task; findings-as-deliverable | `handoff/research.md.tmpl` | `completed` (verb gate per ED-1502) |
| `epic` | Foreground tmux coordinator + children-state summary | `handoff/epic.md.tmpl` | `completed` via auto-derivation (E-1537 §4) |

The coordinator session itself runs foreground tmux — interactive driver,
needs the user. Bg dispatch applies to its CHILDREN, not the coordinator
itself.

For all types, handoffs include the current children count and recommend the
spawned session read with `--children` flag. Zero renders as "(no children)";
no special case.

## 4. Bg-agent dispatch (the epic-child mechanism)

### 4.1 Carrier mechanism — `--name` binding

`endless task spawn --bg <id>` is the dispatch verb. It:

1. Validates the done-ish-status gate (today's set of statuses that require `--force` to claim over: `verify`, `confirmed`, `declined`, `obsolete`, `assumed`, `completed`).
2. Pre-claims: creates worktree + flips status to `in_progress` (today's
   `_perform_claim_work` with `target_session=None`).
3. Renders the type-appropriate handoff to a temp file.
4. Launches `claude --bg --name "E-<task_id>" "$(cat handoff.md)"` as a
   subprocess. Captures the short ID from stdout (the
   `backgrounded · <id> · <name>` line).
5. Writes the binding row directly: `sessions(task_id, claude_short_id,
   kind='background', active_epic_id, started_at)`.

No DB-token table, no env-var carrier, no hook-side lookup. Claude's `--name`
parameter IS the carrier; `claude agents --json` is the live discovery
surface. SessionStart hook still fires inside the bg agent (source=`startup`)
and may decorate the row with the full `session_id` from hook payload, but
isn't load-bearing.

E-1558 (filed) tracks the future addition of a DB-token mechanism for
non-Claude harnesses.

### 4.2 `worktree.bgIsolation: "none"` — REQUIRED for v1

Per the Agent View docs, `claude --bg` creates a nested isolation worktree
under `.claude/worktrees/` before edits. That defeats endless's pre-claim
(the bg agent would edit Claude's worktree, not the tracker's). The
`worktree.bgIsolation: "none"` setting (v2.1.143+) disables it.

`just claude-settings-init` already manages worktree-local
`.claude/settings.json`. Extend it to set `worktree.bgIsolation: "none"` so
bg agents stay in the launch cwd (the tracker worktree).

E-1559 (filed) tracks the long-term `WorktreeCreate` hook alternative. Open
issue density (~11 open issues, two regressions in June 2026, including
#68214's critical isolation-failure bug) means this stays maybe-phase until
Agent View exits research preview.

### 4.3 Throttling — soft warning

Quota scales linearly with parallel bg agents (10× for 10 agents).
Community sweet spot is 3–5 parallel; Pro plans hit thread limits at ~2.

`endless task spawn --bg <id>` issues a soft warning at dispatch when N bg
agents are already active for the project (default threshold 3,
configurable in `.endless/config.json` as `bg_throttle_warn`). Does NOT
block dispatch — coordinator decides whether to proceed. Warning text
surfaces the quota cost and thread-limit risk.

### 4.4 Children-state computation

Coordinator's epic handoff includes a state breakdown computed at
render-time:

```sql
SELECT status, count(*) FROM tasks WHERE parent_id = ? GROUP BY status
```

Rendered as e.g. "3 ready, 2 needs_plan, 1 in_progress, 4 terminal". Lets
the coordinator session pick its operational mode without re-querying.

### 4.5 Coordinator operational modes (in handoff)

The epic handoff guides the coordinator session into the right mode based
on the rendered state breakdown:

- **Zero children**: drive decomposition. Plan-mode + AskUserQuestion +
  parallel Explore sub-agents for context. File children via `endless task
  add --parent <epic-id>`.
- **All `needs_plan`**: Model 1 (planning orchestrator). Draft each child's
  plan inline; user reviews; attach via `endless task update <child> --text`.
- **All `ready`**: Model 2 (dispatcher). Run `endless task spawn --bg
  <child-id>` for each child (subject to throttling warning). Monitor via
  `claude agents --json --all` or `endless session list`.
- **All `in_progress`**: observe. No new dispatch. Surface child statuses.
- **All terminal**: epic should be `completed` via auto-derivation. Ask
  whether to reopen (add children) or accept as done.
- **Mixed**: surface breakdown; ask user which mode to start in. No
  auto-action.

## 5. Attach mechanisms

Two distinct verbs:

- **`endless task spawn --attach <id>`** — opens a NEW tmux window for an
  existing bg agent. Mechanism: `tmux new-window`; send-keys to run
  `claude attach <short-id>`. The supervisor keeps hosting the bg agent;
  attach just connects the new pane to it. Used when a child agent needs
  dedicated focus separate from the coordinator's pane.

- **`endless task attach <id>`** — exec into the bg agent IN THE CURRENT
  pane. Mechanism: `os.execvp("claude", ["attach", short_id])`. The
  endless Python process is replaced by claude in-place. Use case:
  sitting at a fresh shell, want to drop into a running bg agent without
  opening a new window.

Both detach paths (`Ctrl+Z`, `/exit`, `←`) leave the bg agent running.
Only `claude stop <id>` or `Ctrl+X` ends it.

## 6. Handoff templates

### 6.1 Format and renderer

Templates migrate from Python `string.Template` to **Go `text/template`**.
Renderer: a hidden `endless internal template render <name>` subcommand
that reads a JSON vars blob on stdin and writes the rendered output to
stdout. Python spawn shells out per render.

### 6.2 Location

`templates/` (top-level repo dir). Sub-organized:

- `templates/handoff/task.md.tmpl`
- `templates/handoff/bug.md.tmpl`
- `templates/handoff/research.md.tmpl`
- `templates/handoff/epic.md.tmpl`
- `templates/coordinator/explore-child-context.md.tmpl`
- `templates/coordinator/plan-child.md.tmpl`
- `templates/coordinator/dispatch-child-impl.md.tmpl`
- `templates/coordinator/verify-child.md.tmpl`

The existing `docs/templates/handoff.md` migrates to
`templates/handoff/task.md.tmpl` (with syntax conversion `$var` → `{{.Var}}`).

`templates/` is the home for now. Revisit on E-1486 (Python → Go port);
likely moves to a Go-embedded location at that point.

### 6.3 Coordinator templates

Endless ships a SET of templates the coordinator session uses for
sub-tasks (Explore context-gathering, child plan drafting, dispatch
handoff, verify-reviewer). Coordinator can override locally but defaults
to template. Keeps prompts evolvable and consistent across coordinators.

## 7. DB schema changes

Single new column:

- `sessions.active_epic_id INTEGER REFERENCES tasks(id) NULL`

Semantics:

| Session kind | `active_task_id` | `active_epic_id` |
|---|---|---|
| Non-epic foreground (today) | task being worked | NULL |
| Epic coordinator | initially = epic id; changes as user switches view | epic id (stable) |
| Bg agent dispatched from coordinator | child task id | parent epic id |

Bg agent rows live in `sessions` alongside coordinator rows. `tmux_pane`
is NULL for bg agents. Window-name rendering reads `active_epic_id +
active_task_id`:

- `active_epic_id IS NOT NULL AND active_epic_id != active_task_id`
  → `[E-EEEE:E-CCCC]`
- `active_epic_id IS NOT NULL AND = active_task_id` → `[E-EEEE]`
- `active_epic_id IS NULL` → `[E-NNNN]` (today's behavior)

"Currently focused bg agent" is **derived**: a bg agent is focused iff
`bg.active_task_id = coordinator.active_task_id AND bg.active_epic_id =
coordinator.active_epic_id`. No stored `focused` flag.

`sessions.kind` discriminator added: `tmux` (today's default) or
`background`. Used to filter `endless session list` views and to NULL the
`tmux_pane` constraint for bg rows.

## 8. Session lifecycle and recovery

Bg agents survive: foreground Claude exit, terminal close, tmux pane
close, machine sleep (v2.1.142+).
Bg agents die on: machine shutdown, 1h idle (unpinned), `claude stop`.

Recovery surfaces (filed as follow-ups, NOT in scope for E-1552):

- **E-1560** (next): `endless session restore [<id>|--all]` wraps
  `claude respawn` for post-reboot recovery.
- **E-1561** (next): `endless tmux restore` reconnects to surviving bg
  agents and respawns a coordinator after tmux crash.

E-1552's design assumes these will exist. The CLI surface today should
not actively prevent them from being added.

## 9. Status gate behavior

The done-ish-status gate stays uniform across types: the statuses
`verify`, `confirmed`, `declined`, `obsolete`, `assumed`, `completed`
require `--force` to spawn. Same for all types and both fg/bg paths.

For epic in `completed` (auto-derived): `--force` is "reopen this
initiative" — the epic handoff frames it as reopen.

## 10. Resolution of original open questions

| Original question | Resolution |
|---|---|
| Resolution mechanism: auto / `--mode` / interactive? | Auto-detect children-state at render-time; coordinator chooses operational mode in-session. No CLI mode flag. |
| Does orchestrator claim the epic? | Yes. Coordinator pre-claims the epic via today's spawn flow. |
| Children remain claimable by own sessions? | Each child's claim happens at `endless task spawn --bg <child-id>` time (deterministic, Go/Python-side), not the coordinator's responsibility. |
| Worktree creation timing | Per-child worktree created by `endless task spawn --bg <child-id>` at dispatch time. Not pre-claimed all at once. |
| Agent View integration | `claude agents --json --all` is the canonical bg-agent state source. `endless session list` queries it for live state. |
| Explore sub-agent dispatch | Endless ships a SET of coordinator templates in `templates/coordinator/`. Coordinator selects per use-case; can override locally. |
| Relationship to E-1136 | Co-exist. E-1552 = interactive coordinator. E-1136 = unattended daemon. Different operational shapes. |

   endless tracker worktree, not under `.claude/worktrees/`.
5. **Attach paths** — `task spawn --attach <child>` opens new tmux
   window running `claude attach <id>`; `task attach <child>` execs
   into claude in current pane.
6. **Throttling warning** — N+1th bg dispatch emits warning, proceeds.
7. **Session restoration story** — after machine sleep+wake, bg agents
   resume; after stop+respawn, agents recoverable via `claude respawn`.
8. **Status gate unchanged** — spawn on done-ish-status non-epic tasks
   refuses without `--force`; with `--force`, demotes and dispatches.
