# E-1552 — Decide endless task spawn behavior when invoked on various task types

## 1. Context

`endless task spawn` today claims a task, creates its worktree, opens a tmux
window, and pastes a generated handoff. The flow is **type-blind**: spawn reads
only `id`, `title`, `status`, `project_id`. The handoff template
(`docs/templates/handoff.md`) is a single document that assumes the spawned
session implements code and ends in `verify`.

E-1538 lands a `task_types` FK with values `task`, `bug`, `research`, `epic`.
E-1537 adopts `epic` as a coordinator-only type that never implements code
directly — its deliverable is planning and dispatch of children. That change
breaks the assumption baked into spawn: spawning an epic and pasting the
current handoff produces a session that tries to implement a container.

The impetus for this decision is the introduction of `epic`, but the
broader question is what spawn should mean for each declared type. Each type
carries different deliverable expectations (`research` ends in `completed`
not `verify`; `bug` reproduces before fixing; `epic` coordinates), and any of
the four types can have children. This task delivers a single coherent
design covering all 24 cells of {type} × {children-state}.

Inputs:
- `docs/research-2026-06-10-claude-multi-agent-usage.md` — research surfacing
  Model 1 (planning orchestrator) and Model 2 (implementation dispatcher).
- E-1136 — Go/Python coordinator daemon proposal for the unattended
  all-`ready` case.
- E-1537 §6.5 — coordinator session semantics this work extends.
- E-1538 — `task_types` FK seeding (gates this work — must land first).

## 3. The matrix

The design space is {`task`, `bug`, `research`, `epic`} × {zero, all
`needs_plan`, all `ready`, all `in_progress`, all terminal, mixed children}.

**Key reduction:** for types `task`, `bug`, `research`, the spawned session
works on **the parent task itself** — children are typically follow-ups
(verification matrices, drive-by discoveries, sub-research). Children-state
does not change what spawn does at the CLI level. Children-state surfaces in
the handoff only as context ("this task has 3 open children — file
discoveries against them, not new tasks").

For type=`epic`, the spawned session is a coordinator (per E-1537 §6.5) and
children-state IS the work. Children-state varies what the coordinator
**does inside its session** — not what spawn does at the CLI level.

Conclusion: **spawn's CLI flow stays uniform across all 24 cells; the
handoff template branches.** This keeps the CLI surface stable, pushes
complexity into a renderable artifact, and matches how endless already
handles per-type semantics elsewhere (verb gates, completable-status gates).

## 4. Recommended design

### 4.1 Spawn flow is type-uniform; handoff is type-aware

`endless task spawn <id>` continues to: (1) validate status gate, (2)
pre-claim, (3) open tmux window, (4) render handoff, (5) paste. No CLI
branching by type. The renderer reads `tasks.type_id` (joined to
`task_types.slug`) and selects a per-type handoff fragment.

This is the same pattern endless already uses for type-conditional behavior
(suggestions, task detail rendering — see `task_cmd.py:2791-2796`,
`suggestions_cmd.py:82`).

### 4.2 Per-type handoff variants

Four handoff templates in `docs/templates/handoff/`:

- `task.md` — current handoff (default). End state: `verify`.
- `bug.md` — same shape, framing emphasizes reproduce-first and that
  drive-by fixes get filed as separate `--type bug` tasks. End state:
  `verify`.
- `research.md` — same shape, but step 6 directs end state to `completed`
  (research's terminal status per E-1537 §3 / ED-1502), and step 5
  emphasizes findings-as-deliverable (memory:
  `feedback_audit_task_delivers_findings`). The session does not implement
  code; it produces the report.
- `epic.md` — replaced with a coordinator handoff (see §4.3).

Renderer selects based on `task_types.slug`. Unknown / null type
(legacy rows not yet tagged) falls back to `task.md`.

### 4.3 Epic coordinator handoff

The `epic.md` handoff frames the session as **coordinator, not
implementer**. Sections:

1. **Role** — drive children, don't implement code in this session.
2. **Children-state summary** — computed at handoff-render-time from a
   `SELECT status, count(*) FROM tasks WHERE parent_id=? GROUP BY status`
   query. Rendered as e.g. "3 `ready`, 2 `needs_plan`, 1 `in_progress`,
   4 terminal".
3. **Operational mode guidance** — depends on the children-state summary:
   - **Zero children**: drive child decomposition. Use plan-mode +
     `AskUserQuestion` + parallel `Explore` sub-agents (read-only, safe per
     the research doc §1) to gather context per candidate child. File each
     child via `endless task add --parent E-<epic-id>`.
   - **All `needs_plan`**: Model 1 — draft each child's plan inline, user
     reviews, attach via `endless task update <child> --text`. Status
     auto-promotes to `ready` (memory: `feedback_plan_attach_promotes_to_ready`).
   - **All `ready`**: Model 2 — dispatch by running `endless task spawn
     <child-id>` per child. Each child gets its own tmux window today;
     coordinator monitors via `endless task active` and `endless session
     list`. Background-session dispatch (`claude --bg`, Agent View) is
     phase-2 follow-up — see §5.
   - **All `in_progress`**: observe. Run `endless task active --parent
     E-<epic-id>` (filter to be filed; see §6) and surface status to the
     user. No new dispatch.
   - **All terminal**: epic should already be `completed` via E-1537 §4
     auto-derivation. Coordinator handoff instructs the session to ask the
     user whether the epic genuinely needs to reopen (add new children) or
     whether spawn was a mistake.
   - **Mixed**: coordinator surfaces the breakdown and asks the user which
     operational mode to start in. No auto-action; mixed-state epics are
     where human judgment is most needed.
4. **Discipline** — coordinator never implements code itself. Code work
   happens in spawned child sessions. Coordinator may edit the epic's
   `--text` plan (`endless task update <epic-id> --text ...`) as
   discoveries surface.
5. **End state** — set epic to `completed` only via auto-derivation when
   all children are terminal (E-1537 §4). Coordinator never sets the epic
   to `completed` directly. The coordinator session can be released
   (`endless task release`) when the user is done driving for now;
   another coordinator session can claim later (handoffs are normal per
   E-1537 §6.5).

The children-state summary is computed by the renderer, not by the
spawned session. This keeps the children-fetch query in one place
(spawn-side, with `--db main` already resolved) and prevents the spawned
session from having to do its own bucketing.

### 4.4 Status gate behavior across types

`_CLAIM_REQUIRES_FORCE` stays uniform for all types. The set
{`verify`, `confirmed`, `declined`, `obsolete`, `assumed`, `completed`}
is correct for every type:

- `task`/`bug`: `verify`/`confirmed`/`assumed` are real terminals.
- `research`: `completed` is the terminal (not `verify`); spawn on
  `completed` research is the "reopen for revision" case — needs `--force`.
- `epic`: `completed` is the auto-derived terminal; spawn on `completed`
  epic with `--force` is "reopen this initiative" — the coordinator
  handoff guides the user to either add children or accept the epic as
  done.

No type-specific carve-outs.

### 4.5 Claim binding and worktrees

For all types, spawn pre-claims the parent and creates the parent's
worktree. For epic, this gives the coordinator a workspace for plan-text
edits and coordination notes. Children's worktrees are created
independently when each child is spawned — the epic coordinator does not
pre-claim children.

**Coordinator never claims its children.** Each child claim is held by
the child's own spawned session. This preserves the
one-session-per-task invariant the rest of endless depends on (memory:
`feedback_session_id_not_interchangeable`).

### 4.6 Mode flag — rejected

`--mode plan|impl|join` was considered for explicit override of
children-state-derived operational mode. Rejected: the coordinator
session, not the CLI, decides the mode based on user input and the
state summary it sees. If the user wants to force a mode, the
coordinator session can do it from inside (e.g. user says "plan
children even though they're all ready"). Adding a CLI flag duplicates
the in-session affordance and creates a second source of truth for
"what is this coordinator doing right now."

## 5. Resolution of original open questions

| Original question | Resolution |
|-------------------|------------|
| Resolution mechanism: auto, `--mode`, interactive? | Auto-detect at handoff-render-time; surface state to coordinator; coordinator chooses operational mode in-session (§4.3, §4.6) |
| Does orchestrator claim the epic? Children claimable by own sessions? | Yes; yes (§4.5) |
| Orchestrator pre-claims `ready` children before dispatching? | No — each child claims itself via its own `endless task spawn` (§4.5) |
| Agent View integration — what does endless need to know? | Phase 2 follow-up. Today: dispatched children appear in `endless session list` and `endless task active` as ordinary spawned sessions. Background-session discovery is a separate task (see §6). |
| Explore sub-agent dispatch — template provided by endless? | No. Coordinator constructs Explore prompts as needed. Endless does not own Claude sub-agent prompts. (Optional future: a `docs/templates/explore-child-context.md` reference — filed under §6 maybe-phase if friction surfaces.) |
| Relationship to E-1136 | **Co-exist** (§5.1). |

### 5.1 Relationship to E-1136

E-1552 delivers the **interactive coordinator** mode: a human-driven
Claude session that decides per-state what to do.

E-1136 (`later` phase, `needs_plan`) delivers the **unattended daemon**
mode: a long-running Go/Python process that watches a task filter (e.g.
phase=now status=ready) and dispatches per-task sessions without a
human driver.

These are complementary operational shapes, not alternatives:
- Interactive (E-1552 epic spawn) = early-phase epic where the user is
  actively driving decomposition and dispatching first batches.
- Daemon (E-1136) = backlog grooming and unattended mop-up of `ready`
  work across many epics, including in projects that don't have an
  active coordinator session.

E-1136 is **not obsoleted** by this work and stays on the `later` lane.
E-1136's plan should reference this decision when it picks up.
