# E-1552 — Decide endless task spawn behavior when invoked on an epic

## 1. Scope

Decide how `endless task spawn` behaves when invoked on an epic. Today `task spawn` claims and starts a single Claude session on a single task. For an epic, the natural behavior varies by the state of the epic's children. This task delivers a single coherent design that covers all cases.

Implementation may decompose into multiple do-tasks afterward; this task delivers the **decision**.

## 2. Cases to design

Each case is `endless task spawn <epic-id>` invoked when the epic's children are in the following states:

1. **Zero children** — epic has no children yet. Spawn means "start writing children" (a planning mode that creates children from blank).
2. **All `needs_plan`** — planning orchestrator (Model 1 of docs/research-2026-06-10-claude-multi-agent-usage.md): orchestrator session uses parallel Explore sub-agents to gather context, drafts each child's plan, user reviews inline.
3. **All `ready`** — implementation dispatcher (Model 2 of the same research): orchestrator dispatches a Claude background session per child, each in its tracker-managed worktree, monitored from Agent View.
4. **All `in_progress`** — orchestrator-joins coordination: surface child statuses, no new work to start, possibly trigger a `task next` view scoped to the epic.
5. **All terminal** (`completed`/`confirmed`/`assumed`/`declined`/`obsolete`) — no-op or reopen prompt. Spawn on a finished epic is probably a user error.
6. **Mixed states** — the realistic common case. Spawn might offer interactive choice, infer-from-majority, or run each case-relevant action in turn.

## 3. Open design questions

- **Resolution mechanism**: auto-detect from child states, `--mode plan|impl|join` flag, interactive choice, or a combination?
- **Claim binding**: does the orchestrator session claim the epic? Children remain claimable by their own sessions?
- **Worktree creation**: per-child worktrees are created at `task claim` time. Does the orchestrator pre-claim all `ready` children before dispatching, or do the background sessions claim themselves?
- **Agent View integration**: what does endless need to know about background sessions dispatched by an orchestrator? Are they discoverable via `endless session list`?
- **Explore sub-agent dispatch**: for the planning case, how are Explore prompts structured? Does endless provide a per-task prompt template, or does the orchestrator construct them?
- **Relationship to E-1136**: E-1136 proposes a Go/Python daemon for the all-`ready` case. The two mechanisms (interactive Claude orchestrator vs. deterministic daemon) may co-exist, or one may obsolete the other. Settle the relationship as part of this decision.

## 4. Out of scope

- Implementation of each case's behavior. Implementation tasks get filed after the design lands.
- The `WorktreeCreate` hook integration (future direction; this task can recommend it but not commit to it).
- Cases where `task spawn` is invoked on a non-epic task (today's behavior, unchanged).

## 5. Inputs

- `docs/research-2026-06-10-claude-multi-agent-usage.md` — the research report that surfaced Model 1 and Model 2.
- E-1136 — the Go/Python daemon alternative for the all-`ready` case.
- E-1537's plan (specifically §6.5 epic management semantics) — the coordinator session pattern this work extends.

## 6. Deliverable

One or more decision records (`endless decision add`) capturing the spawn-on-epic design. Implementation tasks filed afterward if implementation work is required. Update to `endless guide` if documented behavior changes.
