# E-1416: Guard against dirty files in worktree as pre-condition for land

## Context

When `endless worktree land <id>` runs and the **worktree's** working tree has any uncommitted file, the rebase step fails with git's generic "You have unstaged changes" error — naming no files and offering no recovery. Hit during E-1307 land, manually resolved by adding the untracked snapshot pair and committing.

The fix is to inspect the worktree's tree as a pre-condition for rebase and **refuse loudly** on any dirt found — auto-managed *or* unmanaged — with a clear, actionable message. No silent auto-commit in the worktree; if auto-managed files are dirty in a worktree, that's an upstream writer bug to surface (Mike's "loud failure on invalid state" rule), not paper over.

## What `endless worktree land` does today

In plain terms (code at `src/endless/worktree_cmd.py:872`):

1. Look at the **main checkout's** working tree. Sort the dirt into two piles:
   - **(a) auto-managed** (db-ledger, snapshot pairs, verbs files — see `AUTO_COMMIT_GLOBS` at `worktree_cmd.py:54-59`)
   - **(b) unmanaged** — everything else
2. If (b) is non-empty → refuse to land, list the files, suggest commit / move / set-aside.
3. If (a) is non-empty → `git add` + commit on main as `"Endless: auto-record session activity"`.
4. Go into the **worktree** and run `git rebase main` there.
5. ff-merge the rebased branch back into main.
6. Remove the worktree.

## The gap

Step 1 only looks at the **main checkout's** tree. The **worktree's** own tree is never inspected. Step 4's rebase runs `cwd=worktree_path`, so any uncommitted file in the worktree makes git abort. The error message at `worktree_cmd.py:982-991` mentions a different recovery path (the E-972 case where a *committed* worktree commit modifies auto-files) and does not match this failure mode.

## Why the worktree had uncommitted snapshots (E-1307 incident)

`cmd/endless-hook/claude.go:944-957` (per E-1354): when the E-1281 sandbox is active, `snapshotPlanFile` writes the pair into `.endless/plans/snapshots/` and **explicitly skips the commit**. Reason at the time: committing would target the worktree's task branch, which the E-1309 guard refuses. E-1354 assumed snapshots would be discarded with the sandbox; in practice they survive until land tries to rebase.

This worktree right now has exactly this dirt:
```
?? .endless/plans/snapshots/20260518T020019-c1424923.json
?? .endless/plans/snapshots/20260518T020019-c1424923.md
```

E-1374 (status `needs_plan`, phase `maybe`) proposes moving snapshots to `.git/info/endless/snapshots/e-NNN/` — outside git tracking. If that lands, snapshot dirt becomes impossible by construction. **E-1416 does not depend on E-1374** — it catches the symptom regardless of whether snapshots ever move.

## Approach: pre-rebase guard in `land_worktree()`

Insert a new pre-rebase check between current step 3.7 and step 4. Pseudocode:

```python
# Step 3.8 (new): guard against dirty worktree tree before rebase.
try:
    wt_auto, wt_user = _git_status_partition(worktree_path)
except subprocess.CalledProcessError as e:
    raise click.ClickException(f"git status in worktree failed: {e.stderr or e}")

if wt_auto:
    # Auto-managed dirt in a worktree means an upstream writer is broken
    # (didn't self-commit). Surface loudly rather than paper over with a
    # silent auto-commit.
    file_list = "\n  ".join(wt_auto[:20])
    more = "" if len(wt_auto) <= 20 else f"\n  ... and {len(wt_auto) - 20} more"
    raise click.ClickException(
        f"worktree for {canonical} has uncommitted auto-managed files; "
        f"cannot land.\n\n"
        f"Files:\n  {file_list}{more}\n\n"
        f"These files are owned by endless writers that are supposed to "
        f"commit them at write time. Their presence here means a writer "
        f"is broken or skipped its commit. Report the writer that "
        f"produced these files; do not auto-commit them manually."
    )

if wt_user:
    file_list = "\n  ".join(wt_user[:20])
    more = "" if len(wt_user) <= 20 else f"\n  ... and {len(wt_user) - 20} more"
    raise click.ClickException(
        f"worktree for {canonical} has uncommitted user changes; "
        f"cannot land.\n\n"
        f"Files:\n  {file_list}{more}\n\n"
        f"Resolve from inside the worktree:\n"
        f"  - commit on {branch} (most common)\n"
        f"  - move the file aside (mv outside the worktree)\n"
        f"  - revert if unwanted (git checkout -- <file>)\n"
        f"then retry land."
    )
```

Reuses `_git_status_partition()` (at `worktree_cmd.py:522`) and `AUTO_COMMIT_GLOBS` (at line 54). No new helpers. ~30 lines including the two messages.

The existing step 4 rebase error path stays as-is (still covers the genuine E-972 case it was written for). The "unstaged changes" failure mode is now caught before reaching it.
