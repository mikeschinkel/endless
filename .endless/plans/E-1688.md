# Rename `session next` → `session status`, add live `session monitor`

## Why
`session next` (E-1465, landed) is mis-named: the view shows the session's whole **status** — current task, parent, in-flight-elsewhere, blocked, dependents — not just "what's next." And its dominant use is a persistent live pane (`top`-like), which deserves a monitor verb. Split the one verb into an accurately-named snapshot/live pair.

## What
- **`endless session status`** — render the view ONCE (rename of `session next`). The scriptable / agent-callable snapshot ("what's left for this session?").
- **`endless session monitor`** — the live dashboard: repeatedly render `status` (the `top`-like pane kept open all day). **One shared renderer** — `monitor` loops `status`; it replaces the prototype's `session-next.sh` watch loop.
- **Remove `session next`** — unshipped, single-user, so no alias / back-compat (per the no-migration-for-unshipped rule).

## Notes
- `monitor` owns the live-loop concerns the prototype shell handled: redraw-only-on-change (no flashing), fill terminal width, clean Ctrl-C exit, and per-window focal resolution (compose `GetActiveTaskForPane` + `@endless_task_id` fallback, per ED-1523 and E-1465's plan).
- `monitor` lightly overlaps endless's internal `monitor` package (not user-facing) — acceptable; "the session monitor" is coherent with it.
- `project next` should follow the same split later (`project status` + `project monitor`) — separate task, deliberate this time.

## Sequencing with siblings
The dependent-rows (E-1685) and `--tree` (E-1684) features attach to this same view and reference `session next` in their plans. This rename should ideally land **before** them so they build on the final `status`/`monitor` names; whichever lands first, the other adapts.
