# E-987 — Resolve event-log auto-append vs clean-main tension

## Context

E-987 is a `research` task whose v1 design (locked 2026-04-30) defined how `endless worktree land` reconciles append-only auto-managed files with the "clean-main precondition" gate. The v1 plan auto-commits dirty endless-managed files on `main` as `Endless: auto-record session activity`, rebases the worktree onto `main`, and ff-merges. For rebase conflicts on auto-managed files (the "architectural drift" path), the v1 plan surfaces a copy-pasteable recovery hint:

```
git rebase --abort
git checkout main -- .endless/events/ .endless/plans/snapshots/
endless worktree land <task-id>
```

The 2026-05-11 addendum (commit 36f63c4) on E-987's text records that E-1240 demonstrated a case the recovery hint mishandles: a worktree branch's feature commit on `.endless/verbs.json` would be silently destroyed by the suggested `git checkout`. The addendum opened four design questions. This plan resolves them in light of work that landed after the v1 design.

## What has actually changed since v1

The v1 plan's framing — auto-commit-at-land as the primary reconciliation mechanism — is being superseded piecewise by tasks filed after E-987 was locked. The relevant tasks (none of which I'm touching here, but which set the trajectory):

| Task   | Status        | Effect on E-987's premise |
|--------|---------------|---------------------------|
| **E-1268** | confirmed (landed) | `verbs.json` → `verbs.jsonl` with `merge=union` in `.gitattributes`. Closes the textual rebase conflict on parallel verb appends. The E-1240 headline scenario no longer reproduces. |
| **E-1208** | needs_plan (now) | Auto-commit `verbs.jsonl` immediately on `endless verb add`, not at land. Moves the commit point forward of `land`. |
| **E-1206** | needs_plan (now) | Auto-commit each `.endless/db-ledger/` write immediately, not at land. Same shift in commit timing. |
| **E-1216** | needs_plan (now) | Move plan-file writes from main to the active task worktree. Plan files stop being a main-dirt source. |
| **E-1207** | needs_plan (now) | Validate plan-file path against the task's worktree on `task update --text`. Enforcement layer for E-1216. |
| **E-1202** | needs_plan (now) | Forbid plan files at `.endless/plans/E-NNN.md` in main's working tree. Hardens E-1216. |
| **E-1061** | later, needs_plan | Generic post-edit commit prompt for any CLI mutating auto-committed files. |

**Architectural shift (per Mike's framing):** once endless is hardened, *neither endless nor Claude should ever leave main dirty.* "Any dirt" on main is a bug (errant search-and-replace, agent error, user edit) — auto-commit is not the right response. Auto-commit belongs at the point of write (E-1206, E-1208), or on a posterity boundary (snapshots → at `land`).

After those tasks land, `land`'s auto-commit step becomes a **failsafe for residual/legacy dirt**, not the primary commit mechanism. The "dirt-on-main" loop in `worktree_cmd.py:702–727` should eventually be reduced from the steady-state path to a guarded fallback that *flags* unexpected dirt rather than silently committing it.

## Resolution of the four addendum questions

### Q1 (Detection of feature-intent vs auto-record on rebase conflict)

**Defer.** No concrete use-case remains:

- E-1268 closed the verbs.jsonl path via `merge=union`.
- Per E-972, db-ledger and snapshots never receive worktree-side writes in normal operation; conflicts on them indicate a routing bug.
- Once E-1206 / E-1208 / E-1216 land, the auto-commit-on-main step that creates the substrate for conflicts will be a failsafe, not the steady-state path.

If a new use-case emerges (a future auto-managed file gets feature-edit semantics, or a regression reopens the path), revisit and build the partitioned recovery then. Until then, the residual sharp edge in the current `worktree_cmd.py:744–759` recovery hint is documented here; no code change.

### Q2 (db-ledger / snapshot conflict action)

**Refuse with "routing bug — investigate."** Per E-972 these conflicts shouldn't occur in normal operation. If they do, auto-discarding worktree-side content silently destroys the only evidence of where the routing broke. This is a design directive for whoever picks up E-1019 (whenever Q1 reopens).

Note: this does not contradict Mike's "we all know with Git you can never say never" framing on hand-editing. Snapshots and verbs *can* be hand-edited in main when committed (that's normal). The "routing bug" framing applies specifically to *worktree-side* commits on db-ledger / snapshots, which contradict E-972.

### Q3 (Scope split)

**E-1019 owns the eventual implementation** of any partitioned recovery hint. Status stays `needs_plan` until Q1 reopens. No new sibling task under E-968.

### Q4 (E-1017 hand-edit prohibition wording)

**No carve-out needed.** E-1017's "hand-edit" prohibition is a strong preference, not a draconian rule (per Mike's correction). Code-driven commits from a worktree that modify auto-managed files are not hand-edits, and Mike's stated position is that even hand-edits on main (if committed) are acceptable for less-critical files (snapshots = posterity; verbs.jsonl = actively maintained). The wording does not need a feature-intent carve-out.

**Separate concern:** E-1017's text still uses the old `events/` directory name. Post-E-1197 it should read `db-ledger/`. Pending discussion on placement (see "Open question" below).
