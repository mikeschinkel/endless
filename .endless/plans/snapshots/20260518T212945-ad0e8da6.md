# Pre-rebase check in `worktree land`: scope-down to `.endless/`

## Context

`endless worktree land` invokes `git rebase` without a structured pre-flight
check of the worktree's working tree. When the tree is dirty, rebase fails
partway with a generic "You have unstaged changes" error — no file list, no
recovery hint.

Commit `3e22fd1` ("E-1416: guard worktree land against dirty worktree tree
before rebase") already added `_guard_dirty_worktree()` at Step 3.8 in
`src/endless/worktree_cmd.py:1020`, just before the rebase at Step 4. The
guard partitions dirty files via `_git_status_partition()` (reusing
`AUTO_COMMIT_GLOBS` from line 54) and raises distinct `click.ClickException`
messages for auto-managed dirt vs everything else.

This task description narrows the scope: the safety net should fire only
when `.endless/`-prefixed files are dirty. Dirt outside `.endless/` should
fall through to rebase's native handling (step 3 of the spec). The current
implementation is broader — it aborts on ANY user dirt anywhere in the
worktree.

## Why narrow the scope

E-1361/1362/1363/1364 are removing the original sources of `.endless/`
dirt by moving snapshots and verbs.json out of the working tree. After they
land, the only remaining ways for `.endless/` to be dirty are:

- a new writer added without auto-commit hygiene
- a manual user edit under `.endless/`
- a partially-staged plan file

These are all cases where giving the user a named-file error is genuinely
better than git's generic message — because the file is endless-owned and
the recovery action is "commit it as a separate Endless: … commit".

Dirty files outside `.endless/` (e.g., `src/foo.py`) are not endless's
concern. The user wrote them; rebase's native error is the right behavior.
Aborting early on those is overreach.

## Scope

In `_guard_dirty_worktree()` at
`src/endless/worktree_cmd.py:572-614`:

1. Keep the `_git_status_partition()` call to get `(wt_auto, wt_user)`.
2. Filter `wt_user` down to entries whose path starts with `.endless/`
   (call it `wt_user_endless`). Discard entries outside `.endless/`.
3. Preserve the existing two-branch refusal:
   - `wt_auto` non-empty → existing "writer is broken" message (no change).
   - `wt_user_endless` non-empty → existing "commit / move aside / revert"
     message, **but** add a literal recovery command example:
     `git add <files> && git commit -m "Endless: record ledger entry"`
     (or another commit subject from `AMENDABLE_COMMIT_SUBJECTS` at
     line 66, whichever fits the path shape).
4. If both `wt_auto` is empty AND `wt_user_endless` is empty, return
   normally — rebase at Step 4 handles non-`.endless/` dirt natively.
