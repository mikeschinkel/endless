# E-1426 â€” Retire `.endless/sessions/*.json` companion files

## Context

`internal/monitor/companion.go` writes a per-Claude-session JSON file at `<projectRoot>/.endless/sessions/claude-<harness_uuid>.json` on every SessionStart / UserPromptSubmit / post-task-mutation, and removes it on SessionEnd. The file carries `harness`, `harness_session_id`, `endless_session_id`, `pane_id`, `cwd`, `pid`, `started_at`, `worktree_path`. Python CLI code globs that directory via `_read_live_companions` (`session_cmd.py:1000`) to answer "which pane belongs to which endless session" and similar questions.

Every field is now either already in the `sessions` DB row (`session_id` is the harness UUID, `id` is the endless_session_id, `process` is the pane id) or derivable on demand (`cwd` via `tmux display-message`, `worktree_path` via `WorktreePathForTask`). The file has accumulated three real failure modes that the DB-only path will fix:

- **E-1408** â€” `SetProcess` skips empty `TMUX_PANE` at SessionStart, leaving `process` NULL silently.
- **E-1419** â€” A session started in a worktree writes its companion under the worktree's `.endless/sessions/`, and main-DB resolution from a worktree cwd walks past it.
- **E-1422** â€” A pane with no companion has no recovery path, blocking CLI use after the E-1401 gate.

E-1426 replaces the file with idempotent UPSERTs on the `sessions` table fired by **every** Claude hook event. This is the single change that subsumes all three.

## Decisions (Mike, 2026-05-20)

| # | Question | Decision |
|---|---|---|
| 1 | Where the UPSERT happens | **Go hook binary writes the DB directly** via existing `monitor.*` package (forced â€” keying on `process` would destroy durable rows; keying on `session_id` is forced; the hook already opens the DB every event) |
| 2 | UPSERT key | **`session_id`** (the harness UUID; column is already `UNIQUE`) |
| 3 | Hook coverage | **Every event** (SessionStart, UserPromptSubmit, PreToolUse, PostToolUse, ExitPlanMode, Stop, PreCompact, SessionEnd) â€” single UPSERT path, self-healing within one tool call if `TMUX_PANE` was empty at SessionStart |
| 4 | Migration path | **Hard cutover** â€” companion writes and reads deleted in one PR; per-event refresh makes the upgrade window self-heal within one tool call |
| 5 | Liveness / reaper | **No `pid` column, no `start_time` column.** Liveness = `state != 'ended'` AND (for `%`-form `process` values) the pane still exists per `tmux list-panes -a`. Reaper marks dead-pane rows `ended` opportunistically at SessionStart, throttled via `ShouldThrottle`. New SessionStart for an already-occupied pane marks the prior occupant `ended` in the same transaction (collision invalidation). Crashed-Claude-but-pane-alive is explicitly NOT instrumented; if it becomes a problem, address then. |
| 6 | `cwd` handling | **Derive on demand via `tmux display-message -p '#{pane_current_path}'`.** Non-tmux Claude sessions lose `cwd` display (accepted; non-tmux Claude isn't a current pathway). No new column. |
| â€” | Archived-session handling | **Out of scope for E-1426.** Archived sessions (parked in a tmux session literally named `archive`) have live Claude pids, so they show as "live" under both companion-file and DB-only schemes â€” no regression. New task to be filed for the archive subsystem itself (the `docs/private/endless-tmux-archive-research.md` design has never been ticketed). |

`process` column stays the abstraction name â€” do NOT rename to `pane_id` (`endless-channel/main.go:60` already uses a `pid:N` fallback for the non-Claude channel binary, and the column may carry similar values for future non-tmux Claude harnesses).

## Implementation

### Go side

**`internal/monitor/companion.go`** â€” delete `CompanionFile` struct, `WriteCompanion`, `RemoveCompanion`, `writeCompanionAtRoot`, `removeCompanionAtRoot`, `companionPathInDir`, `companionDirAtRoot`. Keep `WorktreePathForTask` (still needed by hook worktree-gate logic) â€” move it to `internal/monitor/worktree.go` if `companion.go` ends up empty, otherwise leave the trimmed file.

**`internal/monitor/session.go`** â€” add `TouchSession(sessionID, projectID, process string) error`:

```go
// UPSERT shape (sketch):
INSERT INTO sessions (session_id, project_id, platform, state, process, started_at, last_activity)
VALUES (?, ?, 'claude', 'needs_input', NULLIF(?, ''), ?, ?)
ON CONFLICT(session_id) DO UPDATE SET
    last_activity = excluded.last_activity,
    process       = COALESCE(NULLIF(excluded.process, ''), sessions.process);
```

Then, in the same transaction, run the collision-invalidation:

```go
UPDATE sessions
SET state = 'ended'
WHERE process = ?  -- the new non-empty process
  AND session_id != ?  -- not this session
  AND state != 'ended';
```

Existing `BindSessionToTask` / `StartChatSession` / `SetProcess` / `BackfillProcess` keep their current semantics â€” `TouchSession` is the per-event hot path; the others handle state transitions.

Add `ReapDeadPanes(projectID int64) error`:

```go
// 1. Shell out to: tmux list-panes -a -F '#{pane_id}'
// 2. Collect alive %-panes into a set.
// 3. UPDATE sessions SET state='ended'
//    WHERE state != 'ended'
//      AND process GLOB '%[0-9]*'   -- only %-form values
//      AND process NOT IN (<alive set>)
//      AND project_id = ?;
// 4. If tmux is not running (exit != 0), no-op cleanly.
```

Add a throttle key in the existing `ShouldThrottle` machinery (e.g. `reap_dead_panes`, 60s) so the reaper runs at most once per minute even if SessionStart fires repeatedly.

**`cmd/endless-hook/claude.go`** â€” at the top of `ProcessHook` (after the payload is parsed, before the event switch), call:

```go

```

## Verification

**Go unit tests** (new + extend):
- `TouchSession`: insert path creates a `needs_input` row with `process` set when non-empty; update path overwrites `last_activity` and only overwrites `process` when the new value is non-empty; `state` is preserved across calls.
- Collision invalidation: a `TouchSession` for `(sessionA, %5)` followed by `TouchSession` for `(sessionB, %5)` leaves `sessionA.state = 'ended'`.
- `ReapDeadPanes`: with a stubbed `tmux list-panes` returning `{%5, %12}`, a row with `process='%99'` is marked `ended`; rows with `process='%5'` are untouched; rows with `process='pid:1234'` are untouched (only %-form is touched); rows already `state='ended'` are not rewritten.
- E-1408 regression: a hook ProcessHook call with `TMUX_PANE=""` at SessionStart followed by a PreToolUse with `TMUX_PANE="%5"` leaves the row with `process='%5'`.
- E-1419 regression: ProcessHook from a worktree cwd writes to the single shared DB (sandbox if `worktree_sandbox=true`); no `.endless/sessions/` directory is created anywhere.

**Python tests**:
- `_live_sessions` parses the `endless session list-live` JSON and returns the expected shape; handles empty result, missing tmux, malformed JSON (subprocess error path).
- `_current_endless_session_id`: pane-match â†’ returns eid; no-match â†’ `None`; multiple matches â†’ most-recent `last_activity` wins (mirroring `GetLiveSessionByProcess`).
- `_list_sibling_claude_session_eids`: filters by sibling-pane set; excludes `state='ended'`.
- `task_release_resolve` stale-binding: a `state='working'` row for a binder whose pane has been reaped resolves as not-live; auto-clear fires.
