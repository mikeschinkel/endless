## Context

User's argument for filing this as a sibling, not folding into E-1139: E-1139 is decision-relation work; this is task-relation work. They feel different even if the implementation shape is similar. Keep the surfaces separated for clarity and so each can land or be revised independently.

Load-bearing motivation (user's words): cleanup tasks "deserve to be identified independently so that AI doesn't choose to ignore them." Folding cleanup into a parent task lets agents declare done when the test passes; making the cleanup its own task with an explicit relation gives it independent visibility (`task active`, `task complete` validation, etc.).

Distinction from `blocks`:
- `blocks` = sequential dependency. A→B→C feature work where each step gates the next. Strong "Y can't proceed without X."
- new type = "Y is the natural follow-up to X; X ships without waiting on Y, but Y exists because X shipped and surfaced something that wants attention."

## Naming tradeoff (open question)

### Option α: `cleans_up` / `cleaned_up_by`

Pros:
- Concrete, narrow, reachable. Aligns with E-1003's principle (drop `informs` as too vague — narrow-and-reachable verbs over abstract catch-alls).
- The motivating use case (E-1139 retype work, post-shipping data cleanup) is exactly cleanup.
- Hard to misuse. If the relation isn't cleanup, the user reaches for something else.

Cons:
- Excludes legitimate post-ship work that isn't strictly cleanup: doc updates after a feature lands, dashboard instrumentation, removing a feature flag after rampup, comms/notification tasks. Users would fall back to `blocks` for those, overloading `blocks` again.

### Option β: `follows_up` / `has_follow_up`

Pros:
- Broader; catches cleanup AND other post-ship work (docs, instrumentation, ramp-down, etc.).
- The "AI ignores them" load-bearing property applies to all of these, not just cleanup.

Cons:
- Vague enough to drift toward catch-all use. Risks becoming the new `relates_to` — meaningful in principle, noise in practice.

### Option γ: both (add two types)

Pros:
- Precision when wanted (`cleans_up` for cleanup), breadth when needed (`follows_up` for everything else).
- Schema cost is zero — `dep_type` column is permissive per E-958.

Cons:
- More vocabulary surface to maintain and document.
- Two near-synonyms invite "which one applies here?" friction.

### Recommendation

Start narrow with `cleans_up` only. Reason: vocabulary expansion is additive and cheap (no CHECK constraints), so a future task can promote `follows_up` if broader follow-up cases recur. Starting broad and trying to enforce narrowness later is harder. YAGNI applies to the broader form until real demand surfaces.

This recommendation should not pre-empt the user's call — the plan asks them via AskUserQuestion before finalizing.
