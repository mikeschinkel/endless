# Plan: File parent task for Endless tmux status line — ephemeral apply + runtime printer

## Context

Mike wants a second tmux status line (`status-format[1]`, leaving the existing `status-format[0]` user menu untouched) that surfaces Endless context — current task ID + title at minimum — and serves as a launching surface for **interactive popup menus**: hotkey-triggered (prefix + key → `display-menu`) and right-click-triggered (`MouseDown3...` → `display-menu`), matching the patterns in `~/.init/tmux/conf.d/70-mouse-menus.conf`.

The brief covers tmux mechanics (status-format[N], `#(shell-cmd)` polling, `set-option` push, status-interval) and proposed the printer command — this plan narrows scope to "ephemeral only, today" and adds the menu-framework dimension the brief omits.

## Two verbs — final shape

| Verb | Who calls it | Effect |
|---|---|---|
| `endless tmux status` | **End user**, once per tmux server start | Issues a batch of `tmux` commands against the running server: enable second status line, set `status-format[1]` to call the printer, install hotkey + mouse menu bindings. **No file I/O.** Reverses on tmux server restart. |
| `endless tmux status-line` | **Tmux**, on each `status-interval` refresh | Reads tmux pane env + DB, prints one styled line of text to stdout. <50ms target. Always exits 0. |

Naming: short verb for the human (typed every tmux start), longer verb for the runtime invocation (configured once, never typed). Matches Mike's correction.

## What `endless tmux status` does (the apply command)

A single command issues this sequence of `tmux` calls against the running server:

1. **Enable second status line:**
   `tmux set-option -g status 2`

2. **Wire status-format[1] to the printer:**
   `tmux set-option -g status-format[1] '#[fg=colour245]#(endless tmux status-line)#[default]'`

3. **Tighten refresh cadence (per brief):**
   `tmux set-option -g status-interval 2`

4. **Install hotkey-triggered popup menu (case b):**
   - `tmux bind-key e display-menu -T "Endless" -x M -y W <items>` (prefix + e)
   - Items use the same idioms as `~/.init/tmux/conf.d/70-mouse-menus.conf`: hotkey letter shortcuts, `''` separators, conditional dimming via `#{?...,,-}`, `run-shell` for actions, `command-prompt` where input is needed.

5. **Install right-click popup menu (case c, best-effort):**
   - `tmux bind-key -n MouseDown3StatusRight display-menu -T "Endless" -x M -y W <same items>` — anchors the menu to the status-right region of the *first* status line, since tmux does not provide per-region mouse events scoped to `status-format[N>0]`. Limitation noted; a child task can investigate `#[push-default]`/`#[mouse]` workarounds for finer-grained click regions on row 1.
   - `M-MouseDown3StatusRight` variant for Alt+right-click consistency with Mike's existing mouse menus.

6. **Issue `tmux refresh-client -S`** to redraw immediately (per brief recommendation).

All six are idempotent — re-running `endless tmux status` overwrites the same options/bindings, no duplication.

### Minimal v1 menu content

The menu skeleton ships with a small starter set, deliberately thin so future child tasks can flesh it out:

```
Endless
─────────────
[d] Detail        — run-shell "endless task show $(endless task active --id-only) | tmux display-popup -E ..."
[v] Verify         — run-shell "endless task update $(endless task active --id-only) --status verify"
[t] Task tree      — run-shell "endless task list --tree | tmux display-popup -E ..."
                  ''
[r] Refresh         — refresh-client -S
```

Hotkeys mirror the menu's bracketed letter (Mike's existing menu idiom). All actions read the active task via `endless task active`; `endless task active --id-only` is implicitly proposed here as a needed CLI surface (a "needs verification" item below).

## What `endless tmux status-line` does (the printer)

- Reads `TMUX_PANE` + tmux user vars (`@session_uuid` etc., set by `internal/monitor/tmux.go:GetTmuxContext`).
- Queries the DB for the active task (single indexed read; uses existing `endless task active` plumbing).
- Formats a one-line string with tmux `#[…]` style escapes. Default v1 shape (per brief):
  ```
  #[fg=colour39][E-NNNN]#[default] <title> #[fg=colour245]· <project> · <status>#[default]
  ```
- Always exits 0 (per brief — non-zero in `#()` shows ugly empty bar).
- Suppresses stderr.
- Latency target <50ms.

## Open question: what is "current"? (per design brief §"Open question")

The printer must decide what "current task" means in a session-scoped status line. The brief lists four options (cwd-based, session-name-based, Endless-tracked global, explicit `--project`). **This plan defers the answer** — it's a design decision the parent task's plan-text should resolve before implementation, but does not need answering at *task-filing* time. Note in the plan-text that this is a critical pre-implementation question.

## Out of scope (explicitly — deferred to child tasks)

These are **not** blockers; they're the natural feature backlog once the parent ships:

- **Permanent install** — companion file at `~/.config/endless/tmux-status.conf` + `setup tmux-status` install verb (with auto-edit / manual / print-only modes; Mike's "won't be correct for me" power-user case)
- **Lineage breadcrumb in status text** (consumes E-1128)
- **Sibling-window task summary** (consumes E-1125)
- **Session focus title `EF-NNN` in status text** (consumes E-1031)
- **Color/state theming** (in_progress=green, verify=yellow, blocked=red)
- **Configurable format string** in `.endless/config.json`
- **Richer menu content** (more items, dynamic per task state)
- **Per-region mouse events on status-format[1]** — investigate `#[push-default]`/`#[mouse]` for clickable elements within the second status line itself, not just anchored to status-right of row 0
- **Truncation/collapse logic** for narrow tmux panes
- **Push-based updates** — Endless calls `tmux set-option` directly on state changes (per brief; defer until polling proves laggy)
