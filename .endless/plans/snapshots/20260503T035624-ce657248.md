## Action 2: verb-storage discussion — options matrix

The conversation needs concrete options on the table. These are all viable; some are mutually exclusive, some compose.

### Status quo: E-1112 — anchor to main checkout

Project-config / verbs writes from any worktree resolve to main's working tree. Machine layer also writes (`~/.config/endless/config.json` + `~/.config/endless/verbs.json`).

**Pros**
- Single canonical project state. New worktrees and new clones inherit immediately.
- Aggregates all worktree-session learning into one place that gets pushed.
- No duplicate-verb waste across sessions.
- Cross-machine sync rides on `git pull`/`push` for free.

**Cons**
- Main's working tree is dirtied by ambient session activity (`endless phrase add`, `verb add`). Other sessions trying to `worktree land` see dirty main and need the auto-commit list (E-1056) to make progress.
- Verb addition is *not branch-attributable* — a verb added during work on E-1059 lands directly in main's history under whatever commit auto-commits absorb it into, not E-1059's branch.
- Cross-session race when two worktrees both run `phrase add` simultaneously (last-writer-wins on the JSON file). Deduped on the value level, but the file's git diff order is non-deterministic.

### Option A: out-of-working-tree project state (split the file)

Verbs and matchers move from `<main>/.endless/{config,verbs}.json` to `<main>/.git/info/endless/` or `~/.config/endless/projects/<name>/`. Other project metadata (project name, status, etc.) stays in `.endless/config.json`.

**Pros**
- Eliminates the dirty-main class of problem at its source. Auto-commit-of-config no longer needed for verbs.
- E-1112's aggregation property still works — single file, all sessions write to it.
- No race conflicts during `git rebase` because the file is not in any branch.

**Cons**
- Project state is no longer git-tracked. Cross-machine sync requires a separate mechanism (or relies on machine-config, which is portfolio-wide and not per-project).
- Loses the "this commit added these verbs" provenance for project history.
- New clone of the project starts with empty verbs (would need a seed file in the working tree that the read order falls back to).

### Option B: read-many-write-machine-only

Project-level verbs file becomes seed-only: read at startup, never written. All new `phrase add` writes go solely to machine config.

**Pros**
- Main is never dirtied by session activity at all.
- E-1112's aggregation works for the current machine via the machine layer (which is already accumulating).
- Project-level seed gives new clones a starting point without coupling to runtime writes.

**Cons**
- Per-project learning does not sync between machines. A verb learned on machine A while working on project X never reaches machine B's clone of X (until manually promoted).
- Promotion-to-project becomes a separate step (`endless verb promote`?) the user has to remember.

### Option C: worktree-local writes + reconciliation on land

Each worktree writes verbs to its own `.endless/verbs.json`. Land-flow merges the worktree's verbs into main's verbs at land time.

**Pros**
- Branch-attributable. Verbs added during E-1059 ride with E-1059's branch and commit message.
- No cross-session race (each worktree owns its file).

**Cons**
- The thing E-1112 deliberately moved away from. Lost-on-drop: dropping a worktree without landing loses its verbs unless explicitly migrated.
- Requires a merge step in `worktree land`. JSON merge is straightforward (additive on `value` key) but is now load-bearing land logic.
- During parallel sessions, two worktrees independently add the same verb (waste, exactly the problem E-1112 solves).

### Option D: keep E-1112, add a JSON merge driver for the conflict case

Status quo + a custom git merge driver registered in `.gitattributes` that does additive JSON merge on `verbs.json` and `config.json`.

**Pros**
- Preserves E-1112's intent. Resolves the "two worktrees auto-commit conflicting JSON" failure mode at land time without manual intervention.
- Composes with anything else.

**Cons**
- Per-clone install (`.gitconfig` driver registration). Not portable to fresh checkouts unless `endless setup` installs it.
- Doesn't address the dirty-main-blocks-land symptom — auto-commit list still needed.

### Option E: keep E-1112, change events-log storage too (E-987 territory)

Status quo for verbs/config, AND move events log out of working tree (the Option D from E-987's analysis I struck through). Eliminates the *other* dirty-main contributor; auto-commit list shrinks to just config.json which auto-commits with negligible noise per E-1056.

**Pros**
- Cleanest separation: ambient session dirt (events) is no longer in the working tree at all; deliberate project state (verbs/config) stays in main and auto-commits.
- Restores `worktree land`'s clean-main precondition to a meaningful invariant.

**Cons**
- Events lose git history. Cross-machine sync of events log requires separate replication.
- Counter to E-901 ("`.endless/events/` must be committed to git, never gitignored"). Reopens that decision.

### Combinations worth considering

- **A + D**: out-of-tree storage and a merge driver for the seed file in case someone hand-edits both branches.
- **A + E**: clean both contributors (verbs + events) out of main's working tree; auto-commit list becomes empty.
- **B + D + retain E-1112's project-as-seed**: write only to machine, but project file stays as the read-time seed; merge driver guards the rare case it does change.
