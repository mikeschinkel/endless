## Background

`endless worktree land <task-id>` succeeds at the merge step but fails to remove the worktree dir. Reproduced on E-1186 land 2026-05-09 with:

```
fatal: '/Users/mikeschinkel/Projects/endless/.endless/worktrees/e-1186' contains modified or untracked files, use --force to delete it
```

Two untracked-by-design files in `.endless/` are the cause:
- `.endless/worktree.json` — companion file. Never committed because committing it would rotate every worktree's contents whenever another is created/destroyed.
- `.endless/worktree.lock` — ownership marker per E-971's lock model. SessionEnd hook is supposed to delete it on session end; in practice, it survives — observed alive at land time on E-1186.

Mike's chosen scope (confirmed in chat 2026-05-09): **fix both** layers — land-side cleanup AND the SessionEnd-hook-deletes-lock root cause. Don't just paper over the symptom.

### Phase 2 — Design (Plan agent — optional, but recommended given two fix sites)

Cover BOTH fixes:

**A. Land cleanup.** Make `endless worktree land` robust. Decide between (and explain tradeoffs):
- (a) Delete `.endless/worktree.{json,lock}` explicitly before `git worktree remove`
- (b) Pass `--force` to `git worktree remove` unconditionally

**B. SessionEnd lock deletion.** Make the lock reliably die on session end. Investigate and decide:
- Is the SessionEnd hook handler for lock deletion currently wired?
- If wired, why did it not fire for the E-1186 session (or did it fire but get re-claimed)?
- Stale-lock policy: what should land do if it finds a lock NOT owned by the current PID? Per E-971: `kill -0 <pid>` liveness check.
- Crash safety: SessionEnd doesn't fire on `kill -9`. The land path is the natural cleanup point even if SessionEnd works perfectly.

The two fixes are complementary: SessionEnd is the happy path; land cleanup is the safety net.
