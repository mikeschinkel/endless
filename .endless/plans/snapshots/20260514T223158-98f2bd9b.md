# E-1281 — Fix worktree sessions writing to global production DB

## Context

When Mike develops endless itself inside a per-task worktree (`.endless/worktrees/e-NNN/`), every `endless` command run from that worktree writes to the **real** production DB at `~/.config/endless/endless.db` — polluting Mike's actual task ledger with throwaway dev-time noise. Tests, sanity scripts, and CLI runs all contaminate live state.

The task brief distinguishes two roles for endless:

- **System-under-test role.** Developing endless itself → worktree DB writes should land in an isolated sandbox.
- **Tool role.** Developing acme-cli (or any other project) → worktree DB writes are real audit data and MUST land in the production DB.

Detection must be automatic — Mike shouldn't have to think about it.

The existing `endless-sandbox` family (E-995) already ships the env-var-based isolation primitive: setting `XDG_CONFIG_HOME=<sandbox-dir>` redirects both Python's `config.py:_config_root()` and Go's `monitor.ConfigDir()` to a sandbox location. The infrastructure exists; what's missing is **auto-engaging it when running inside an endless-self-dev worktree**.

E-1309 (route ledger commits to main checkout) is a sibling fix in `needs_plan`/unclaimed. The two work conceptually together — E-1281's sandbox supersedes E-1309 when sandbox is active (sandbox writes don't touch main at all). No conflict expected; E-1309 keeps the non-sandbox happy path safe.

## Current state (verified by exploration)

### Path-resolution chokepoints

- **Python**: `src/endless/config.py:8-17` — `_config_root()` reads `XDG_CONFIG_HOME`, falls back to `~/.config`. Module-level constants `CONFIG_DIR`, `CONFIG_FILE`, `DB_PATH` are computed at import time. **Single chokepoint** — every consumer imports from here.
- **Go**: `internal/monitor/db.go:20-43` — `ConfigDir()` reads `XDG_CONFIG_HOME`, falls back to `~/.config`. `DBPath()` returns `ConfigDir()/endless.db`. **Single chokepoint** for all Go binaries.

### Env vars already honored
| Var | Effect | Site |
|-----|--------|------|
| `XDG_CONFIG_HOME` | Redirects config/DB root in both Python and Go | `config.py:9`, `monitor/db.go:22` |
| `ENDLESS_SANDBOX` | Python CLI blocks unsafe subcommands when set; sandbox boundary guard | `cli.py:79` |
| `ENDLESS_AUTO_MIGRATE` | Controls auto-migration on DB open | `db.py:18` |

### Existing sandbox infrastructure (shipped, partially)
- `cmd/endless-sandbox/` — `run`, `enter`, `list`, `prune`, `destroy` subcommands.
- Sets `XDG_CONFIG_HOME=<sandbox-dir>` and `ENDLESS_SANDBOX=<sandbox-dir>` for child processes.
- Stores sandboxes in `~/.cache/endless/sandboxes/<name>/`.
- `--clone` is **deferred** (E-1087) — sandboxes start empty today.
- `os.Root` / `os.OpenRoot` filesystem chroot enforcement is in place.

### E-998 hook-override pattern
`just claude-settings-init` writes `<worktree>/.claude/settings.json` with hook commands pointing at `<worktree>/bin/endless-hook` (vs the global `/usr/local/bin/endless-hook` symlink). Detection: `git rev-parse --git-dir == --git-common-dir` → main checkout.

### Subprocess env inheritance
`event_bridge.emit_event` (Python → Go endless-event) uses `subprocess.run(cmd, ...)` with **no `env=`** override, so Go subprocesses inherit Python's environment. Setting `XDG_CONFIG_HOME` in Python propagates to all Go calls automatically.

## Verification plan (placeholder until design locked)

1. From inside `/path/to/endless/.endless/worktrees/e-XXX/`, `endless task list` does NOT read/write `~/.config/endless/endless.db`.

5. Verbs.jsonl writes from the worktree do not commit to main (E-1309 adjacent).
