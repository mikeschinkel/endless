# E-1281 — Fix worktree sessions writing to global production DB

## Context

When Mike develops endless itself inside a per-task worktree (`/path/to/endless/.endless/worktrees/e-NNN/`), every `endless` command run from that worktree writes to the real production DB at `~/.config/endless/endless.db` — polluting Mike's actual task ledger with throwaway dev-time noise.

The fix must distinguish two roles automatically:
- **System-under-test role.** Inside an endless worktree → use a sandbox DB.
- **Tool role.** Inside an acme-cli worktree (or any non-endless project's worktree) → use the real DB. This is real audit data.

Locked design choices (from planning conversation with Mike):

- **Architecture.** Endless ships a **generic sandbox primitive**. Endless source code itself does NOT auto-detect "am I a self-dev worktree" — that concern lives in the **Justfile / dev-workflow layer**, which can detect it via existing `git-dir` vs `git-common-dir` patterns. Ship-in-product-not-machine-config applied in reverse: dev-time concerns belong in the dev-time tooling, not in the shipped CLI.
- **Storage.** Sandbox lives at `~/.cache/endless/sandboxes/worktree-e-NNN/`, reusing the existing E-995 sandbox directory convention. Name is stable (`worktree-` + task ID).
- **Initial state.** Controlled by `--mode` flag on a new `endless sandbox init` subcommand: `empty` (default), `seed` (current task + parents), `clone` (full DB — deferred to E-1087). Different tasks have different needs.
- **Injection mechanism.** Wrapper scripts in `<worktree>/bin-sandbox/` that export `XDG_CONFIG_HOME=<sandbox-dir>` and exec the matching worktree-built binary in `<worktree>/bin/`. PATH-prepended via `<worktree>/.claude/settings.json` so Claude sessions inside the worktree pick them up automatically.
- **Migration.** Start fresh. Existing prod-DB pollution stays put; Mike cleans manually as he notices it. No migration script in scope.

## Current state (verified)

### Path-resolution chokepoints (both already honor `XDG_CONFIG_HOME`)

- Python: `src/endless/config.py:8-17` — `_config_root()` reads `XDG_CONFIG_HOME`, falls back to `~/.config`. Module-level constants computed at import time.
- Go: `internal/monitor/db.go:20-43` — `ConfigDir()` reads `XDG_CONFIG_HOME`, falls back to `~/.config`. All Go binaries flow through here.

### Existing sandbox machinery

- `cmd/endless-sandbox/` has `run`, `enter`, `list`, `prune`, `destroy`. Sets `XDG_CONFIG_HOME` and `ENDLESS_SANDBOX` for child processes. Storage at `~/.cache/endless/sandboxes/<name>/`. Boundary guard via `os.OpenRoot` in place. **Missing: `init` subcommand** (create without running/entering).
- E-1087 (deferred) will add `--clone` deep-copy.

### Subprocess env inheritance

`event_bridge.emit_event` uses `subprocess.run(cmd, ...)` with no `env=` override, so Go subprocesses inherit Python's environment. Setting `XDG_CONFIG_HOME` on the Claude-session shell propagates everywhere automatically.

### Worktree discrimination idiom

`git rev-parse --git-dir == --git-common-dir` → main checkout. Used in `Justfile:claude-settings-init` and `cmd/endless-hook/main.go:isInMainCheckout()`.

## Implementation

### Phase 1 — Add `endless sandbox init` subcommand

**File: `cmd/endless-sandbox/init.go` (new)**

```
endless sandbox init <name> [--mode {empty|seed|clone}] [--force]
```

Behavior:
- Provisions `~/.cache/endless/sandboxes/<name>/` with `.sandbox-meta.json` (mode=`persistent`, creator_pid, name, created_at).
- `--mode empty` (default): creates the directory, lets first DB access auto-migrate the empty schema. Same shape as the current ephemeral path.
- `--mode seed`: copies the current task row + its parent chain + the project row (with `path` rewritten to point at the worktree). Lazy implementation OK — can be a separate file `init_seed.go` that calls into existing project/task reads. **Note: seed mode could be deferred to a follow-up task if it complicates the critical path; default `empty` is enough to ship E-1281.**
- `--mode clone`: deferred. Print a clear error: `"--mode clone not yet implemented; see E-1087"`. Don't fall through to `empty`.
- Idempotent without `--force`: if `<name>` already exists, exit 0 with a notice. With `--force`, destroy and recreate.
- Prints the sandbox path to stdout on success (so Justfile recipes can capture it).

Wired into `cmd/endless-sandbox/main.go`'s subcommand dispatch alongside the existing `run`/`enter`/`list`/`prune`/`destroy`.

Python `endless sandbox init` group entry in `src/endless/sandbox_cmd.py` (or wherever the Click group lives — confirm during implementation) shells out to the Go binary, matching the existing pattern.

### Phase 2 — Add `just dev-sandbox-init` Justfile recipe

**File: `Justfile` (modify)**

Following the same pattern as `claude-settings-init` (refuse to run from main checkout; idempotent; absolute paths):

```just
dev-sandbox-init:
    #!/usr/bin/env bash
    set -euo pipefail
    git_dir="$(cd "$(git rev-parse --git-dir)" && pwd)"
    git_common_dir="$(cd "$(git rev-parse --git-common-dir)" && pwd)"
    if [ "$git_dir" = "$git_common_dir" ]; then
        echo "dev-sandbox-init: refuses to run from the main checkout. Run from a worktree." >&2
        exit 1
    fi
    worktree_root="$(pwd)"
    # Extract task ID from worktree path (.endless/worktrees/e-NNN/...)
    task_id="$(basename "$worktree_root" | sed -n 's/^e-\([0-9][0-9]*\).*/\1/p')"
    if [ -z "$task_id" ]; then
        echo "dev-sandbox-init: cannot parse task ID from worktree path $worktree_root" >&2
        exit 1
    fi
    sandbox_name="worktree-e-${task_id}"
    sandbox_path="$(endless sandbox init "$sandbox_name" --mode empty)"
    # Generate wrapper scripts in <worktree>/bin-sandbox/
    mkdir -p bin-sandbox
    for bin in endless endless-event endless-hook endless-channel endless-tmux endless-serve endless-sandbox; do
        real="$(command -v "$bin" || true)"
        if [ -z "$real" ]; then continue; fi
        # Prefer worktree-built binary if present (we're testing this code path)
        if [ -x "$worktree_root/bin/$bin" ]; then
            target="$worktree_root/bin/$bin"
        else
            target="$real"
        fi
        cat > "bin-sandbox/$bin" <<EOF
#!/usr/bin/env bash
export XDG_CONFIG_HOME="$sandbox_path"
export ENDLESS_SANDBOX="$sandbox_path"
exec "$target" "\$@"
EOF
        chmod +x "bin-sandbox/$bin"
    done
    echo "dev-sandbox-init: $sandbox_name at $sandbox_path; wrappers in $worktree_root/bin-sandbox/"
    echo
    echo "Next: run 'just claude-settings-init' to wire the wrappers into Claude sessions."
```

**Why a separate `bin-sandbox/` directory:** `<worktree>/bin/` already holds the worktree's built binaries (used by E-998's `claude-settings-init` to make Claude hook entries point at the worktree's `endless-hook`). Putting wrappers in the same dir would shadow them. `bin-sandbox/` keeps them distinct.

**Why wrappers exec worktree-built binaries (not the global symlink):** The whole point of the self-dev worktree is to test candidate code. If we ran the global binary against the sandbox, we'd isolate the data but not be testing the code under development. Wrapper picks worktree-built when available, falls back to global.

### Phase 3 — Extend `just claude-settings-init` to PATH-prepend bin-sandbox/

**File: `Justfile` (modify the existing claude-settings-init recipe)**

The existing recipe writes hook commands pointing at `<worktree>/bin/endless-hook`. Extend it to ALSO write a top-level `env` block in `.claude/settings.json`:

```json
{
  "env": {
    "PATH": "<worktree>/bin-sandbox:<worktree>/bin:${PATH}"
  },
  "hooks": { ... },
  "enabledPlugins": { ... }
}
```

This causes Claude-session-spawned shells (and subprocess calls Claude makes) to find the wrappers first. The wrappers export XDG_CONFIG_HOME and exec the underlying binary.

The recipe also needs to update each hook entry's `command` to point at `<worktree>/bin-sandbox/endless-hook` instead of `<worktree>/bin/endless-hook`, so the hook itself runs against the sandbox DB. (The wrapper then execs `<worktree>/bin/endless-hook`, which is still the worktree-built binary under test.)

### Phase 4 — Update CLAUDE.md worktree-setup section

**File: `CLAUDE.md` (modify)**

Current worktree setup is:
```

```

Add `just dev-sandbox-init` and reorder:
```

```

The README/CLAUDE.md text should explain WHAT this gives: "Claude sessions and any `endless` invocations from inside the worktree route to an isolated sandbox DB at `~/.cache/endless/sandboxes/worktree-e-NNN/`. The real production DB is left alone."

### Phase 5 — Cleanup hooks

**Concern:** When `endless worktree drop` or `endless worktree land` removes the worktree, the sandbox at `~/.cache/endless/sandboxes/worktree-e-NNN/` is left behind.

**Approach:** Out of scope for E-1281's critical path; file as a follow-up task linked with `--cleans-up E-1281`. Manual cleanup via `endless sandbox destroy worktree-e-NNN` works until the auto-cleanup ships. The existing `endless sandbox prune` will pick up orphans by stale `creator_pid` (per E-995 design).
