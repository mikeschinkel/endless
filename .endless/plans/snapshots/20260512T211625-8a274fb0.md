# E-1274 ‚Äî Fix spawn pre-binding session to task without creating worktree

## Context

`endless task spawn E-NNN` opens a new tmux window with Claude and pastes the task's prompt. Today it sets the tmux window option `@endless_task_id=NNN` *before* the spawned Claude starts. When the spawned Claude session starts, its `SessionStart` hook reads that tmux option and calls `monitor.StartWorkSession()`, which in a single Go function both (a) writes `sessions.active_task_id=NNN` and (b) runs `UPDATE tasks SET status='in_progress' WHERE id=NNN AND status IN (...)`. No events are emitted; the state is mutated by direct SQL.

So by the time the spawned session runs `endless task claim E-NNN` (per orchestration boilerplate), the session is already bound and the status is already flipped. `claim_item` calls `_check_task_ownership()` which returns True (self-bound), then short-circuits at `task_cmd.py:2134-2156` ‚Äî prints "E-NNN is already active in session N", creates the worktree (since E-1229 added that), and returns.

Two parts of the original spec are no longer accurate (state has moved since 2026-05-11):
- "no worktree" ‚Äî short-circuit *does* create the worktree as of E-1229.
- "status still ready/needs_plan" ‚Äî status *is* flipped, by `StartWorkSession`'s direct SQL.

What's actually left to fix:
- The "already active in session N" message reads as an error to the spawned Claude, which expected its `claim` to be a fresh action.
- State mutations bypass the event log: no `task.status_changed`, no `task.claimed` event is written for the spawn-then-claim flow.
- The "To work on this task: ..." help text from the full claim path is skipped.

Mike's chosen direction (from question 1): **spawn does the full claim ‚Äî status flip + worktree creation ‚Äî before launching Claude.** The spawned Claude lands in a worktree on a task that is already `in_progress`. The SessionStart auto-bind still sets `sessions.active_task_id` after the session starts; with status already flipped, its `UPDATE tasks SET status='in_progress'` becomes a no-op (the WHERE clause excludes `in_progress`). The full state-change audit trail now lives in events emitted by spawn, not in silent SQL.

## Approach

Python-only change. The work happens in `src/endless/task_cmd.py` and one paragraph in `docs/guide/orchestration.md`. No Go changes ‚Äî `StartWorkSession`'s status-flip behavior is preserved for non-spawn paths (e.g., a user manually setting `@endless_task_id` on a tmux window), where it remains the only mechanism that flips status.

### 1. Extract a shared claim-work helper

New private function in `task_cmd.py`, called by both `claim_item` and `spawn_plan`:

```python
def _perform_claim_work(
    item_id: int,
    title: str,
    current_status: str,
    target_session: int | None,
    force: bool,
) -> tuple[Path, bool]:
    """Emit claim events and create the worktree. Returns (wt_path, created).

    Does NOT resolve target_session ‚Äî caller passes it (None means no
    session binding, e.g., spawn before Claude starts).
    """
    # status flip if eligible
    if current_status != "in_progress":
        emit_event(
            kind="task.status_changed",
            project=proj_name,
            entity_type="task",
            entity_id=str(item_id),
            payload={"old_status": current_status, "new_status": "in_progress"},
        )
        _emit_field_changes(item_id, title,
                            [("status", current_status, "in_progress")])

    # bind to session if given
    if target_session is not None:
        emit_event(
            kind="task.claimed",
            project=proj_name,
            entity_type="task",
            entity_id=str(item_id),
            payload={"session_id": target_session},
        )

    # create worktree (idempotent)
    return create_task_worktree(item_id, title or "task", _project_root())
```

The done-ish-status gate (`_CLAIM_REQUIRES_FORCE`) and the multi-session ownership check (`_check_task_ownership` raise path) are kept in `claim_item` and replicated/called from `spawn_plan` ‚Äî they're guards that fire *before* this helper runs.

### 2. Refactor `claim_item` to use the helper

Existing public behavior preserved. The full path (no pre-existing binding) calls `_perform_claim_work(item_id, title, current_status, target_session, force)`. The short-circuit path (owned by current session AND status is `in_progress`) stays as-is ‚Äî friendly notice + ensure worktree exists. This keeps `claim` idempotent if the spawned Claude (or user) runs it after spawn already claimed.

### 3. Update `spawn_plan` to do the claim work before the tmux window

Insert the claim work between the prompt validation and the tmux window creation:

```python
# After: prompt validation, --worktree validation
# Before: tmux new-window

# Refuse if task is done-ish without --force (mirrors claim's gate)
if not force and current_status in _CLAIM_REQUIRES_FORCE:
    raise click.ClickException(...)

# Refuse if another live session owns the task
_check_task_ownership(item_id, current_eid=None)  # None = no self; only raises on other-owner

# Do the claim work ‚Äî no session yet (Claude hasn't started)
wt_path, _ = _perform_claim_work(
    item_id, title, current_status,
    target_session=None,
    force=force,
)

# Default cd_target to the worktree; --worktree still overrides
if worktree is None:
    cd_target = str(wt_path)
```

Add a `--force` flag to `spawn` (CLI option), mirroring `claim --force` for the done-ish-status demotion case.

`--worktree <path>` still overrides cd_target. When given, spawn still creates the canonical worktree (since the claim work demands it), but cds to the user-supplied path. (Alternative considered: skip canonical worktree creation when `--worktree` is given ‚Äî but that leaves the task half-claimed if `--worktree` points elsewhere. Better to keep the canonical worktree as the source of truth and let `--worktree` be purely a cd override.)

### 4. SessionStart auto-bind: no Go change

`monitor.StartWorkSession` continues to do both binding and (conditional) status flip. After spawn's pre-claim, the status is already `in_progress`, so the `UPDATE ... WHERE status IN ('needs_plan','ready','blocked')` clause excludes the row and the status flip is a no-op. The binding write (`sessions.active_task_id`) is the only effect ‚Äî which is what we want.

This means non-spawn paths that set `@endless_task_id` manually still get the full StartWorkSession behavior. That preserves the E-1027 invariant ("companion file's worktree_path reflects the bound task on first write") for those cases.
