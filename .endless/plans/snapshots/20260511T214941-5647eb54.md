# Plan: Per-session task-activity tracking and reporting

## Context

Mike keeps asking Claude "what did we do today?" / "what tasks did we file?" / "what's awaiting verification?" Mid-session and end-of-session, the examples in `examples/session-report-archetype-{1..5}.txt` show the shape of the reports he wants — tables of tasks added, claimed, shipped, confirmed, plus decisions filed and dependencies recorded — all scoped to "this session."

The information is already in the event ledger (`.endless/db-ledger/db-entries-*.jsonl`): every task mutation emits an event with a kind (`task.created`, `task.status_changed`, etc.) and a payload. **What's missing is the link from event → session.** Today the event `actor.id` is `$USER@$HOSTNAME`, which can't distinguish one session from another running as the same user.

This plan adds that link and builds a CLI/tmux-popup report on top.

## Answer to Mike's open question

> Are there any other ways we should keep track of sessions besides task add and task claim?

Yes — the full set of session-task activities worth surfacing in reports:

| Trigger | Event kind | Maps to which archetype |
|---|---|---|
| `task add` | `task.created` | archetype 1 ("three filed"), 2 (filed-but-not-implemented) |
| `task claim` / `task bind` | `task.claimed` | archetype 3 (worked on) |
| `task release` | `task.released` | archetype 3 |
| `task update --status verify/confirmed/assumed/declined/obsolete` | `task.status_changed` | archetype 2 (awaiting sign-off, at-assumed, obsolete), 3 (status column) |
| `task update --title/--text/--phase/--tier/...` | `task.fields_updated` | archetype 3, 4 (plan imported) |
| `task move` | `task.moved` | rarely surfaced; useful for "reorganized this in session" |
| `task remove` | `task.deleted` | "tasks removed this session" |
| `task block` / `task unblock` / `task link` / `task unlink` | `task_dep.created` / `task_dep.deleted` | archetype 4 ("blocked_by", "implements", "documents") |
| `decision add` | `task.created` with `type=decision` | archetype 4 ("Decisions linked to E-787") |

All of these emit events; recording session_id on the actor lets us filter and group.

Out of scope (still useful but not the headline): note-attached, plan-attached without status change, conversation messages (those are in `session_messages`, separate concern).

## Architecture

Two-phase implementation:

**Phase A — wire session_id through event emission.** Foundation.
**Phase B — surface as CLI verb + tmux menu item.** User-visible.

Phase A is the load-bearing change. Phase B is a thin layer over the event log.

### Phase A: extend `Actor` with `SessionID`

The `Event.Actor` struct (`internal/events/event.go:34-38`) today has only `Kind` and `ID`. Add an optional `SessionID` string field:

```go
type Actor struct {
    Kind      ActorKind
    ID        string
    SessionID string `json:",omitempty"`
}
```

Wire it through:

1. `endless-event` Go binary (`cmd/endless-event/main.go`) gains a `--session-id` flag that populates `Actor.SessionID`.
2. Python `emit_event` (`src/endless/event_bridge.py`) calls `_current_endless_session_id()` (already at `task_cmd.py:1921`) — when it returns a non-None value, pass it via `--session-id`.
3. Same import works in `task_cmd.py` since `_current_endless_session_id` is already there. For other Python modules that call `emit_event` (`session_cmd.py`, `worktree_cmd.py`, etc.) the resolver is re-imported.
4. Hook side (`cmd/endless-hook/claude.go`) already knows the session ID (it's the harness's session UUID from the payload) — pass it via the same flag.
5. `ActorKind` keeps its existing values; we do NOT change `actor.kind` to "session" — `cli`/`hook`/`web` remain accurate descriptions of WHERE the event came from. The new `SessionID` field separately captures WHICH session it was part of.

Existing events in the ledger with no `session_id` are left untouched — reports filter and skip them.

### Phase B: report command

New CLI verb: `endless session activity [<session-ref>] [--since <when>] [--kinds <list>] [--json]`.

- Default `<session-ref>`: current session (via `_current_endless_session_id`).
- `--since`: filter events newer than the given timestamp / relative ("1h", "today", session-start).
- `--kinds`: comma-separated subset of event kinds to include (default: all task-touching kinds).
- `--json`: machine-readable output for tools and tests.

Output groups events by category, with task ID / title / current status. Roughly the shape of archetype 2:

```
Session ES-356 — endless project — started 2026-05-11T05:15
  Filed (3): E-1278, E-1279, E-1280
  Claimed (5): E-1256, E-1257, E-1258, E-1261, E-1262
  Shipped → verify (4): ...
  Confirmed (8): ...
  Dependencies set: 2
  Decisions filed: 1 (E-1281 documents E-1278)
```

### Reading events back

Events are read from `.endless/db-ledger/*.jsonl` segment files. There's already a reader in `internal/events/` used by `endless-event validate-db` (`internal/events/projection.go` or similar — the agent found it indirectly). The activity command iterates segment files for the project root, filters by `actor.session_id == <target>`, optionally by timestamp range, then formats.

No new SQL table is introduced. The event log is the source of truth; queries are projections.

If performance becomes an issue once a session has thousands of events, the follow-up is a `session_activity` materialized view — but that's a future child task, not this slice.

### Tmux integration

Add a menu item "Session Activity" to the prefix+e / right-click menu (in `cmd/endless-tmux/show_menu.go`), invoking `tmux display-popup -E 'endless session activity | less'`. Hotkey letter `a` (currently free in the menu).

## Existing utilities to reuse

- `_current_endless_session_id()` in `src/endless/task_cmd.py:1921` — session-ID resolver (env var + TMUX_PANE fallback). Don't re-implement.
- The Kairos timestamp format (`internal/kairos/`) — already used by every event; filtering by "since" needs to parse this format.
- The session row schema (`sessions.id`, `sessions.session_id`) — task IDs in payloads are integers; session IDs in actors will be the `sessions.id` integer for consistency with Endless's internal ID convention. Display as `ES-NNN` per E-1261's planned naming.
- Existing `endless session show` rendering style (`src/endless/session_cmd.py:105+`) — match its table formatting for visual consistency.

## Verification

End-to-end:

1. **Plumbing test:** in a Claude session with `ENDLESS_SESSION_ID` set, run `endless task add "Test task"`. Read the most recent line in `.endless/db-ledger/db-entries-*.jsonl` and confirm `actor.session_id` matches the env var.
2. **CLI test:** `endless task add` / `endless task claim` / `endless task update --status verify` a sequence of test tasks. Run `endless session activity` — confirm each appears under the correct group (filed / claimed / shipped).
3. **Other-session isolation:** start a second Claude session with a different `ENDLESS_SESSION_ID`; run `endless task add` from it. Confirm the first session's `activity` does NOT show the new task; the second session's `activity` DOES.
4. **Historical events:** confirm that events emitted before this change (no `session_id` on actor) are silently skipped and don't error.
5. **Tmux menu:** prefix+e → `a` opens the activity report in a `less` popup.
6. **JSON output:** `endless session activity --json` produces parseable JSON matching the table groupings.
7. **`endless session activity --kinds task.created`** filters to just filed tasks.

## Out of scope (deferred to child tasks)

- **Materialized `session_activity` table** — only needed if event-log scanning becomes slow.
- **Auto-recap at session end** — could include the activity report; separate concern.
- **Note-to-task linkage** — notes today aren't directly linked to sessions or tasks (per the explore agent's map).
- **Conversation message correlation** — tying activity events to specific conversation turns. The transcript is in `session_messages`; that's a join for another day.
- **Per-window aggregation** — `tmux window` × `session` rollups for "what's happening across windows." Separate concern.
- **Backfill historical events** — pre-existing ledger entries stay session_id-less.
- **E-1262 (retain session-to-task history)** — partially overlapping. After this lands, E-1262 can probably be `declined` or scoped down to just "preserve `active_task_id` snapshot after task confirm" since the activity log captures the same information.

## Sequencing

Implement Phase A first (one task — plumbing), verify the ledger captures session_id correctly. Then Phase B (one task — CLI + menu). Each is roughly half-day-sized.

File two tasks under the existing E-1031 (Session Focus) parent — both fit there since they're "session-level metadata" features. Phase B as a child of Phase A so deps are clean.
