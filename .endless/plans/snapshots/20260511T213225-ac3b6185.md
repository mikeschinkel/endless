# E-1208 — Record verbs.jsonl changes immediately on endless verb add

## Context

Today `endless verb add <name>` (and the auto-registration path from `task_cmd.validate_title`) writes a new entry into `<project>/.endless/verbs.jsonl` but does **not** commit. The file is left dirty in the working tree until either someone manually commits or the next `endless worktree land` bundles it into the auto-record commit.

Both outcomes are wrong:
- Manual-commit creates a coordination tax across sessions.
- Bundling at land couples a global-config event (registering a verb) to an unrelated worktree-land timing.

Per E-987's resolution, **write-time auto-commit** is the correct pattern for endless-managed files. Today's dirt was visible in this session: `.endless/verbs.jsonl` is modified on main because the in-flight `task add` calls auto-registered the verbs `narrow` and `ensure`.

The task description (and E-1199's global-config exception) authorize **direct commit to main, bypassing the task/worktree association**: the verb registry is global config, and the commit travels with main regardless of which working tree the calling session was in.

## What's actually being changed

Two behavior changes ride together:

1. **Route the project verbs.jsonl write to main's path.** Today `matchers.project_verbs_path()` walks up from cwd, so a worktree session writes to the worktree's `.endless/verbs.jsonl`. The replacement walks the registered-project record (the same path resolution `worktree_cmd._project_root()` already does) so writes always target main's tree, regardless of cwd.
2. **Commit the file immediately after a successful write.** Single-file `git add` + `git commit` on main's repo. Skipped if the write was a no-op (verb already present). Machine-layer writes are not touched — `~/.config/endless/verbs.jsonl` is not git-tracked.

Together these mean: a verb registration from any session lands as one commit on main with a message like `Endless: register verb 'foo'`, and the next session in any worktree picks it up via the read path (which also routes to main).

## Failure handling

The file write happens first, then the commit. Three observable failure modes:

| Failure                                              | Response                                               |
|------------------------------------------------------|--------------------------------------------------------|
| `git add` fails (e.g., no .git, no permissions)      | Warn to stderr, return; verb stays persisted in file. |
| `git commit` fails (e.g., pre-commit hook rejects)   | Warn to stderr, return; verb stays persisted in file. Working tree stays dirty. |
| Concurrent writer raced us between write and commit  | Acceptable: `git add` only stages the path we name; the simultaneous writer's path gets staged on their `git add`. Both commits happen; merge=union on verbs.jsonl absorbs textual overlap if needed. |

No rollback on failure: the file has the new verb, even uncommitted. The next CLI invocation that auto-commits (this task post-land) will pick it up. Pre-land, the existing `worktree_cmd._dedup_worktree_verbs_against_main` is a backstop.

## Open questions

1. **Helper location.** `matchers.add_verb` needs `_project_root()` and `_git_run()` from `worktree_cmd`. Three options:
   - **A.** Import them directly into matchers. Creates a forward dependency `matchers → worktree_cmd`. Existing reverse imports are minimal (`worktree_cmd` imports from `task_cmd`, not matchers), so probably no cycle. Lowest churn.
   - **B.** Lift both helpers into a new `src/endless/git_util.py` module. Cleaner separation; small new file. Both `matchers` and `worktree_cmd` import from `git_util`.
   - **C.** Keep `_project_root` and `_git_run` in `worktree_cmd` but expose public re-exports. Half-measure.
2. **Auto-registration noise.** When `task add` auto-registers a verb (E-1264), it currently prints `• Auto-registered verb 'foo': definition...`. With E-1208, an additional commit happens silently. Do we also surface the commit in the output (e.g., `• Auto-registered verb 'foo' (committed to main): definition...`)?
3. **No-registered-project case.** If a verb is added from a directory that has no registered endless project (e.g., the seed-verbs case at first install, or a manual `endless verb add` in a non-project dir), today the machine-layer write succeeds and project_verbs_path() returns `None`. Under the new design, the same behavior: machine-only write, no commit. Confirm this is correct.

## Verification

1. **Happy path**: From a clean main checkout, run `endless verb add ponder --definition "to deliberate over"`. Expect: `.endless/verbs.jsonl` has the new entry, working tree is clean, `git log -1` shows `Endless: register verb 'ponder'`.
2. **Happy path from worktree**: From inside `.endless/worktrees/e-1208/`, run `endless verb add musing --definition "..."`. Expect: main's verbs.jsonl (NOT the worktree's) has the new entry; main's HEAD has the new commit; the worktree's `.endless/verbs.jsonl` is unchanged on disk (will pick up the change via the next `git pull` / rebase).
3. **No-op**: Run `endless verb add ponder --definition "..."` twice. Second call: no commit, no file change, exits with the "already present" message.
4. **Machine-only**: Run `endless verb add foo --machine-only --definition "..."`. Expect: machine layer updated, no commit on main, project file unchanged.
5. **Auto-registration path**: Run `endless task add "Mull over the design"` (assuming "mull" is not registered). Expect: haiku auto-registers `mull`, single commit on main with the verb registration, then the task add proceeds.
6. **Tests**: `just test` — new tests cover the four scenarios above. Existing tests unaffected (they use machine_only=True or check file-only semantics).

## Out of scope

- Removing the `_dedup_worktree_verbs_against_main` helper. Once E-1208 ships, the worktree should never have a divergent verbs.jsonl (writes go to main directly). The dedup helper becomes dead code, but removal lands with E-1272 (the bundled `land` change) where the entire auto-commit step is dropped.
- Reading from main when in a worktree (for non-verbs files). Out of scope; this is just verbs.
- File-write atomicity (lock-then-write). The existing read-modify-write pattern is unchanged; merge=union backstops textual races during rebase.
