# E-1248: Rename `quick-start` → `guide`, restructure as multi-section guide, revise content

## Context

`endless quick-start` echoes a markdown file that onboards a Claude Code session to working with Endless. Three problems:

1. **Verb is wrong.** `quick-start` implies a brief intro, but the doc is the canonical session reference. `guide` is honest about what comes out, no semantic mismatch with CLI conventions (unlike `info`, which conventionally means runtime/system state).
2. **Doc is stale.** Written 2026-04-15; today is 2026-05-11. ~50+ session-facing tasks have shipped since — `task spawn`/`task prompt`, the per-task worktree model (E-1168), `task start` → `task claim` rename (E-1232), commit-to-main policy (E-1199/1200), tmux integration (E-1236), decisions as first-class items (E-907/981), and more. The date-stamped filename implies a snapshot when the file is meant to be evergreen.
3. **Single dump doesn't scale.** As coverage grows, dumping the whole guide on every cold session is wasteful. A section-addressable form (`endless guide <section>`) lets agents and humans pull just what they need.

Outcome: the command becomes `endless guide [<section>]`, content lives in `docs/GUIDE.md` (top-level index) plus `docs/guide/<section>.md` files, and the content reflects what a session needs to know on 2026-05-11.

## Command surface change

`endless guide` (no arg) → prints `docs/GUIDE.md` (the index — short overview + list of sections, each line ≈ `<section>: <one-line summary>` so the agent can pick).

`endless guide <section>` → prints `docs/guide/<section>.md` (raises a clean error listing valid sections if unknown).

`endless guide --list` → prints just the available section slugs (cheap for an agent to check).

Implementation: `src/endless/cli.py` — `@main.command("guide")` taking optional positional `section`. Reads from `docs/guide/<section>.md` when given, else `docs/GUIDE.md`. Unknown section: enumerate `docs/guide/*.md` and raise `ClickException` with the list.

## Field distinctions (revised — `fields` section content)

Cover all task body fields:
- `title` — one-liner; verb-first per E-723/947.
- `description` — brief blurb (<200 words), shown by default in list/show (E-1058).
- `text` — full implementation plan. Set with `--text <file>`. Shown with `task show --text`.
- `prompt` — task-specific LLM prompt (what a spawned session sees). Set with `--prompt <file>`. Shown with `task show --prompt` or printed raw with `task prompt <id>`.
- `analysis` — supporting content developed via research/exploration that is not a proper plan. Lifecycle and rules per E-1073.
- `notes` — freeform content that doesn't fit elsewhere. Use sparingly.
- `outcome` — required at terminal statuses (`confirm`, `assume`) per E-1021/1022/1175. For `decline` use `--reason`.

Verify field names exist before publishing by reading `endless task add --help`, `endless task update --help`, and the schema. Don't trust the doc draft against memory — read the live CLI.

## Decisions section (revised — STRONG instructions)

Lead the `decisions` section with explicit guidance to sessions:

> **Always document decisions where applicable.** When you and Mike resolve a non-obvious question — choosing between approaches, scoping in/out, picking conventions — file it. Use `endless task add --type decision` or attach inline via `--decision <text>` on the originating task.
>
> **Verify with Mike before stating a decision as binding.** Especially: do not record a *prohibition* when Mike only expressed a *preference*. "Ideally we'd …" / "usually we …" / "I'd prefer …" are soft signals. Confirm the rule and its scope before writing a decision. When in doubt, ask.
>
> Inline rationale via `task add ... --decision "<rationale>"` (E-981) is preferred over manually creating a separate decision task — it pairs the decision with the task that prompted it via a `documents` link.

Then list the mechanics: `endless decision list`, `endless decision link`, `endless task add --type decision`, the `--decision` flag.
