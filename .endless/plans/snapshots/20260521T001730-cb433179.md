# E-1337: Stop deleting worktrees at land; record landings in DB; reap by sweep

## Context

Today `endless worktree land` (`src/endless/worktree_cmd.py:1084`) calls `git worktree remove` to delete the worktree directory as the final step of land. That deletion happens out from under any Claude session whose cwd is inside the worktree — the dir disappears, the session breaks. It was first observed during E-1281's land (dir recovered from IntelliJ local history); E-1372 ("SessionStart did not re-register after worktree-delete") is a downstream symptom.

Root cause: cleanup at land time was incidental, not load-bearing. The fix is to stop deleting at land. Land becomes append-only history (rebase + ff-merge + INSERT). A separate reaper sweep handles eventual cleanup, gated on (a) the worktree's task has at least one landing record older than a TTL, and (b) no live process has cwd inside the dir. The branch and its tip refs stay until reap, which keeps the door open for re-landing (post-land bug fix on the same branch) and a future undo feature.

## Schema (migration V10)

```sql
CREATE TABLE IF NOT EXISTS task_landings (
    id               INTEGER PRIMARY KEY,
    task_id          INTEGER NOT NULL,
    session_id       INTEGER,
    branch           TEXT    NOT NULL,
    merge_commit_sha TEXT    NOT NULL,
    landed_at        TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%S', 'now')),
    FOREIGN KEY (task_id)    REFERENCES tasks(id)    ON DELETE CASCADE,
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE SET NULL
);
CREATE INDEX IF NOT EXISTS idx_task_landings_task ON task_landings(task_id, landed_at DESC);
```

Deviations from the original prompt's schema, with reasons:

- **`task_id` is `INTEGER`, not `TEXT`.** `tasks.id` is `INTEGER PRIMARY KEY` (`schema.sql:74-75`). FK type must match. The "E-NNNN" form is the user-facing display only.
- **`session_id` is nullable** and on `ON DELETE SET NULL`. The session that landed may eventually age out; the landing record should survive. (Mirrors `sessions.active_task_id` ON DELETE SET NULL at `schema.sql:70`.)
- **`landed_at` is `TEXT`** with the `strftime` default, matching the project-wide ISO-8601-as-TEXT convention used in every other table (`schema.sql:21,45,60,...`), not `TIMESTAMP`.
- **Index on `(task_id, landed_at DESC)`** — the reaper's primary query is `MAX(landed_at) WHERE task_id = ?`.

Migration follows the V6/V9 pattern: idempotent `CREATE TABLE IF NOT EXISTS`, no down migration, `hasTable` guard not strictly necessary with `IF NOT EXISTS` but matches house style if added.

## Land flow rewrite (`worktree_cmd.py`)

Keep Steps 1-5 as-is. **Remove Step 5.5 entirely** (the live-session refusal at lines 1059-1068 actively blocks the feature: when land is invoked from a Claude session whose cwd is the worktree, that session's own PID owns the lock — it would refuse itself.) **Remove Step 6** (the `git worktree remove` at 1084 and `git branch -d` at 1094).

Replace with:

```python
# Step 6 (new): record the landing. Worktree, branch, and refs all stay.
merge_sha = _git_capture(["rev-parse", "HEAD"], cwd=main_root).strip()
session_id = os.environ.get("ENDLESS_SESSION_ID", "")
_endless_db_invoke(
    "landings",
    "record",
    "--task-id", canonical,
    "--branch", branch,
    "--merge-sha", merge_sha,
    "--session-id", session_id,  # may be empty; the Go side handles ""→NULL
)
click.echo(click.style("•", fg="green") + f" Landed {canonical} ({branch}) into {base_branch}")
return
```

Notes:
- `_endless_db_invoke` is a thin subprocess helper (or use an existing one — check whether `worktree_cmd.py` already has a Go-shell-out pattern; if so, reuse). The new Go subcommand parses "E-NNNN" → integer task_id by stripping the prefix and verifying the row exists.
- No state-file cleanup (`.endless/worktree.json`, `.endless/worktree.lock`). They're gitignored already (E-1218); they become stale once landed and the reaper removes them when it reaps the dir.
- The `for attempt in range(...)` retry loop wraps everything; on second attempt (re-land scenario after a follow-up commit), `_dedup_worktree_verbs_against_main`, `_drop_orphan_amendable_commits`, rebase, ff-merge all still run — each landing is idempotent at the record-insertion level (a new row, not an UPSERT).

## Reaper module

`internal/monitor/reap_worktrees.go`, paralleling `reap.go`'s style:

```go
func ReapStaleWorktrees(projectID int64, projectRoot string, ttl time.Duration) error
```

Per-directory logic (`<projectRoot>/.endless/worktrees/*`):
1. Parse dir name `e-NNNN` → integer task_id (skip if unparseable; pre-existing orphan dirs from before E-1337 land here).
2. `SELECT MAX(landed_at), branch FROM task_landings WHERE task_id = ?`. If no rows → in-progress task, skip.
3. Parse `landed_at` as ISO-8601; if `now - landed_at < ttl` → skip.
4. Check `lsof -d cwd +D <dir>` for live processes (see filter below). Any process with cwd inside the dir → skip.
5. Reap: `git worktree remove --force <dir>`, then `git branch -D <branch>` (use `-D` not `-d`; the branch has been merged into main but Git may not realize it post-rebase). Log each reap.

**lsof filtering** (E-1337 prompt §"lsof caveat"):
```go
cmd := exec.Command("lsof", "-d", "cwd", "+D", dir)
var stderr bytes.Buffer
cmd.Stderr = &stderr
out, err := cmd.Output()
// lsof returns exit 1 when no matching open files exist — that's the "nothing
// holds it" success case for us. Other non-zero exits (binary missing, permission)
// are real errors. Discriminate via exit code, not stderr presence; warnings about
// unrelated filesystems (smbfs, /Volumes/.timemachine/...) are pure noise.
if exitErr, ok := err.(*exec.ExitError); ok && exitErr.ExitCode() == 1 && len(out) == 0 {
    return false, nil  // no processes with cwd in dir → safe to reap
}
```

**Pre-existing orphan dirs**: dirs with no rows in `task_landings` are explicitly skipped (step 2). Log them once per sweep at info level so they don't accumulate silently; cleanup is manual.

## Sweep wiring

Per the prompt's stated trigger list:

- `cmd/endless-hook/claude.go` SessionStart handler (line 155, alongside `ReapDeadTmuxPanes`) — same once-per-session cadence the existing reaper uses. No throttle needed.
- `cmd/endless-hook/claude.go` PreToolUse / PostToolUse handlers — these fire hundreds of times per session. Add a **per-process throttle**: skip sweep unless `time.Since(lastSweep) > 60*time.Second`. The reaper is idempotent so throttling has no correctness cost; it just keeps PreToolUse latency bounded.
- `src/endless/task_cmd.py` `claim_task` and `spawn_plan` — invoke after worktree creation succeeds. Shell out to a Go subcommand (`endless reap worktrees` or similar), best-effort, errors logged not raised.
- `src/endless/worktree_cmd.py` `land_worktree` — invoke at the very end, after the landing record is written.

Implementation detail: the throttle state can live in the hook process's memory (a package-level `var lastSweep time.Time`). The hook binary is short-lived per event, so the throttle survives only within a single invocation. To throttle across invocations, write a sentinel file `<projectRoot>/.endless/.last-worktree-sweep` and check its mtime; simpler and no IPC needed.

## Configuration

`<project>/.endless/config.json` gains a `worktree_ttl` field. Format: **duration string** with day/hour/minute/second suffixes: `"14d"`, `"24h"`, `"30m"`, `"3600s"`. Default if absent: `"14d"`.

Parsing in Go: `time.ParseDuration` does NOT support `"d"`. Write a small `parseTTL(s string) (time.Duration, error)` helper that handles `Nd` by converting to `N*24h` then delegating to `time.ParseDuration`. Place it next to the reaper.

Parsing in Python (for any Python-side display): not needed initially — only the Go reaper consumes it. If the Python CLI ever needs to render the TTL (e.g., `endless worktree status` showing "reaps in 3 days"), a sibling Python parser can be added.

Reading `.endless/config.json` follows the existing pattern at `src/endless/config.py:95-114` (`project_config_read`). Go side: there isn't a single Go config-reader yet; cheapest path is `os.ReadFile` + `json.Unmarshal` into a small struct, since this is a single config field.

## Test plan

**Unit (Go, `internal/monitor/reap_worktrees_test.go`):**
- Worktree with `MAX(landed_at)` younger than TTL → not reaped.
- Worktree with `MAX(landed_at)` older than TTL and no live process → reaped (mocked `lsof`).
- Worktree older than TTL but `lsof` finds a process → not reaped.
- Worktree dir with no rows in `task_landings` → not reaped (pre-existing orphan).
- `lsof` exits 1 with stderr warnings about unrelated filesystems → treated as "no live processes", reaper proceeds.
- TTL parsing: `"14d"`, `"24h"`, `"30m"`, `"7d12h"` (combo), invalid `"forever"` rejected.

**Unit (Go, `internal/events/...` or wherever the new insert helper lives):**
- Insert with valid task_id + session_id → row written, FK constraints pass.
- Insert with non-existent task_id → FK error surfaced (not silently dropped).
- Insert with empty session_id → row written with NULL session_id.

**Integration (Python, `tests/...` mirroring existing worktree_cmd tests):**
- Land happy path: from a worktree, run `endless worktree land E-NNN`. Assert: dir still exists; branch still exists in `git branch`; `git worktree list` includes it; `task_landings` has one row matching branch + merge SHA.
- Re-land: in the same worktree, add another commit, run land again. Assert: two rows in `task_landings` for the same task_id; first row preserved; second row's merge_commit_sha differs.
- Land while sitting in the worktree's cwd (the bug scenario): assert the active session's cwd resolves before and after; no exception; no SessionEnd fires.

**End-to-end (manual, documented in the verify handoff):**
- From a Claude session whose cwd is `.endless/worktrees/e-NNNN`, run `endless worktree land E-NNNN`. Confirm:
  1. Land succeeds.
  2. `pwd` still resolves.
  3. `ls .endless/worktrees/e-NNNN` still lists the dir contents.
  4. `git -C ../../.. worktree list` includes the worktree.
  5. `endless task detail E-NNNN` shows the landing.
- Set `worktree_ttl` to `"60s"` in `.endless/config.json` for the test project, wait 60 seconds, fire any hook event (or run `endless task claim` on a different task). Confirm the previously-landed worktree dir is removed and its branch is gone.

## Open design questions for review

1. **Sweep trigger surface**: prompt lists hook firings + `claim` + `spawn` + `land`. My proposal throttles hook-firing sweeps to one per 60s (per project) via a sentinel file. Acceptable, or should hook-firing sweeps run only on SessionStart (matching the existing tmux reaper exactly)? The latter is simpler.
2. **Reaper invocation surface**: should the reaper be exposed as `endless reap worktrees` (user-visible CLI) so it can be manually triggered, or kept internal-only (called by the hook)? A user-visible verb makes debugging trivial.
3. **State-file cleanup**: post-land, `.endless/worktree.json` and `.endless/worktree.lock` inside the landed worktree become stale. They're gitignored, so harmless on disk, and the reaper removes them with the dir. Confirm leaving them in place is fine.
