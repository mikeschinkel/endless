# E-971 Layer D — Worktree adoption hooks

## Context

E-971 ("worktrees and the behavioral gate") gives every editing session its own git worktree so parallel Claude sessions stop colliding on shared files. Foundation slice (`endless worktree {list, current, show, for-task}`) and mutation slice (`land`, `drop`) are already in main. What's missing is the runtime that *binds* sessions to worktrees — without it, sessions still launch in main, edit there, and the per-session isolation never happens.

This slice is **Layer D only**: three additions to `cmd/endless-hook/claude.go` plus the lock primitives that make ownership transfer safe. Layer E (UserPromptSubmit gate) and Layer F (`task start` worktree creation) are explicitly out of scope and will ship in their own slices.

The deliverable is the dogfood test: starting Claude in main creates a session worktree, starting Claude in an adopted worktree claims its lock, and edits from main are blocked at the hook.

## Scope

In scope:

1. SessionStart hook: walk-up adoption + per-session worktree creation + lock claim.
2. PreToolUse hook: extend the existing E-1012/E-917 enforcement with worktree-aware blocks for Write/Edit/NotebookEdit.
3. SessionEnd hook: release any worktree lock claimed by this session.
4. Lock-file primitives in `internal/monitor/worktree_lock.go`: claim, read, stale-check, release.
5. Walk-up helper to find `.endless/worktree.json` from a cwd, bounded at the registered project root.

Out of scope (do not touch in this slice):

- UserPromptSubmit phrase-gate (E-971 Layer E).
- `endless task start` triggering task-flavor worktree creation (E-971 Layer F).
- Any change to `endless worktree land/drop` (already shipped).
- DB schema changes — worktree state stays filesystem-authoritative.
- New CLI verbs.

## Companion-file format

Both flavors live at `<worktree-root>/.endless/worktree.json`. Existing Python code (`src/endless/worktree_cmd.py:42-127`) reads it and only requires `task_id` for landing. Adding a `flavor` field is forward-compatible: the existing code's `_classify` returns "active" whenever the companion exists, regardless of fields, and `for_task` filters by `task_id`, which session-flavor entries lack — so they're correctly skipped during land.

**Task flavor** (already written by future Layer F; documented here for shape parity):
```json
{
  "flavor": "task",
  "task_id": "E-808",
  "base_branch": "main",
  "branch": "task/808-event-logs-authoritative",
  "created_at": "2026-05-01T15:30:00Z"
}
```

**Session flavor** (this slice writes it):
```json
{
  "flavor": "session",
  "session_id": "<harness session uuid>",
  "base_branch": "main",
  "branch": "session/<short-id>",
  "created_at": "2026-05-01T15:30:00Z"
}
```

Where `<short-id>` is the first 8 chars of `payload.SessionID`.

## Lock-file primitives — `internal/monitor/worktree_lock.go` (NEW)

Mirrors the style of `internal/monitor/companion.go` (root-relative paths, atomic writes). A separate file keeps the worktree-ownership concern from bloating `companion.go`.

**Lock path:** `<worktree-root>/.endless/worktree.lock`

**Lock contents (JSON):**
```json
{
  "session_id": "<harness session uuid>",
  "pid": 12345,
  "tmux_pane": "%17",
  "claimed_at": "2026-05-01T15:30:00Z"
}
```

**API:**

```go
type WorktreeLock struct {
    SessionID  string `json:"session_id"`
    PID        int    `json:"pid"`
    TmuxPane   string `json:"tmux_pane,omitempty"`
    ClaimedAt  string `json:"claimed_at"`
}

// ClaimWorktreeLock atomically creates the lock file. Returns
// os.ErrExist if a lock already exists. Caller checks for staleness
// via ReadWorktreeLock + IsWorktreeLockStale before retrying.
func ClaimWorktreeLock(worktreePath string, lock WorktreeLock) error

// ReadWorktreeLock reads and parses the lock file. Returns nil,
// os.ErrNotExist if absent.
func ReadWorktreeLock(worktreePath string) (*WorktreeLock, error)

// IsWorktreeLockStale returns true if lock.PID is no longer alive.
// Uses syscall.Kill(pid, 0) — sends signal 0, which checks process
// existence without actually signaling. Treats PID 0/negative as stale.
func IsWorktreeLockStale(lock *WorktreeLock) bool

// ReleaseWorktreeLock deletes the lock file. Idempotent: missing
// file is not an error.
func ReleaseWorktreeLock(worktreePath string) error

// FindLockBySessionID scans <project_root>/.endless/worktrees/*/.endless/worktree.lock
// and returns the worktree path whose lock has the given session_id, or
// "" if none. Used by SessionEnd to release locks even when cwd has
// moved away from the worktree (or the user never cd'd in).
func FindLockBySessionID(projectID int64, sessionID string) (string, error)
```

**Atomic claim:** use `os.OpenFile(path, O_WRONLY|O_CREATE|O_EXCL, 0644)`. POSIX guarantees `O_EXCL` is atomic across processes. On success, write JSON and `Close`. On `EEXIST`, return `os.ErrExist`.

**Stale check (Unix only — endless is Mac/Linux):** `syscall.Kill(pid, 0)`. Returns `nil` if alive, `ESRCH` if gone, `EPERM` if alive but not signal-able from this user (treat EPERM as alive — process exists). On Windows we'd need `OpenProcess`, but endless is not targeting Windows; document this in a comment.

**Reclaiming a stale lock:** SessionStart logic does the read-check-delete-claim dance. There is a TOCTOU window between "lock looks stale" and "I overwrote it" where two sessions could both observe staleness and race. We accept this risk — the worst case is two sessions both think they own the same worktree, which the next PreToolUse will catch via a divergent `session_id` mismatch IF we check it (we don't currently — just check existence). For v1, the brief explicitly does not require fencing beyond `O_EXCL`; document the TOCTOU as known.

## Walk-up helper — extension to `internal/monitor/worktree_lock.go`

```go
// FindWorktreeRoot walks up from cwd looking for .endless/worktree.json.
// Stops at the registered project root (above which we are no longer in
// this project). Returns the worktree directory path (the dir containing
// .endless/worktree.json), or "" if none found before reaching project
// root. projectRoot must be the absolute, resolved path.
func FindWorktreeRoot(cwd, projectRoot string) (string, error)
```

Implementation: `filepath.Clean(cwd)`; loop `Stat(filepath.Join(dir, ".endless", "worktree.json"))`; if found, return dir; otherwise `dir = filepath.Dir(dir)`; stop when `dir == projectRoot` or `dir == filepath.Dir(dir)` (filesystem root).

Note: do NOT classify "is in main checkout" via this helper. The existing `isInMainCheckout(cwd)` in `cmd/endless-hook/claude.go:623` already handles that via `git rev-parse --git-dir` vs `--git-common-dir`. Use both: walk-up tells you whether there's a worktree companion; `isInMainCheckout` tells you which side of the main/worktree boundary you're on.

## SessionStart extension — `cmd/endless-hook/claude.go:104-127`

After the existing block (InitSession, SetProcess, transcript path, tmuxTaskID, writeClaudeCompanion), and before `handleTaskContextInjection`, insert worktree resolution:

```
1. projectRoot := monitor.ProjectPath(projectID)  // already available via projectID
2. worktreePath := monitor.FindWorktreeRoot(payload.CWD, projectRoot)
3. branch by case:
   case A: worktreePath != "" (cwd is inside a worktree):
       lock, err := monitor.ReadWorktreeLock(worktreePath)
       if err == os.ErrNotExist:
           claim it; proceed
       else if err == nil:
           if IsWorktreeLockStale(lock):
               ReleaseWorktreeLock(worktreePath); claim it; proceed
           else if lock.SessionID == payload.SessionID:
               // Re-entry of same session — idempotent, do nothing
               proceed
           else:
               return hook response refusing registration with message:
                 "This worktree is owned by session <id> (PID <n>).
                  Start a new session in a different worktree, or wait
                  for the owner to release."
               // Do NOT exit 2: SessionStart isn't a tool-use, exit 2
               // semantics don't apply. Instead, return additionalContext
               // and let the session continue without a worktree binding.
               // (PreToolUse will subsequently block edits.)
   case B: worktreePath == "" AND isInMainCheckout(cwd):
       // Create a per-session worktree.
       shortID := payload.SessionID[:8]
       branch := "session/" + shortID
       wtDir := filepath.Join(projectRoot, ".endless", "worktrees", "session-" + shortID)
       Run: git -C projectRoot worktree add -b branch wtDir main
         If branch exists already (rare; collision on first 8 chars), retry
         with full UUID or fall back to a fresh randomly-suffixed name.
       Write .endless/worktree.json (session flavor) inside wtDir
       Claim .endless/worktree.lock with this session's id/pid/pane
       Return additionalContext:
         "Created session worktree at <wtDir>. Run: cd <wtDir>
          before any file edits. main is read-only for this session."
   case C: worktreePath == "" AND not isInMainCheckout(cwd):
       // Edge: cwd is somewhere outside the project entirely, or in a
       // git worktree that lacks an endless companion (foreign). Don't
       // create or claim anything; let existing handlers proceed.
       proceed
4. Continue with handleTaskContextInjection as before.
```

The hook response for case A (refusal) and case B (creation) uses the existing `hookResponse{AdditionalContext: ...}` JSON; the channel-message fallback already in UserPromptSubmit (lines 153-157) shows the same pattern for "carry an instruction back to Claude in a non-blocking way."

**Idempotency:** Re-entries (Claude resumes a session) hit case A with `lock.SessionID == payload.SessionID` and short-circuit. This is essential — SessionStart can fire more than once per Claude UUID across resumes.

## PreToolUse extension — `cmd/endless-hook/claude.go:333`

Insert worktree-aware blocks **before** the existing writeTools enforcement (line 348) and **after** the existing E-1012 git-commit block (lines 339-345):

```go
// Worktree gate (E-971 Layer D): edits must originate from a worktree
// owned by this session, and the worktree's task (if any) must match
// the session's active task. Independent of tracking_mode, like E-1012.
if writeTools[payload.ToolName] {
    projectRoot, _ := monitor.ProjectPath(projectID)
    worktreePath, _ := monitor.FindWorktreeRoot(payload.CWD, projectRoot)

    if worktreePath == "" {
        // No companion found in walk-up. Distinguish main vs foreign worktree.
        if inMain, _ := isInMainCheckout(payload.CWD); inMain {
            blockToolUse(`Edits in main are not allowed (E-971).

main is the integration target. Start your work in a worktree:

  endless worktree current    # see if you are already in one
  cd <worktree-path>          # cd into an existing one

A new session started in main will auto-create a session worktree —
restart this Claude session to get one, then cd there.`)
        }
        // Foreign worktree (no endless companion) — leave alone, fall
        // through to existing checks. Foreign trees are listed-only.
    } else {
        // We are inside an endless-managed worktree.
        comp, err := monitor.ReadWorktreeCompanion(worktreePath)
        if err == nil && comp != nil && comp.TaskID != "" {
            // Task-flavor worktree. Compare against session's active task.
            session, _ := monitor.GetActiveSession(payload.SessionID)
            if session != nil && session.ActiveTaskID != nil {
                worktreeTaskNum, parseErr := parseEndlessTaskID(comp.TaskID)
                if parseErr == nil && worktreeTaskNum != *session.ActiveTaskID {
                    blockToolUse(fmt.Sprintf(
                        "This worktree is bound to %s, but your active task is E-%d.\n\n"+
                        "Either switch tasks:\n"+
                        "  endless task start E-%d\n\n"+
                        "Or move to the worktree for your active task:\n"+
                        "  endless worktree for-task E-%d",
                        comp.TaskID, *session.ActiveTaskID,
                        worktreeTaskNum, *session.ActiveTaskID))
                }
            }
        }
        // Session-flavor worktrees have no task_id; no match check needed.
    }
}

```

## SessionEnd extension — `cmd/endless-hook/claude.go:180-191`

After `RemoveCompanion` and before `EndSession`, release any worktree lock owned by this session. Use `FindLockBySessionID` rather than walk-up-from-cwd, because:

- The user may have cd'd out of the worktree before `Ctrl-D`.
- A session created in main (case B above) holds a lock in a worktree it never cd'd into.

```go
case "SessionEnd":
    monitor.ParseTranscript(payload.SessionID, payload.TranscriptPath)
    monitor.FlagNeedsRecap(payload.SessionID)
    if err := monitor.RemoveCompanion(projectID, "claude", payload.SessionID); err != nil {
        return fmt.Errorf("removing companion file: %w", err)
    }
    // Release any worktree lock this session holds (E-971 Layer D).
    if wtPath, err := monitor.FindLockBySessionID(projectID, payload.SessionID); err == nil && wtPath != "" {
        if err := monitor.ReleaseWorktreeLock(wtPath); err != nil {
            log.Printf("releasing worktree lock at %s: %v", wtPath, err)
            // Non-fatal: lock will be reaped by stale-PID check on next claim attempt.
        }
    }
    if err := monitor.EndSession(payload.SessionID); err != nil {
        return fmt.Errorf("ending session: %w", err)
    }
```

Worktree itself is **not** removed at SessionEnd — only the lock. The worktree persists until `endless worktree land` or `drop`. This matches the brief and the lifecycle table in `.endless/plans/E-971.md:26-30`.

## Decisions / tradeoffs locked in this plan

1. **Companion file gets a `flavor` field**, additive to the existing schema. The existing Python code in `worktree_cmd.py` is forward-compatible: it ignores unknown fields and only requires `task_id` for landing.
2. **Session-flavor worktrees do not have a `task_id`** — they are not landable via `endless worktree land <task-id>`. Disposal happens via `endless worktree drop session-<id>` or by walking away (foreign-classification doesn't apply since the companion *is* present; they classify as `active`). A future slice can add `endless worktree drop --flavor session --stale` for cleanup; not in scope here.
3. **Lock release uses session-id scan, not cwd walk-up**, because a session-in-main holds a lock in a worktree it never cd'd into. `FindLockBySessionID` reads each `.endless/worktrees/*/.endless/worktree.lock`, costs O(N) where N is worktree count (typically <10).
4. **Refusal at SessionStart (case A) returns additionalContext, not exit 2.** SessionStart is not a tool-use; Claude doesn't interpret exit 2 as a session block. PreToolUse will subsequently block edits because the session never claimed a lock — the refusal still takes effect, just via a different mechanism.
5. **TOCTOU on stale-lock reclaim is accepted.** The brief does not call for fencing. If two sessions race to reclaim the same stale lock, both write — last writer wins, and the loser's session_id will appear in the lock briefly until SessionEnd. Worst-case impact: a confusing "owned by session X" message until release. Not worth a flock-based fence in v1.
6. **PreToolUse worktree-on-main edit block is independent of tracking_mode**, matching E-1012's pattern. Worktree-task mismatch block is also unconditional once a worktree is adopted.
7. **`flavor` for session worktrees uses `branch = "session/<short-id>"` and worktree dir name `session-<short-id>`** — kebab-case dir, slash-separated branch (consistent with `task/<n>-<slug>` per E-971's main plan).
8. **Walk-up bound is `monitor.ProjectPath(projectID)`** — stop at the registered project root rather than filesystem root, to avoid surprises when the project is nested under unrelated git repos.

## Verification

Manual end-to-end (run from the dogfood worktree, after building via `PYTHONPATH=<worktree>/src` per the brief):

1. **Per-session creation in main.**
   - `cd <main checkout>` (a fresh terminal, no worktree).
   - `claude` (or however the session starts).
   - SessionStart should run; `endless worktree list` should now show a `session-<8chars>` worktree with branch `session/<8chars>` and an `active` state.
   - The worktree's `.endless/worktree.json` should have `flavor: "session"`, the session UUID, and the branch.
   - `.endless/worktree.lock` inside the worktree should contain this session's UUID and PID.
   - The hook response message ("cd into <path>") should appear in Claude's context.

2. **Adoption of an unowned worktree.**
   - From main: `git worktree add -b foo .endless/worktrees/foo main` and write a minimal `.endless/worktree.json` ({"flavor":"task","task_id":"E-9999","base_branch":"main","branch":"foo","created_at":"..."}). No lock file.
   - `cd .endless/worktrees/foo`, start a new Claude session.
   - `cat .endless/worktree.lock` should show the new session's UUID.

3. **Refusal of owned worktree.**
   - Reuse the worktree from (2), still owned by session A.
   - In a second pane, `cd` to the same worktree and start session B.
   - SessionStart should return an additionalContext refusal naming session A's PID; `cat .endless/worktree.lock` should still show session A.
   - `Write` in session B should be blocked with the worktree-on-main / no-worktree message (since the lock isn't claimed, but cwd has a companion — the PreToolUse path actually allows this case if the companion exists, so we may need to check the lock owner in PreToolUse too. **Followup question:** does PreToolUse need a lock-owner check, or is it sufficient that adoption refusal is informational and the user is expected to comply? The brief says "refuse to register the new session" but does not say PreToolUse rejects edits when the lock is foreign. Resolve during implementation: probably acceptable to leave PreToolUse silent on this, since the refusal message in the SessionStart context is loud enough, and the PreToolUse worktree-task-match check still fires for task-flavor mismatches.)

4. **Stale-lock reclaim.**
   - Reuse worktree from (2). Manually edit `.endless/worktree.lock` to set `pid` to a definitely-dead value (e.g. `999999`). Set `session_id` to a fake UUID.
   - Start a new session, cd to the worktree.
   - SessionStart should detect staleness (`kill -0 999999` fails) and overwrite with the new session's lock.

5. **Edit block in main.**
   - From main checkout, with a session that has no worktree binding: ask Claude to Write a file.
   - PreToolUse should block with the "Edits in main are not allowed" message.

6. **Edit allowed in owned worktree.**
   - From a session-claimed worktree: Write succeeds (existing tracking_mode rules permitting).

7. **Task-worktree mismatch.**
   - Create a task-flavor worktree for E-XXX. cd into it. In another session, set active task to E-YYY (different ID) via `endless task start E-YYY`. Attempt a Write.
   - PreToolUse should block with the task-mismatch message.

8. **SessionEnd releases.**
   - End a session that holds a lock. Verify `.endless/worktree.lock` is deleted from the bound worktree, but the worktree itself (companion + branch + dir) persists.

9. **Existing behaviors unchanged.**
   - Run existing `internal/monitor/companion_test.go` and any hook tests: `go test ./internal/monitor/... ./cmd/endless-hook/...` (paths to confirm during implementation).
   - Manually verify E-1012 still blocks `git commit` from main.
   - Manually verify drift_detection enforcement still fires when active.

## Open questions to resolve at implementation time

These do not block plan approval — they are small enough that the implementer can decide and document in the PR:

1. **PreToolUse lock-owner check?** Should PreToolUse check that the current session owns the lock, or only that there *is* a companion? Argument for owner check: hard guarantee that two sessions can't edit the same worktree. Argument against: the brief specifies SessionStart refusal as the enforcement point; doubling up is belt-and-suspenders.
2. **Branch-collision retry strategy.** If `session/<8chars>` already exists (unlikely but possible across sessions sharing a UUID prefix), retry with longer prefix (12 chars), or full UUID. One-line decision at implementation time.
3. **Should SessionStart skip worktree creation when `payload.CWD` is outside the registered project entirely?** Currently the case-C branch silently passes through. This is the right call (don't create worktrees for sessions in unrelated dirs), but worth a code comment.
