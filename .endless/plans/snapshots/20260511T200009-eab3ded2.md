# E-987 — Resolve event-log auto-append vs clean-main tension

## Context

E-987 is a `research` task whose v1 design (locked 2026-04-30) defined how `endless worktree land` reconciles append-only auto-managed files with the "clean-main precondition" gate. The v1 plan auto-commits dirty endless-managed files on `main` as `Endless: auto-record session activity`, rebases the worktree onto `main`, and ff-merges. For rebase conflicts on auto-managed files (the "architectural drift" path), the v1 plan surfaces a copy-pasteable recovery hint:

```
git rebase --abort
git checkout main -- .endless/events/ .endless/plans/snapshots/
endless worktree land <task-id>
```

E-1240 (Add completed status gated by verbs.json completable flag) demonstrated a case the v1 procedure does not handle: a worktree branch's commit legitimately added new schema fields + new array entries to `.endless/verbs.json`. The rebase conflicted because `main` had auto-recorded a `bind` verb. Following the stock recovery hint would have silently destroyed E-1240's intentional schema additions. The conflict was resolved by hand. The 2026-05-11 addendum on E-987's text records the four design questions opened by that incident.

## Current state (read before designing)

Several things have moved since the addendum was written. The design must absorb them rather than re-litigate scenarios that have already been retired in code:

1. **E-1268 landed (`ef65169 E-1268: convert verbs.json to verbs.jsonl with union merge driver`).**

   - `.gitattributes` carries `.endless/verbs.jsonl merge=union`, so git auto-concatenates both sides' new lines on rebase/merge. The exact E-1240 conflict scenario (parallel appends colliding on the trailing region) no longer reproduces.
   - The Python dedup helper `_dedup_worktree_verbs_against_main` (`src/endless/worktree_cmd.py:458–513`) still runs as belt-and-suspenders to absorb same-`value` duplicates produced by union merge.

2. **E-972 routing rule.** All endless CLI mutations route writes to **main's** `.endless/` directory regardless of which worktree the session is in. Confirmed by reading `db-ledger`/`snapshots` git history: every commit touching those paths is either `Endless: auto-record session activity` on main, an explicit plan-attach commit on main, or the E-1197 ledger migration. **No worktree branch in the current history has commits modifying `.endless/db-ledger/*.jsonl` or `.endless/plans/snapshots/*`.**

3. **Auto-commit glob list (`worktree_cmd.py:53–58`) is now:**

       ".endless/db-ledger/*.jsonl",

4. **Current rebase-failure recovery hint (`worktree_cmd.py:744–759`)** still emits the indiscriminate "checkout `<base>` over all `AUTO_COMMIT_GLOBS`" recipe, which is the unsafe step the E-1240 addendum flagged.

## What's actually still unsafe after E-1268

After E-1268, the realistic remaining failure modes for the v1 recovery hint are narrow:

| Auto-managed path                | Can a worktree branch legitimately commit to it? | Recovery-hint risk                                |
|----------------------------------|--------------------------------------------------|---------------------------------------------------|
| `.endless/verbs.jsonl`           | Yes (feature adds a verb). Union merge handles it. Dedup helper handles same-value collisions. | Recovery hint says `checkout main -- verbs.jsonl` → would discard worktree-side verb additions. Should *not* be issued for this file. |
| `.endless/verbs.json`            | Legacy. Only seen during E-1268 migration window. | Negligible going forward.                         |
| `.endless/db-ledger/*.jsonl`     | Per E-972, no — writes route to main. A worktree-side commit on these is *itself* a bug. | Discarding worktree-side content is "safe" in the sense that the content shouldn't have existed; but silently discarding is wrong — it masks the underlying routing bug. |
| `.endless/plans/snapshots/*`     | Per E-972, no — same as above. Snapshots also have uniquely-timestamped filenames so two writers can't even target the same path. | Same as db-ledger.                                 |

The v1 procedure conflates "this file can never have legitimate worktree-side content" (db-ledger, snapshots — refuse + investigate) with "this file might have legitimate worktree-side content" (verbs.jsonl — must merge, not discard).

## Recommended design

Treat the rebase-failure path as a triage step, not a stock recipe:

**Step 1 — Differentiate by file class.** On rebase conflict, partition the conflicted paths into:
- **Mergeable auto-managed**: `.endless/verbs.jsonl` (and the legacy `.endless/verbs.json` during the migration tail). These have a known schema, and a known dedup helper already exists for them. The action is to abort the rebase, re-run the dedup, and retry — never to discard worktree-side content.
- **Append-only auto-managed**: `.endless/db-ledger/*.jsonl`. Per E-972 these can't have legitimate worktree-side content. The action is to refuse with "this indicates a routing bug — investigate; do not auto-discard."
- **Write-once auto-managed**: `.endless/plans/snapshots/*`. Same as append-only — uniquely-named per write, so conflict implies a routing bug.
- **User-work**: anything not in the auto-commit globs. Refuse — already handled at the pre-rebase partition step.

**Step 2 — Detect feature-intent vs auto-record on the worktree side.** For mergeable auto-managed paths only, before doing anything automated, scan the worktree branch's commits that touched the file:
```
git log <base>..<branch> --pretty=%s -- <path>
```
If every touching subject matches the auto-record prefix (`Endless: auto-record session activity` or `Endless: bundle worktree verb additions`), the worktree-side content is purely ambient and the dedup-retry path is safe. If any touching commit subject does *not* match (the E-1240 case — a feature commit on `verbs.jsonl`), refuse with a path-specific manual-resolution message rather than offering a destructive copy-paste.

**Step 3 — Replace the indiscriminate recovery hint** in `worktree_cmd.py:744–759` with a partitioned error message. The message must (a) name the specific conflicted file(s), (b) classify each into one of the buckets above, and (c) emit only safe, file-specific guidance — never a blanket `git checkout main -- <AUTO_COMMIT_GLOBS>`.

**Step 4 — Status of E-1017 (hand-edit prohibition).** No change required. E-1017 forbids *hand-editing* auto-managed files. A feature commit on a worktree branch that modifies `verbs.jsonl` via code is not a hand-edit. The prohibition wording does not need a carve-out; the implementation needs the partitioned recovery hint instead. Confirm with Mike whether to add a one-line clarification anywhere, since the addendum raises it as Q4.
