# E-1416: Eliminate uncommitted ledger artifacts as a worktree-land failure mode

## Context

When `endless worktree land` is invoked and the **worktree's** working tree has uncommitted files under `.endless/plans/snapshots/` (or other endless-managed paths), the rebase in Step 4 fails with git's generic "You have unstaged changes" error. The error names neither the offending files nor a recovery action.

Hit during E-1307 land. Resolved manually by `git add` + commit-as-"Endless: record ledger entry" before retrying `worktree land`.

## Root cause (from exploration)

The mechanism that produces uncommitted snapshot pairs in the worktree is **deliberate**, not accidental:

- `cmd/endless-hook/claude.go:944-957` (per **E-1354**): when the E-1281 sandbox is active, `snapshotPlanFile` writes the `.md` + `.json` snapshot pair into the worktree's `.endless/plans/snapshots/` and **explicitly skips the commit**. Comment: *"snapshot files stay in the worktree's tree and are dropped with the sandbox."*
- The reasoning was that committing the pair in sandbox mode would target the worktree's task branch, which the **E-1309 guard** in `ensureMainCheckout` refuses (it requires the project's main checkout).
- E-1354's note assumed the snapshots would be discarded with the sandbox. In practice they're left in the worktree as untracked files until the user lands the worktree.

What `worktree land` already does (`src/endless/worktree_cmd.py`):

- `AUTO_COMMIT_GLOBS` at lines 54-59 lists endless-managed paths whose dirt is auto-commitable: `.endless/db-ledger/*.jsonl`, `.endless/plans/snapshots/*`, `.endless/verbs.jsonl`, `.endless/verbs.json`.
- `_git_status_partition()` at lines 522-557 splits a tree's dirt into `(auto_files, user_files)` using those globs.
- `land_worktree()` at line 916 calls partition **on `main_root` only**. The worktree's own tree is never partitioned. Step 2 refuses on `main`'s user-work dirt; Step 3 auto-commits `main`'s auto-files as `"Endless: auto-record session activity"`.
- Step 4's rebase (line 976) runs `cwd=worktree_path`. If the worktree has uncommitted auto-files, git aborts the rebase with "unstaged changes." The error message at lines 982-991 mentions a different recovery path (auto-file modifications inside committed worktree commits — E-972 violation) and does not match this failure mode.

So the existing defense is **half-built**: same machinery, half the coverage.

Other writers that don't self-commit (from second exploration agent) — relevant context but not the immediate failure:
- `src/endless/task_cmd.py:479` writes `.endless/plans/E-{id}.md` via `task update --text-file`. Not in AUTO_COMMIT_GLOBS, currently OK only because users rarely touch it dirty pre-land.
- `src/endless/config.py:120-122` writes `.endless/config.json` (writer chosen not to be auto-commit per E-987 docs at lines 48-50).
- `src/endless/worktree_cmd.py:818-820` writes `.endless/worktree.json`. Gitignored, so non-issue.

## Three candidate approaches

### Option A — Restore commit at the snapshot write site

Undo E-1354's sandbox-skip and find a way to commit snapshots that lands them safely. Options inside A:
- A1: Commit to the worktree's task branch (the E-1309 guard would have to learn that snapshots are an exception, or the guard would need to be relaxed for sandbox-active mode).
- A2: Have the sandboxed hook reach out to the project's main checkout for the commit (defeats the sandbox isolation E-1281 set up).

Same idea generalizes to any other non-committing writer (task plan file in particular), if we want symmetry.

**Pros:**
- Working tree never goes dirty after a successful operation. Every consumer of a clean tree benefits (rebase, `worktree land`, manual `git push`, `git status` noise).
- Commit attribution matches the operation that produced the files.

**Cons:**
- Re-litigates E-1309/E-1354's deliberate sandbox isolation. A1 needs a guard exception; A2 punches through the sandbox boundary.
- Multi-file inventory: every site that ever writes under `.endless/` must commit. New sites must remember the contract.

### Option B — Extend `worktree land` to partition the worktree's tree, not just main's

Before Step 4 rebase, partition the **worktree's** working tree via `_git_status_partition(worktree_path)` and apply the same Step 2/3 logic that already runs on main:
- If worktree has user-work dirt → refuse with the same actionable error (lists files, suggests commit/move/set-aside).
- If worktree has auto-file dirt → auto-commit on the worktree branch as `"Endless: auto-record session activity"`, then proceed to rebase.

Reuses `_git_status_partition()`, `AUTO_COMMIT_GLOBS`, and the existing Step 3 commit pattern. New code is ~15 lines slotted between current Step 3.7 and Step 4 in `land_worktree()`.

**Pros:**
- Smallest surface change. Reuses existing machinery and naming.
- Doesn't re-litigate E-1309/E-1354's sandbox-isolation design — leaves snapshots uncommitted in the sandbox as that design specifies, but converts them to a real commit at land time when they need to merge to main.
- Snapshot auto-commits land on the worktree branch, get rebased onto main with the rest of the work, end up on main in the right order.
- Step 2's existing actionable error already exists for user-work dirt; this just extends its coverage.

**Cons:**
- Only `endless worktree land` is defended. Raw `git rebase` or `git push` from a worktree still surfaces the dirty state (current behavior).
- Auto-commit attribution is "auto-record session activity" rather than the operation that produced the files (same as today's main-side Step 3, so consistent with existing pattern).
- Files left by writers that fall outside AUTO_COMMIT_GLOBS (e.g., `task update --text-file` writing `E-{id}.md`) still cause the refuse path — that's correct behavior, but the error needs to be clear enough to point at the fix.

### Option C — Both A + B

Writers commit at the source (A). Land keeps B as a defensive backstop for manual edits, sandbox-skip cases, and writers we miss.

**Pros:** Defense in depth. Catches design-edge cases (sandbox-skip) AND drift (new writers added without remembering the contract).

**Cons:** Largest implementation surface. Requires resolving E-1309/E-1354's deliberate decisions.

## Recommendation framing for Mike

Exploration changed the picture from the prompt's framing: the existing `land` code already implements the auto-commit-vs-refuse partition for `main`; the bug is that it skips the same check on the worktree. **Option B is essentially "finish the existing pattern"** — a small, scoped change that reuses live machinery and does not re-litigate the deliberate sandbox isolation in E-1354.

**Option A** is the philosophically purer answer (working tree always clean) but reopens E-1309/E-1354's design discussion and requires either a guard exception or sandbox-boundary changes.

**Option C** adds A on top of B as defense-in-depth.

5. Run `endless worktree land NNNN`.
