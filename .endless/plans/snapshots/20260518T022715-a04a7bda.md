# E-1416: Eliminate uncommitted ledger artifacts as a worktree-land failure mode

## Context

When `endless worktree land <id>` runs and the **worktree's** working tree has any uncommitted file, the rebase step fails with git's generic "You have unstaged changes" error — naming no files and offering no recovery. Hit during E-1307 land, manually resolved by adding the untracked snapshot pair and committing.

The title says "ledger artifacts." In practice the dirt that's actually causing pain is **plan snapshot pairs** (`.endless/plans/snapshots/{stem}.md` + `.json`), not `.endless/db-ledger/*.jsonl` files (those self-commit fine via `CommitLedgerSegment`). Treat the scope as "any auto-managed endless file left dirty in the worktree at land time" — that's the symptom class.

## What `endless worktree land` does today

In plain terms (code at `src/endless/worktree_cmd.py:872`):

1. Look at the **main checkout's** working tree. Sort the dirt into two piles:
   - **auto-managed** (db-ledger, snapshot pairs, verbs files — list at `worktree_cmd.py:54-59`)
   - **everything else**
2. If "everything else" is non-empty → refuse to land, list the files, suggest commit / move / set-aside.
3. If "auto-managed" is non-empty → `git add` + commit on main as `"Endless: auto-record session activity"`.
4. Go into the **worktree** and run `git rebase main` there.
5. ff-merge the rebased branch back into main.
6. Remove the worktree.

## The gap

Step 1 only looks at the **main checkout's** tree. The **worktree's** own tree is never inspected. Step 4's rebase runs `cwd=worktree_path`, so any uncommitted file in the worktree makes git abort with "You have unstaged changes." The error message at `worktree_cmd.py:982-991` mentions a different recovery path (the E-972 case where a *committed* worktree commit modifies auto-files) and does not match this failure mode.

## Why the worktree has uncommitted snapshots (root cause)

`cmd/endless-hook/claude.go:944-957` (per E-1354): when the E-1281 sandbox is active, `snapshotPlanFile` writes the pair into the worktree's `.endless/plans/snapshots/` and **explicitly skips the commit**. Reason at the time: committing would target the worktree's task branch, which the E-1309 guard in `ensureMainCheckout` refuses. E-1354's note assumed snapshots would be discarded with the sandbox; in practice they survive until `worktree land` tries to rebase.

This worktree right now has exactly this dirt:
```
?? .endless/plans/snapshots/20260518T020019-c1424923.json
?? .endless/plans/snapshots/20260518T020019-c1424923.md
```

E-1374 (status `needs_plan`, phase `maybe`) proposes moving snapshots to `.git/info/endless/snapshots/e-NNN/` — outside git tracking. If that lands, snapshot dirt becomes impossible by construction.

## Related tasks checked

- **E-1332** (sandbox cache cleanup on land/drop): orthogonal, doesn't touch the rebase failure.
- **E-1337** (archive-and-reap instead of `git worktree remove`): happens after the rebase step E-1416 protects — compatible.
- **E-1391** (worktree.created / .destroyed events): lifecycle visibility, separate.
- **E-1374** (move snapshots out of git tracking): would eliminate snapshot dirt specifically, but doesn't defend against other writers that may emerge later.

## Recommended approach: backstop at land time

Before step 4 rebase, run the same sort-the-dirt logic on the **worktree's** working tree that step 1 already runs on main:

```python
# After step 3.7, before step 4
wt_auto, wt_user = _git_status_partition(worktree_path)
if wt_user:
    raise click.ClickException(
        f"worktree for {canonical} has uncommitted user changes; cannot land.\n\n"
        f"Files:\n  " + "\n  ".join(wt_user[:20]) + ...
    )
if wt_auto:
    _git_run(["add", "--", *wt_auto], cwd=worktree_path)
    _git_run(
        ["commit", "-m", "Endless: auto-record session activity"],
        cwd=worktree_path,
    )
```

Reuses `_git_status_partition()` (already at `worktree_cmd.py:522`) and `AUTO_COMMIT_GLOBS` (already at line 54). The auto-commit lands on the worktree branch; the existing rebase + ff-merge then carries it onto main with the rest of the work. ~15 lines slotted between step 3.7 and step 4.

**Why this approach:**
- Smallest change. Reuses live machinery and naming.
- Doesn't reopen E-1309 / E-1354's deliberate sandbox isolation.
- Independent of E-1374 — works whether snapshots stay in `.endless/` or move to `.git/info/`.
- Defense against the symptom class, not just the one writer that's broken today. Future writers under `.endless/` that fall in the auto-managed list get covered for free.

**Tradeoffs:**
- Only `endless worktree land` is defended. Raw `git rebase` or `git push` from inside a worktree still surfaces dirt directly (current behavior — not a regression).
- Attribution is `"Endless: auto-record session activity"`, same as main-side step 3 today. Consistent, but not as specific as committing at the write site would be.

## Critical files

- `src/endless/worktree_cmd.py:872` — `land_worktree()`. Add the worktree-side check between current step 3.7 and step 4.
- `src/endless/worktree_cmd.py:522` — `_git_status_partition()`. Already accepts an arbitrary tree root; no change needed.
- `src/endless/worktree_cmd.py:54-59` — `AUTO_COMMIT_GLOBS`. The set the backstop relies on. If new file categories should auto-commit (e.g., `.endless/plans/E-*.md` task plan files), add them here.
- `src/endless/worktree_cmd.py:982-991` — rebase failure message. Refine so the "unstaged changes" case (now handled before reaching step 4) is no longer the matching error.

## Verification

End-to-end in a scratch worktree:

1. From the **main** checkout: `endless task add "verify E-1416 backstop" --parent E-1416` → returns ID, say `E-NNNN`.
2. `endless task claim NNNN` → creates worktree, sandbox active.
3. In a Claude session inside that worktree, write something to `.endless/plans/E-NNNN.md` — the `PlanFileWrite` hook fires `snapshotPlanFile`, which writes a pair to `.endless/plans/snapshots/` and (per E-1354) skips the commit.
4. Confirm via `git status` inside the worktree that the pair is untracked.
5. From the **main** checkout: `endless worktree land NNNN`.
6. Expectation: land detects the untracked pair as auto-managed, commits it on the worktree branch as `"Endless: auto-record session activity"`, rebases onto main, ff-merges, removes worktree. Final main contains the snapshot commit. No git "unstaged changes" error.

Negative test:

7. Repeat steps 1-3, but ALSO create a non-auto-managed dirty file (e.g., edit a `.go` file in the worktree without committing).
8. Run `endless worktree land NNNN`.
9. Expectation: land refuses with a message naming the dirty `.go` file. Snapshot pair untouched. Same wording as the existing main-side step 2 message, adapted to mention the worktree.

Regression check: a worktree with no `.endless/` dirt lands exactly as before.
