# E-1298 — Detect stale binary after worktree land and rebuild or surface hint

## Context

E-1206 surfaced this problem concretely: after `endless worktree land E-1206` merged the new `CommitLedgerSegment` logic into main, the symlinked binary at `/usr/local/bin/endless-event` was still the pre-E-1206 build from earlier that day. Every endless command for ~5 hours kept hitting the stale binary, writing JSONL lines without committing them. The fix was a manual `just build` from main. Want to make this automatic.

The root cause is general: `endless worktree land` ends in success when git is fully reconciled, but it does not refresh the on-disk Go binaries that the global `/usr/local/bin/endless-*` symlinks point at. Any Go-touching change is invisibly applied to source but not to the running CLI until someone manually rebuilds.

Python is unaffected — `uv tool install -e .` makes the Python source editable, so changes are live immediately. The gap is strictly for the Go binaries.

## What's actually being changed

Add a post-success step to `src/endless/worktree_cmd.land_worktree` (after step 6, before the success echo at `worktree_cmd.py:817`):

1. Determine whether the just-landed commits touched any Go source.
2. If yes, rebuild the affected binaries.
3. If the rebuild fails, surface a clear warning but **do not fail the land** — the merge already succeeded.

## Detection

After the ff-merge succeeds, capture the prior `main` tip (we already have it from the rebase step) and the new `main` tip (`HEAD`). Run:

```
git diff --name-only <prior-tip>..HEAD
```

Match any path against the Go-source pattern set:
- `*.go`
- `cmd/**`
- `internal/**`
- `go.mod`, `go.sum`

If any path matches → trigger rebuild. Otherwise skip silently.

(`.templ` and `.css` source files are committed alongside their generated `_templ.go` / `output.css` artifacts, so a `go build` alone is sufficient to refresh the CLI binaries — no `templ generate` or `tailwindcss` step needed inside endless's auto-rebuild.)

## Rebuild

Direct `go build` calls — **no shelling to `just`** (per the "Just is dev-only" rule: user-facing verbs must not depend on the Justfile).

Enumerate `cmd/*` directories and run for each:

```
go build -o bin/<name> ./cmd/<name>
```

The symlinks at `/usr/local/bin/endless-*` already point at `bin/<name>`, so a successful rebuild immediately refreshes the installed CLI without additional symlink work.

## Gating

Only run the rebuild when the current project's `go.mod` module path is `github.com/mikeschinkel/endless`. For any other endless-managed project, this auto-rebuild step is silently skipped — those projects have their own build systems and we should not assume anything about them.

Future generalization (different task): a post-land hook mechanism (sibling of E-1086's post-create hook) lets any project define what should happen after its worktrees land. For E-1298 v1, the hardcoded endless gate is the right size.

## Opt-out

`endless worktree land` gets a `--no-build` flag. Default behavior is auto-rebuild on Go change. The flag lets the user skip the rebuild (useful when they intentionally want to keep the old binary running, e.g. mid-experiment).

## Failure handling

If `go build` returns non-zero for any binary, emit a yellow warning line:

```
• Landed E-XXXX but auto-rebuild failed for endless-<name>: <error>
  Run 'just build' from main to refresh binaries.
```

The land's success status is unchanged. The user is informed and can act.

## Code surface

## Tests

- `tests/test_worktree_land_rebuild.py` (new file). Use a temp git repo with a fake `cmd/foo/main.go` and verify:
  - Land with a Go-touching change triggers a rebuild → `bin/foo` exists after the land.
  - Land with non-Go changes only → no rebuild attempted (verify `bin/` is unchanged).
  - `--no-build` flag skips rebuild even for Go-touching change.
  - Module-name gate: when `go.mod` module is NOT `github.com/mikeschinkel/endless`, rebuild is skipped.
  - Build failure surfaces a warning but doesn't fail the land (test by injecting a syntax-error .go file).

## Verification

1. **Happy path**: from a worktree, edit a file under `internal/`. Land the task. Expect: `bin/endless-*` modification times advance; the land succeeds; the new code is immediately reflected when running endless commands.
2. **Skip path**: land a task that only touched `.endless/plans/E-NNN.md` (or other non-Go file). Expect: no `go build` invocation; `bin/` modification times unchanged.
3. **Opt-out**: land with `--no-build` on a Go-touching task. Expect: success message; no rebuild; binaries stay stale until `just build` is run manually.
4. **Build failure**: land a task that has a syntax error in a .go file (committed locally). Expect: yellow warning surfaces, land still reports success.
5. **Other project**: from a non-endless project (no matching module name), land a task. Expect: no rebuild attempted.

## Open questions

1. **Detection vs unconditional rebuild.** Detection costs one `git diff --name-only` call (~10ms) but skips ~10-20s of build time for non-Go lands (e.g. docs-only). Unconditional is simpler code but pays the build cost every land. Recommendation: detect.
2. **Opt-out flag default.** Default = auto-rebuild on. Confirm.
3. **Module-name gate scope.** Hardcode `github.com/mikeschinkel/endless` as the gate? Or use a more general check (any project with `cmd/*` subdirectories and `go.mod`)? Hardcoded gate is more conservative — recommended for v1 since the broader pattern isn't proven yet.
4. **Build failure stance.** Warn but don't fail the land (vs. fail loudly per E-1206/E-1208 pattern). Reasoning: land already happened; failing now would mislead the user about whether the merge stuck. Compromise: warn loudly, exit zero. Confirm this divergence from the "fail loudly" pattern is acceptable.
