# E-987 — Resolve event-log auto-append vs clean-main tension

> Part of E-968 (parallel-session support). Companion to E-971 (the worktrees + behavioral gate work that needs `land` to refuse dirty main).
>
> **Status**: design locked 2026-04-30. Implementation pending.

## The conflict

Two prior designs collide:

1. **E-808 (commit c9425fb)** made the event log authoritative. Every endless CLI mutation appends to `.endless/events/events-*.jsonl`. The events log is committed and replayable.
2. **E-971 (planned)** says main is the sacred clean integration target. `endless worktree land` refuses dirty main.

Per E-972, all events route to main's `.endless/events/` regardless of which worktree the session is in. So every CLI invocation leaves main dirty. Without resolution, every `land` call would refuse.

## Resolution: Option F (auto-commit endless-managed files independently)

Define **endless-managed files** (a configurable list) that `land` auto-commits as a separate step BEFORE attempting the worktree merge. Independent commit, not bundled into the worktree's land commit.

### Auto-commit list (locked)

| Path | Why it's auto-managed | Source |
|---|---|---|
| `.endless/events/*.jsonl` | Append-only log, written on every mutation | E-808 |
| `.endless/plans/snapshots/*.md` | Write-once historical record per plan-file write | E-969 |
| `.endless/plans/snapshots/*.json` | Sidecar metadata for above | E-969 |

### Explicitly NOT in the auto-commit list

| Path | Why it's user-work, not auto |
|---|---|
| `.endless/config.json` | User edits matter (E-970 matchers added/disabled by user action) |
| `.endless/plans/<task-id>.md` | Per-task plan content; user-attached |
| `.endless/data.sql` | Periodic dump created by explicit command, not automatic. See E-1018 for the data.sql lifecycle decision (deferred). |
| Any source code | Obviously user-work |

### Configurability

Hardcoded defaults in endless. Project can extend via `.endless/config.json` `auto_commit_paths` array. Project cannot REMOVE defaults (one-way override) so the architecture's invariants stay intact.

## Land flow (with retry loop)

```
MAX_RETRIES = 8

for attempt in 1..MAX_RETRIES:
    auto_files_dirty = git_status_match(AUTO_COMMIT_PATHS)
    user_files_dirty = git_status_minus(AUTO_COMMIT_PATHS)

    if user_files_dirty:
        refuse with actionable message:
          "main has uncommitted user changes; commit them, move them to a
           worktree, or set them aside before landing E-XXX. Files: ..."

    if auto_files_dirty:
        git add <auto paths>
        git commit -m "Endless: auto-record session activity"

    try:
        cd <worktree>
        git rebase main
        cd <main>
        git merge --ff-only <worktree-branch>
        git worktree remove <worktree-path>
        return success

    except blocked-by-dirty-events-log:
        # Concurrent writer raced us; retry the auto-commit step
        continue

fail with:
    "another session is appending to events log faster than land can
     converge after MAX_RETRIES attempts; try again later."
```

### Why rebase, not merge commit

- **Linear history**: per-task commits land on top of the auto-commit, no merge bubbles.
- **Conflict surface**: rebase against an auto-commit is virtually never problematic because (per E-972) auto-managed files are written to main's location only — worktree branches don't modify them. Rebase has nothing to conflict on.
- **Explicit identity**: the auto-commit appears as a distinct commit in `git log`, named clearly (`Endless: auto-record session activity`). Easy to skip when reviewing per-task work.

### Race condition handling

The retry loop addresses concurrent writers:

```
T0: Land begins; auto-commit step picks up events at state-X
T1: Session B (concurrent) appends event Y to events log
T2: Land attempts rebase / ff-merge
T3: Merge refuses because events log is dirty again
T4: Loop: re-run auto-commit step (now picks up event Y too)
T5: Retry merge — succeeds
```

`MAX_RETRIES = 8` is a sane cap. Each iteration is fast (one commit, one merge attempt). If we exceed the cap, something pathological is happening (busy loop of writes); fail with a message and let the user retry later.

## Hand-edit prohibition (cross-reference)

E-1017 (decision, confirmed) makes hand-editing of endless-managed files forbidden. Auto-managed files are write-once-by-endless. This is what makes the rebase-conflict-free property reliable: nothing should EVER produce conflicting writes on these files from a worktree.

## Recovery from architectural drift

When the rebase DOES conflict on an auto-managed file (a future-feature bug, a pre-E-972 worktree, a malicious hand-edit), `land` fails with the conflict. v1 documents the recovery procedure as part of the error message:

```
git rebase --abort
git checkout main -- .endless/events/ .endless/plans/snapshots/
endless worktree land <task-id>  # retry
```

E-1019 captures this and the eventual `--resolve-auto` automation if it ever bites.

## Critical files

- `internal/git/worktree.go` (new in E-971; this work extends it) — auto-commit step + retry loop logic.
- `src/endless/worktree_cmd.py` (new in E-971; this work extends it) — `endless worktree land` CLI implementation.
- `internal/monitor/config.go` or equivalent — read `auto_commit_paths` from config (hardcoded defaults + project overrides).
- Documentation (CLAUDE.md or similar) — reference E-1017's hand-edit rule near any code that writes to events log or snapshots.

## Verification

1. **Happy path**: clean main + uncommitted events appends + worktree branch with task work → `land` auto-commits events, ff-merges successfully, worktree removed.
2. **User-work blocks**: dirty user-work file (e.g., source code) on main → `land` refuses with actionable message.
3. **Concurrent writer race**: while `land` runs, append to events log via another session → loop iterates → succeeds within a few retries.
4. **Pathological race**: simulate continuous appends from another session → `land` fails after MAX_RETRIES with the bounded-loop error.
5. **Architectural drift conflict**: hand-edit events log in worktree (violating E-1017), then `land` → rebase conflict, error message includes recovery commands.

## Addendum (2026-05-11): E-1240 demonstrated the unsafe-recovery-hint case

The "Recovery from architectural drift" section above documents v1's procedure for handling rebase conflicts on auto-managed files:

```
git rebase --abort
git checkout main -- .endless/events/ .endless/plans/snapshots/
endless worktree land <task-id>  # retry
```

That procedure assumes the worktree's writes to auto-managed files were **unintentional** — ambient dirt that should be discarded. E-1240 demonstrated a case the procedure does not handle:

- E-1240 added a new optional `completable` schema field to `.endless/verbs.json` plus 7 new verb entries. All intentional, all part of the feature.
- During `endless worktree land E-1240`, the rebase hit a conflict on `verbs.json` because main's auto-record had separately appended a new verb (`bind`).
- The stock recovery hint (now also surfaced on the `db-ledger` and `verbs.json` paths) said to `git checkout main -- .endless/verbs.json` and retry — which would have **silently destroyed E-1240's intentional schema additions**.

The conflict was resolved by hand (kept both sides). But the recovery hint is unsafe as a general procedure: it cannot tell "ambient auto-dirt" from "intentional feature content on an auto-managed file."

### Design questions opened (carried forward to E-1019)

1. **Detection.** How does `worktree land` differentiate the two cases? Candidate signals:
   - Commit metadata: was the worktree-side conflicting commit authored by the auto-record path (commit message starts with "Endless: auto-record session activity")?
   - Content-shape: did the worktree add NEW top-level keys / NEW array entries / NEW schema fields, vs. only mutate values of pre-existing entries?
   - Explicit feature-intent marker on the commit
   - Whether the file is touched by other (non-auto-record) commits on the worktree branch
2. **Action.** When the unsafe case is detected, what should `worktree land` do?
   - Refuse with a clear message
   - Drop into an interactive structural-merge helper (verbs.json has known schema — see `worktree_cmd._dedup_worktree_verbs_against_main` from E-1141/E-1138, which already does value-level merges for the auto-commit path)
   - Reuse the dedup helper for the rebase path
   - Document only
3. **Scope split.** Does this expand E-1019, file a sibling task under E-968, or split across both? Note: this design's analysis above is scoped to event logs (append-only). verbs.json is structured-with-schema, so the right design may diverge.
4. **Hand-edit prohibition.** E-1017 forbids hand-editing endless-managed files. E-1240 is **not** a hand-edit — it's a feature commit that legitimately modified the schema. The prohibition needs to either explicitly carve out "feature-intent" schema changes, or the auto-managed file set needs to grow a notion of "this is structured config that features may legitimately modify."

### Why this changes the v1 plan

The "Recovery from architectural drift" v1 procedure is correct *for event logs*. The error message that surfaces it must not promote that procedure for **any** auto-managed file uniformly — it's actively dangerous for `verbs.json`. Either the error message must be file-specific (event logs → checkout-from-main is safe; verbs.json → structural merge required), or the recovery must be one step removed from a copy-pasteable command.
