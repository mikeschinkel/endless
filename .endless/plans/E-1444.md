# E-1444: Allow endless task add from plain terminal without a Claude session

## Context

The E-1401 attribution gate in `src/endless/event_bridge.py:emit_event()` (lines ~99-108) refuses any event whose `actor_kind` is in `_ATTRIBUTION_REQUIRED = {"cli", "hook"}` unless a session_id can be resolved. The gate exists for a real reason: every event SHOULD carry session attribution so per-session activity queries (E-1284, E-1322 session_tasks) stay meaningful and audit trails aren't silently truncated.

Today the gate fires before `--force` is consulted, with no escape hatch for legitimate non-Claude-pane callers (triage filings from a plain shell, scripts, cron, post-incident ledger entries). The error message lists three remediations — run inside a Claude pane, export `ENDLESS_SESSION_ID`, or `task bind` to a sibling — and `--force` is not among them, so callers don't discover any bypass even if one existed.

Reproducer (verified today):
```sh
endless task add "probe" --type chore             # refused, no session
endless task add "probe" --type chore --force     # still refused; --force ignored
```

The bug is self-recursive: a session filing the ticket about this gate is itself blocked by the gate unless it has a live Claude pane.

## Design options (DO NOT CHOOSE — present to Mike and have him decide)

### A. Widen `--force` to also bypass the attribution gate
- Pros: one flag for "I know what I'm doing"; matches the existing pattern in `claim_item` where `--force` bypasses both the done-ish-status gate AND the no-session-binding refusal.
- Cons: conflates two semantically distinct overrides under one knob; future expansion of `--force` further dilutes its meaning.
- Wire-up: `emit_event` would need to learn about a bypass parameter; `task add` passes `--force` through to it. Error message gains a fourth bullet: `- Pass --force to file without a session binding (records actor.kind=system).`

### B. Add a dedicated `--no-session` / `--actor=system` flag
- Pros: clean separation; the flag's purpose is obvious from its name; intent is preserved in shell history and CI scripts.
- Cons: one more flag to remember; doesn't compose with `--force`'s existing semantics (would users need both for some operations?).
- Wire-up: new `--no-session` Click option on `task add` (and any other `emit_event`-using verbs that should support the bypass — `task update`, `task remove`, etc.). When set, callers pass `actor_kind="system"` to `emit_event`, which already treats `system` as not-requiring attribution. Error message names the flag directly.

### C. Auto-downgrade to `actor.kind="system"` when stdout is not a tty AND no session resolves
- Pros: no flag friction for scripts/cron; ergonomic for "obviously automated" callers.
- Cons: implicit behavior change is brittle — an interactive user piping output (`endless task list | less`) is suddenly not-a-tty and would silently file without attribution. Hides the attribution miss in cases where the user intended a session-bound filing but mis-configured their environment.
- Wire-up: in `emit_event`, before the gate, check `not sys.stdout.isatty()` and downgrade `actor_kind` from `cli` to `system`. Error message updates would be defensive ("if you meant to file with attribution, run from a Claude pane").

### Mike's call
Recommend talking through which mix fits the project's tolerance for implicit behavior. A combination (B + a small subset of C — e.g., auto-downgrade only when ENDLESS_SCRIPTED=1 is set, not by tty heuristic) may end up cleanest. The wrong answer is to leave the gate uniformly closed — that's the current state and it bit you today.

## Error message update (orthogonal to A/B/C choice)

Whichever bypass we add, the error message at `event_bridge.py:99-108` must mention it. Today's text:

```
Cannot determine the Endless session for this pane.

To fix, do one of:
  - Run this command from a Claude session pane.
  - Export ENDLESS_SESSION_ID="$(endless session id)".
  - Run `endless task bind <task-id>` from this pane to connect it to a
    sibling Claude session in the same tmux window.
```

After this task, the list grows by one bullet naming the chosen bypass. Match the existing voice — actionable, not apologetic.

## Implementation sites

- `src/endless/event_bridge.py:24` — `_ATTRIBUTION_REQUIRED` constant.
- `src/endless/event_bridge.py:99-108` — the gate itself + error string. Same file for the bypass plumbing.
- `src/endless/cli.py` — `task add` group; add the new flag (option B) or thread `--force` through to `emit_event` (option A).
- `src/endless/task_cmd.py` — `add_task()` (and any siblings: `update_task`, `remove_item`, etc.) need to receive the new flag and pass it down. Audit for any other verb that ends up calling `emit_event`.

## Verification

1. From a bare shell with no `TMUX`, no `ENDLESS_SESSION_ID`, no sibling Claude pane:
   ```sh
   endless task add "probe" --type chore <chosen-bypass-flag>
   ```
   Should succeed. The emitted event has `actor.kind="system"` (or whatever shape the chosen design records).
2. The same command WITHOUT the bypass flag still refuses, with the updated error message that now mentions the bypass.
3. From a Claude pane (control case), `endless task add "probe"` without any bypass still works as it does today — no regression on the happy path.
4. Spot-check: emitted events from a bypassed `task add` flow appear in the JSONL ledger under db-ledger/, with `actor.session_id` absent or empty. They DO NOT appear in `session_tasks` (no session to attribute the touch to) — verify by querying after a bypass-filed task.
