# E-1444: Allow endless task add from plain terminal without a Claude session

## Context

The E-1401 attribution gate in `src/endless/event_bridge.py:emit_event()` (lines ~99-108) refuses any event whose `actor_kind` is in `_ATTRIBUTION_REQUIRED = {"cli", "hook"}` unless a session_id can be resolved. The gate exists for a real reason: every event SHOULD carry session attribution so per-session activity queries (E-1284, E-1322 session_tasks) stay meaningful and audit trails aren't silently truncated.

Today the gate fires before `--force` is consulted, with no escape hatch for legitimate non-Claude-pane callers (triage filings from a plain shell, scripts, cron, post-incident ledger entries). The error message lists three remediations — run inside a Claude pane, export `ENDLESS_SESSION_ID`, or `task bind` to a sibling — and `--force` is not among them, so callers don't discover any bypass even if one existed.

Reproducer (verified during E-1337 filing):
```sh
endless task add "probe" --type chore             # refused, no session
endless task add "probe" --type chore --force     # still refused; --force ignored
```

The bug is self-recursive: a session filing the ticket about this gate is itself blocked by the gate unless it has a live Claude pane.

## Approach (decided 2026-05-21 with Mike)

Add a new `--no-session` flag implemented **once** as a shared option, applied to every CLI verb whose `emit_event` call sets `actor_kind` to `cli` or `hook` and therefore goes through the attribution gate today. The flag downgrades the event's `actor.kind` from `cli` to `system` — `system` is already exempt from `_ATTRIBUTION_REQUIRED`, so the existing gate code needs no logic change beyond accepting the downgrade.

The audit of which verbs need the flag is part of the implementation, not a separate task. The principle: **the flag attaches to commands that need a session today**, not to all write commands. Read-only verbs and commands that already use `actor_kind="system"` are untouched.

### Shared-flag implementation (Click)

Click supports option sharing via:
1. **A reusable decorator** — define `no_session_option` once and apply with `@no_session_option` on each command function. Cleanest read at the call site.
2. **A custom `click.Group` subclass** that adds the option to every subcommand automatically — DRY but magical (every command, even ones that don't need it, gains a useless flag).
3. **Click's `add_command_option`/eager-option patterns** — less common, more boilerplate.

**Choice:** option 1 (reusable decorator). Each command opts in explicitly, so the audit is visible in code review. No silent inheritance of flags by commands that shouldn't have them.

```python
# src/endless/cli_helpers.py (or similar)
no_session_option = click.option(
    "--no-session",
    is_flag=True,
    help="File this command without a Claude session binding "
         "(actor.kind='system'). Use when running from a plain shell, "
         "cron, or script.",
)
```

Each command then:
```python
@worktree_cmd.command("...")
@no_session_option
def cmd(no_session, ...):
    ...
```

The command receives `no_session: bool` and passes it through to `emit_event` (which gains a new parameter or, simpler, the caller flips `actor_kind` to `"system"` before invocation).

### Considered alternatives (rejected)

- **Widen `--force`** to also bypass attribution. Rejected: `--force` already overloads two semantics in `task claim` (status-demotion bypass + no-binding-required); adding a third blurs intent. Shell history of `endless task add --force` reads as "I'm overriding something" but doesn't say what.
- **Auto-downgrade `actor.kind` to `system` when stdout is not a tty AND no session resolves.** Rejected: implicit attribution loss is brittle. An interactive user piping output (`endless task list | less`) becomes not-a-tty and silently files without attribution. Hides the attribution miss in cases where the user intended a session-bound filing but mis-configured their environment. The opt-in approach surfaces the intent.

## Error message update

`event_bridge.py:99-108` gains a fourth bullet naming the new flag:

```
Cannot determine the Endless session for this pane.

To fix, do one of:
  - Run this command from a Claude session pane.
  - Export ENDLESS_SESSION_ID="$(endless session id)".
  - Run `endless task bind <task-id>` from this pane to connect it to a
    sibling Claude session in the same tmux window.
  - Pass --no-session to file without a Claude session binding (sets
    actor.kind=system; suitable for cron, scripts, and plain-shell
    triage filings).
```

- `src/endless/cli.py` — apply the decorator to each command that needs it. Audit candidates: `task add`, `task update`, `task remove`, `task status`, `task move`, `task claim`, `task release`, `worktree land`, plus any session-status/session-messaging commands that touch `emit_event`. Concrete list is part of the implementation PR.
- `src/endless/task_cmd.py`, `src/endless/worktree_cmd.py`, `src/endless/session_status_cmd.py` — function signatures gain `no_session: bool` and thread it to the `emit_event` call.

6. Lint check: every command whose body invokes `emit_event(actor_kind="cli"|"hook", ...)` either has `@no_session_option` applied OR is documented in the PR description as deliberately omitted (with reason).


## Edge case: `--db sandbox` from a plain shell in a self-dev worktree (added 2026-06-14)

Surfaced during E-1555 verification (2026-06-12). Mike ran `endless task add ... --db sandbox` from a plain zsh shell inside the E-1555 worktree, expecting the sibling-pane resolver (layer 4 of `_current_endless_session_id`) to pick up the orchestrator Claude session in `%204`. Instead the gate fired with "Cannot determine the Endless session for this pane."

**Mechanism:** the sibling-pane lookup in `_current_endless_session_id` (`task_cmd.py:2014`) calls `_live_sessions(project_root)` (`session_cmd.py:1041`), which shells out to `endless-go --db sandbox session-query list-live`. The orchestrator Claude session's row lives in the **main** DB (`process='%204'`, `state='working'`) — it was created by `task claim` against `--db main`. The sandbox DB contains only the seeded `00000000-…` placeholder (`process=''`, no pane binding) from `seed_worktree.go:82-89`. Result: under `--db sandbox` the resolver is structurally blind to the orchestrator session row, so layers 3 and 4 fall through and the gate refuses.

```
$ uv run endless sql --db sandbox "SELECT id, session_id, process, state FROM sessions"
   1  00000000-0000-0000-0000-000000000000             working
$ uv run endless sql --db main    "SELECT id, session_id, process, state FROM sessions WHERE state!='ended' ORDER BY id DESC LIMIT 2"
 638  258da5b5-…                              %207     working
 637  c33dac5f-…                              %204     working   <- orchestrator
```

Same shape as the original E-1444 reproducer (plain shell → gate refuses), but the cause is different: not "no session exists anywhere," but "session exists in main DB, this command reads sandbox DB." Same `--no-session` escape hatch resolves it for the user, since the user just wants to write a test event without attribution. So this case is **covered by E-1444's planned fix** — but worth documenting two things:

1. **Verification step to add:** after `--no-session` lands, smoke-test a self-dev sandbox emission from a plain shell next to a live Claude pane (the case above). Confirm `--no-session` succeeds and the resulting event has `actor.kind="system"` in the sandbox JSONL ledger.
2. **Not addressed by --no-session, deliberately:** the underlying cross-DB resolver gap (sandbox-DB read can't see main-DB session rows) remains. Fixing it properly would mean either teaching `_live_sessions` to consult the main DB for session lookups even when `--db sandbox` is active, or seeding the sandbox sessions table with a row reflecting the spawning Claude session's pane. Both are larger changes than this task's scope. If `--no-session` proves sufficient in practice, the cross-DB lookup gap can stay open; if Mike finds himself wanting attributed sandbox events from a plain shell, file a follow-up.
