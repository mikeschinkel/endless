# E-1444: Allow endless commands from a plain terminal without a Claude session

## Context

The E-1401 attribution gate in `emit_event()` refuses any event whose `actor_kind` is
`cli`/`hook` unless a session id can be resolved. Today that means a user at a plain
terminal (no `TMUX`, no `ENDLESS_SESSION_ID`, no sibling Claude pane) cannot run
`endless task add` — Mike hit this directly and found it too limiting. The gate exists for a
real reason (per-session attribution / audit trails) and should still catch *discoverable*
misuse (Claude reachable but DB-invisible), so the fix is to **auto-downgrade only when we
can positively confirm no Claude session is reachable**, plus an explicit `--no-session`
override flag for the niche "Claude pane present but I want this filing unattributed" case.

## Approach

Two changes, both small.

### 1. Auto-downgrade in `emit_event` (handles the common case)

When the gate would have refused (`actor_kind in {"cli","hook"}` and resolver returned None),
do a **positive structural check** before refusing:

```
if not in tmux (TMUX env unset)                       → auto-downgrade to system (stderr note)
elif no `claude` process in any current-window pane    → auto-downgrade to system (stderr note)
else                                                   → keep the existing refusal
                                                        (the sandbox / Claude-but-DB-invisible
                                                         case lands here — see E-1585)
```

Auto-downgrade emits the event with `actor.kind="system"` and prints one line to stderr:

```
note: no Claude session detected; recorded as system
```

The structural check is *positive* (confirm Claude is structurally absent), not "resolver
returned None." The distinction matters: a sandbox-DB read where Claude IS reachable but its
session row only lives in main produces `resolver=None` even though Claude is present —
that's a discoverable misuse the gate should still surface. **E-1585** (filed `--cleans-up
E-1444`) tracks the cross-DB resolver gap as a separate concern; that case keeps refusing
here until the resolver learns to look across DBs (or sandbox gets seeded with the relevant
session rows).

The original plan's rejection of auto-downgrade was based on a brittle tty-status heuristic
("piping output becomes not-a-tty"). The tmux/process-state check has none of that fragility
— it's a direct read of reachability, not display state.

### 2. Explicit `--no-session` flag (handles niche overrides)

For the niche cases auto-downgrade doesn't cover — wanting `system` attribution from *inside*
a Claude pane, or overriding the multi-sibling auto-pick — keep a shared `--no-session`
Click option. Implemented as a reusable decorator that stores the flag in a
`contextvars.ContextVar`; `emit_event` reads the var and, when set, **skips resolution
entirely** and forces `actor_kind="system"` (no stderr note — this path is explicit).

Opt-in stays visible in code review: each command carries `@no_session_option` explicitly.
The decorator uses `expose_value=False` so most command-function signatures don't change;
the two commands that need the flag value for control flow (`claim`, `spawn`) read the
contextvar directly.

The contextvar mechanism also propagates to nested internal emits
(`recover_task_text` during worktree creation) with zero extra plumbing.

## Scope of the `--no-session` flag

**Gets the flag** (commands a user might plausibly run from a plain shell or want to detach
from sibling auto-pick):
`task add`, `task update`, `task remove`, `task confirm`, `task assume`, `task decline`,
`task complete`, `task move`, `task replace`, `task import`, `task next revise`, `task claim`,
`task spawn`, `worktree land`.

- **`task claim`**: keeps its own `--force` refusal (claim creates a worktree + changes
  status, more consequential than just an emit — explicit intent worth requiring). With
  auto-downgrade now fixing the gate, `claim --force` from a structurally-no-Claude shell
  works again (the E-1242 path that the gate was breaking). `--no-session` on claim is a
  parallel explicit override; both lead to the same outcome.
- **`task spawn`**: spawn from a plain terminal is explicitly designed for non-Claude
  spawners (`pid-fallback marker`) but its pre-claim emit currently hits the gate.
  Auto-downgrade fixes this. `--no-session` remains for an explicit override.

**Excluded** (inherently session-scoped; flag would be contradictory or inert):
`task bind`, `task release`, `session status add`.

## Implementation sites

- **New `src/endless/cli_helpers.py`** — `no_session_var: ContextVar[bool]` and the
  `no_session_option` decorator (`is_flag=True`, `expose_value=False`, callback sets the
  var). Imports only `click` + `contextvars`.
- **New helper for the positive Claude-reachability check** — `_no_claude_reachable()` or
  similar. Reads `TMUX` env; if set, enumerates current-window panes via
  `tmux list-panes -F '#{pane_current_command}'` and returns True iff none are `claude`.
  Lives near the existing session resolver so it can share helpers. Verify whether the
  existing resolver already encapsulates this (`_resolve_session_id_with_prompt` or
  `_live_sessions`); reuse over duplication.
- **`src/endless/event_bridge.py`** —
  - import `no_session_var` from `cli_helpers`.
  - in `emit_event()` (lines ~73–108):
    - read `no_session = no_session_var.get()`.
    - if `no_session`: skip resolution; set `actor_kind="system"`; proceed.
    - else: run existing resolution; if still None and `actor_kind in _ATTRIBUTION_REQUIRED`,
      consult `_no_claude_reachable()`; if True → set `actor_kind="system"`, print stderr
      note, proceed; else → raise the existing `ClickException` (no message change needed —
      no flag-bullet to advertise since the structural-absence path no longer surfaces it).
- **`src/endless/cli.py`** — `@no_session_option` on each of the 14 in-scope commands.
- **`src/endless/task_cmd.py`** —
  - `claim_item` (~2358): read `no_session_var.get()`; if set, skip resolution
    (`target_session = None`) and treat as session-less proceed. The contextvar also gets set
    by the auto-downgrade path inside emit_event, so nested `_perform_claim_work` emits
    inherit the right behavior. (Confirm during implementation that no extra plumbing is
    needed.)
  - `spawn_plan` (~3035): no body change beyond the decorator on the command — the pre-claim
    emits read the contextvar / hit auto-downgrade automatically.
- **Tests** — extend `just test`:
  - plain shell, no tmux: `task add` succeeds, event has `actor.kind="system"`, stderr note
    printed.
  - tmux window with sibling Claude pane: `task add` (no flag) attributes to the sibling
    (no auto-downgrade).
  - inside a Claude pane: `task add --no-session` emits as system, no stderr note.
  - sandbox-vs-main case (synthetic): resolver returns None but a `claude` process IS in a
    current-window pane → gate still refuses (regression-protects E-1585's space).
  - Claude-pane control: `task add` (no flag) still attributes to the session.

## Verification

1. **Primary** — bare shell, no tmux, no `ENDLESS_SESSION_ID`:
   `endless task add "probe" --type chore` succeeds; ledger event has `actor.kind="system"`;
   stderr shows the note.
2. **Explicit override from Claude pane** — `endless task add "probe" --no-session` succeeds,
   `actor.kind="system"`, no stderr note.
3. **Sibling auto-pick still works** — from a plain shell inside a tmux window with one
   sibling Claude pane, `endless task add "probe"` attributes to the sibling (no
   auto-downgrade).
4. **Discoverable-misuse path preserved** — synthesize the E-1585 case (Claude pane reachable
   but resolver blind under `--db sandbox`); the gate still refuses.
5. **`claim --force` from plain shell** — currently broken by the gate; now succeeds with
   `actor.kind="system"` on the status_changed emit.
6. **No regression in Claude pane** — `endless task add` (no flag) still attributes to the
   session.
7. `just build && just test` green.
