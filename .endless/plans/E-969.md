# W1 — Plan-file Path Stability

> Part of E-968 (parallel-session support). See `.endless/plans/E-968.md` for full context.

## Problem (revised)

The original framing assumed `tasks.plan_file_path` was a column that went stale. It isn't — the path is on `sessions`, and tasks store plan content in the `text` column directly. So the "stale path on tasks" failure mode doesn't manifest as I described.

The **real residual problem** is content loss during plan-mode flow:

1. **Within-session overwrite**: Claude writes plan A to `~/.claude/plans/foo.md`, then later in the same session writes plan B to the same file. Plan A content is gone before SessionEnd ever sees it. If A was meaningful and never attached, it's lost.
2. **Never-attached plans**: Claude writes a plan but never runs `task update --text`. Next conversation's harness reuses the filename and overwrites. Content gone before any task held it.

Both cases boil down to: harness plan files are volatile, and the only durable storage today is `tasks.text` (populated by an explicit user/Claude action).

## Fix

Three pieces, layered:

### 1. Eager content-addressed snapshots (PostToolUse Write hook)

When PostToolUse fires for a Write to a plan file (existing `isPlanFile` check in `cmd/endless-hook/claude.go`), additionally:

- Compute SHA-256 of the just-written content (first 8 hex chars suffice for filename).
- Copy content to `.endless/plans/snapshots/<YYYYMMDDTHHMMSS>-<sha8>.md`.
- Write a sidecar at `.endless/plans/snapshots/<YYYYMMDDTHHMMSS>-<sha8>.json` with `{session_id, written_at, source_path, sha256}`.
- Idempotent: if a snapshot with the same (session_id, sha256) already exists, skip.

Every plan write is captured at the moment it exists. Subsequent overwrites in the harness path don't lose anything.

### 2. Per-task copy on attach (`task update --text`)

Modify `update_plan` in `src/endless/task_cmd.py:1427` so when `--text <path>` is provided, it ALSO writes the content to `.endless/plans/<task-id>.md`. The DB's `tasks.text` column remains the source of truth; the per-task file is a stable, endless-owned export at a predictable path.

This means once a task is attached, its plan content lives at three layers, each with a clear role:

- `tasks.text` (DB) — canonical, queryable.
- `.endless/plans/<task-id>.md` — stable file mirror, editable in `$EDITOR`, diffable in git.
- `.endless/plans/snapshots/<ts>-<sha>.md` — historical record (now redundant for this task; eligible for pruning per E-984).

### 3. Snapshot discovery CLI

New module: `src/endless/plan_cmd.py`. CLI:

- `endless plan snapshots [--session <id>] [--today] [--unattached]`
  - Lists snapshots with timestamp, session ID, source path, first-line excerpt, attach status.
  - `--session <id>`: filter by exact session ID (deterministic, from sidecar JSON).
  - `--today`: filter by date (today only).
  - `--unattached`: only show snapshots whose content hash doesn't match any existing task's `text` column.
- `endless task update <id> --text-snapshot <snapshot-id-or-path>`: convenience for attaching a specific snapshot to a task (same effect as `--text <path>` but resolves snapshot IDs).

Output is `--json` capable for agent consumption.

## Out of scope (deferred to other tasks)

- **Pruning** of `.endless/plans/snapshots/`: filed as E-984 (age + size cap).
- **Drift detection** between `tasks.text` and `.endless/plans/<task-id>.md`: not addressed in v1; can be added later if file-based editing becomes a workflow.

## Critical files

- `cmd/endless-hook/claude.go` — extend `handlePostToolUse` (the existing plan-file Write branch around line 232) with snapshot logic. Snapshot path built relative to project root (E-972 ensures we get the right one even from inside a worktree).

## Schema impact

Zero. No new columns. `tasks.text` (existing) is canonical. `tasks.source_file` (existing) is not used by this task — provenance is the file path `.endless/plans/<task-id>.md` (derivable from task ID).

## Verification

- **Snapshot capture**: write a plan file from a session, confirm `.endless/plans/snapshots/<ts>-<sha>.md` exists with matching content and a sidecar JSON containing the session ID.
- **Within-session overwrite survival**: write plan A, write plan B to the same harness path, confirm both A and B exist as separate snapshots.
- **Per-task copy on attach**: `endless task update E-XXX --text <path>`, confirm `.endless/plans/E-XXX.md` is created with matching content.
- **Discovery by session**: write plans in two sessions; `endless plan snapshots --session <id1>` lists only session 1's snapshots; `--session <id2>` lists only session 2's. No cross-contamination.
- **Idempotency**: same content written twice in the same session produces only one snapshot (sha+session de-dup).
