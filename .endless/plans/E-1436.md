# E-1436: Implement `endless task next revise` command

## Spec source

E-1421's plan at `<main>/.endless/plans/E-1421.md`. Relevant sections: "CLI surface" → revise, "Concurrency model", "JSON shape (input/output)", "Confirmed design decisions" (soft cap >10 warning, hard cap >25 refusal).

## Scope

Implement `endless task next revise --file <json>` — full-rewrite of the curated list:

1. Read JSON from `--file`.
2. Validate schema (lanes, integer priorities, items, soft cap >10 warning, hard cap >25 refusal).
3. Read latest `revised_at` + `session_id` from `project_next_revisions`. Print "list last revised Xh ago by session N" (or "first revision") for cross-session collision visibility.
4. `BEGIN IMMEDIATE TRANSACTION`.
5. Delete existing `project_next_lanes` and `project_next_items` for this project (CASCADE handles items via the lanes FK).
6. Insert new lanes and items from the JSON.
7. Insert a `change_kind='revise'` row in `project_next_revisions` with `session_id` (from the resolver), `revised_at=NOW`, `json_snapshot` = input JSON.
8. `COMMIT`.
9. Emit `--json` of the resulting state.

On `SQLITE_BUSY` (concurrent revise/move): surface the conflict clearly and exit non-zero.

1. Revise from scratch (empty list): JSON with 2 lanes, 3 items each → rows appear; revision row in audit table.

3. Soft cap: 11 items → warning printed, command succeeds.
4. Hard cap: 26 items → command rejects with error before BEGIN.
5. Collision message: revise twice from different sessions; second prints "list last revised <seconds> ago by session N" before applying.
6. Concurrent BEGIN IMMEDIATE conflict (simulated): one wins, other gets clear conflict error.
7. `--json` output matches DB state after.

## Implementation decisions (resolved with Mike, 2026-05-26)

The spec predates the event-sourced write path, so three points were decided in chat:

1. **Caps + structure validation live in the Go executor (write-path gate), not the CLI.** `endless-event` validates structure and the >25 hard cap *before* taking the lock (refusal writes nothing); the >10 soft cap returns a warning the CLI prints. Per gates-not-guardrails.
2. **The collision read is done in Go, printed by the CLI just after.** The executor reads the prior revision inside the transaction and returns `{revised_at, session_id}`; the CLI prints "list last revised Xh ago by session N" (or "first revision"). Keeps DB reads in Go (E-894); a single CLI call has no abort window before BEGIN, so "just after" is equivalent.
3. **Lock-first ordering (mirrors the create path).** A new `events.BeginImmediate()` takes `BEGIN IMMEDIATE` *before* the events-authoritative JSONL ledger append, so a losing concurrent revise fails at BEGIN (SQLITE_BUSY) leaving no orphan ledger line and no audit row.

## Implementation summary

- `internal/events/event.go` — `KindProjectNextRevised` + `EntityProjectNext` (registered in `ValidKinds`/`validEntityTypes`).
- `internal/events/payload.go` — `ProjectNextRevisedPayload` (lanes/items; `pending_triage` ignored — hook-managed).
- `internal/events/project_next.go` — `ValidateProjectNextRevise` (caps gate), `execProjectNextRevised` (get-or-create header, capture prior revision, full lane/item rewrite, `change_kind='revise'` audit row), result structs.
- `internal/events/executor.go` — `BeginImmediate()` lock-first helper, `ExecuteResult.ProjectNext`, dispatch case.
- `cmd/endless-event/main.go` — `runProjectNextRevise` (validate-before-lock → BEGIN IMMEDIATE → ledger append → execute → COMMIT → emit `{ts,kind,prior_revision,warning,state}`).
- `src/endless/cli.py` — `task next` is now a group (`invoke_without_command=True`, bare command unchanged) hosting `revise --file [--project] [--json]`.
- `src/endless/task_cmd.py` — `revise_next_list` + `_format_relative`. `--json` keeps stdout pure JSON (advisory lines to stderr).
- Tests: `internal/events/project_next_test.go` (executor, validation/caps, BEGIN IMMEDIATE conflict), `tests/test_task_next_revise.py` (CLI wiring). No schema change (E-1434 added the tables).
