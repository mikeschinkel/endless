# W2 — Matchers in Config (revised)

> Part of E-968 (parallel-session support). See `.endless/plans/E-968.md` for full context. Obsoletes E-757.
>
> **Revised after design discussion**: original plan put phrases in a SQL table. Concluded matchers don't need to be in the DB at all (no JOIN consumers), and putting them in config files solves several problems at once: rebuild-db can't lose user-added matchers, git tracks per-project learning, and machine-level config accumulates cross-project knowledge.

## Why

Several places in endless hardcode lists of strings: title-verb whitelist in `task_cmd.py` (existing E-757 proposal), action regexes in `cmd/endless-hook/claude.go`, etc. The behavioral gate in W3 (E-971) needs another such list (pivot triggers). Centralize all of these into a single config-driven structure.

## Approach: matchers in config files (not DB)

Matchers live in JSON config files at two layers:

- **Project layer (default destination for writes)**: `<project>/.endless/config.json`, under a `matchers` property. Committed with the project so per-project learning persists in git history.
- **Machine layer (mirror of all project additions)**: `~/.config/endless/config.json`, under the same `matchers` property. Accumulates everything from every project so new projects benefit from past learning without re-discovery.

**Read order**: project + machine merged additively per (type, scope) pair. No override semantics in v1; if someone later wants per-project disable of a machine matcher, add a feature then.

**Write order**: every `endless phrase add` writes to BOTH project (current cwd's `.endless/config.json`) AND machine (`~/.config/endless/config.json`). De-duplicate within each file. If invoked outside a project, only machine is written.

## Config schema

```json
{
  "matchers": [
    { "type": "verb",      "method": "exact",     "match": ["add", "fix", "build", ...] },
    { "type": "pivot",     "method": "substring", "match": ["actually", "wait", "by the way", ...] },
    { "type": "pivot",     "method": "substring", "case_sensitive": true, "match": ["PIVOT"] },
    { "type": "start",     "scope": "task", "method": "regex", "match": "endless\\s+task\\s+start\\s+(\\d+)" },
    { "type": "complete",  "scope": "task", "method": "regex", "match": "endless\\s+task\\s+complete\\s+(\\d+)" },
    { "type": "chat",      "scope": "task", "method": "regex", "match": "endless\\s+task\\s+chat" },
    { "type": "beacon",    "scope": "channel", "method": "regex", "match": "endless\\s+channel\\s+beacon" },
    { "type": "connect",   "scope": "channel", "method": "regex", "match": "endless\\s+channel\\s+connect\\s+(\\S+)" },
    { "type": "send",      "scope": "channel", "method": "regex", "match": "endless\\s+channel\\s+send" }
  ]
}
```

### Field semantics

- **`type`** (required): the kind of matcher (`verb`, `pivot`, `start`, `complete`, `chat`, `beacon`, `connect`, `send`, ...). Reusable across scopes.
- **`scope`** (optional): qualifier within a type. `start`/`complete`/`chat` are scoped to `task`; `beacon`/`connect`/`send` are scoped to `channel`. Verbs and pivots have no scope.
- **`method`** (required): the matching algorithm.
  - **`exact`**: input string equals one of the `match` values. Consumer extracts the relevant token; matcher does set membership.
  - **`substring`**: input contains any `match` value as substring.
  - **`regex`**: input matches the regex pattern (single string).
- **`match`** (required): list of strings for `exact`/`substring`; single regex string for `regex`.
- **`case_sensitive`** (optional, default `false`): match case exactly.
- **`enabled`** (optional, default `true`): set to `false` to silence a matcher without removing it.

## CLI

- `endless phrase add <type> <value> [--scope <s>] [--method exact|substring|regex] [--case-sensitive] [--machine-only]`
  - Defaults: writes to project AND machine. `--machine-only` skips the project write (useful when working outside a project, or for personal preferences you don't want in any project's git history).
  - `--method` defaults: `exact` for `verb`, `substring` for `pivot`, `regex` for any other type (heuristic; user can override).
- `endless phrase list [--type <t>] [--scope <s>] [--all]`
  - `--all` includes disabled matchers; default shows enabled only.
- `endless phrase disable <selector>` / `endless phrase enable <selector>`
  - Selector format: `<type>[:<scope>]:<value>` (e.g., `verb:fix`, `start:task:endless\\s+task\\s+start\\s+(\\d+)`).
  - Sets `enabled: false` on the matching entry in the appropriate config file.
- `endless phrase remove <selector>`
  - Deletes the matching entry from the project config (and optionally machine with `--machine-only` flag).

## Seed strategy

On first endless run, if no `matchers` property exists in `~/.config/endless/config.json`, write the default seeds (current `_TITLE_VERBS`, hardcoded action regexes from `claude.go`, plus the seeded pivot list from the W3 design). Idempotent: subsequent runs detect the property and don't re-seed.

## Critical files

- new: `src/endless/matchers.py` — load/save config, lookup helpers (`get_verbs()`, `get_pivots()`, `get_action_regexes()`, etc.).
- new: `src/endless/phrase_cmd.py` — CLI implementations.
- `src/endless/cli.py` — register `phrase` command group.
- `src/endless/task_cmd.py` — `validate_title` reads verbs from `matchers.get_verbs()` instead of `_TITLE_VERBS`. Remove `_TITLE_VERBS` constant.
- new: `internal/matchers/matchers.go` — Go-side load/lookup matching the Python format.
- `cmd/endless-hook/claude.go` — `handlePostToolUse` loads action regexes via `matchers.GetActionRegexes()` instead of package-level `MustCompile` vars.
- `src/endless/config.py` and equivalent Go config layer — read/write the `matchers` property of `config.json`.

## Schema impact

**Zero**. No DB changes. The `phrases` table proposed in the original plan is no longer needed.

## Verification

- `endless phrase add verb foo` writes `foo` to both project and machine `config.json`. Both files re-read produce the entry.
- `endless task add "Foo bar"` succeeds (verb `foo` recognized).
- Manually deleting `foo` from project config but leaving in machine: `endless task add "Foo bar"` still works (machine layer fallback).
- `endless phrase add pivot "experimental" --machine-only` writes to machine only; project's `.endless/config.json` unchanged.
- `endless phrase disable verb:fix` sets `enabled: false`. `endless task add "Fix bug"` is rejected. `endless phrase enable verb:fix` restores.
- Action regex flow: `endless phrase add start --scope task --method regex --match "endless task begin (\\d+)"` adds an alternate start pattern. Hook sees `endless task begin 5` in bash output and fires the start handler.

## What we're explicitly not doing

- DB-based matchers (revisited and rejected; see top note).
- Per-project disable of machine-level matchers (deferred; YAGNI).
- Live file watching (load-once-per-process is sufficient given hook-process lifetime).
- Event-sourcing matcher mutations (file changes are tracked by git, which is the right history mechanism for config).
