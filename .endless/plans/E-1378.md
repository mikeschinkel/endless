# E-1378: Extract decisions from the tasks table — Go-side foundation

## Context

Decisions are stored as `tasks` rows with `type='decision'`. The shared table means task-only operations get misapplied to decisions: worktree auto-create on `task update --text`, `task confirm` / `task assume` on items where verify-then-confirm has no meaning, lifecycle states (`needs_plan`, `verify`) that don't map to decisions at all. Observed 2026-05-16 with E-1373 (worktree created just to hold a plan file; `confirmed` used where the intent was `completed`), prompting band-aid gates E-1375 and E-1376 (both now obsolete, superseded here).

This task lands the **Go-side foundation**: the storage shape, the data-movement change file, the event vocabulary, and the projector / `rebuild-db` updates that keep clones consistent. The user-facing CLI surface (`endless decision *` refit, dispatchers, `--decision` flag rewrite, `WHERE type != 'decision'` cleanup, web reads) ships in **E-1507**, which is `blocked_by` this task and branches off `task/1378-extract-decisions-tasks-table-into`.

## Schema

### `decisions` table (added to `schema.sql`)

```sql
CREATE TABLE IF NOT EXISTS decisions (
    id                INTEGER PRIMARY KEY,
    project_id        INTEGER NOT NULL,
    title             TEXT NOT NULL,
    description       TEXT,
    text              TEXT,
    status            TEXT NOT NULL DEFAULT 'proposed',
    origin_task_id    INTEGER,
    origin_session_id INTEGER,
    notes             TEXT,
    rejection_reason  TEXT,
    created_at        TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%S', 'now')),
    updated_at        TEXT NOT NULL DEFAULT '',
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
    FOREIGN KEY (origin_task_id) REFERENCES tasks(id) ON DELETE SET NULL,
    FOREIGN KEY (origin_session_id) REFERENCES sessions(id) ON DELETE SET NULL
);

CREATE TRIGGER IF NOT EXISTS decisions_updated_at AFTER UPDATE ON decisions
BEGIN
    UPDATE decisions SET updated_at = strftime('%Y-%m-%dT%H:%M:%S', 'now')
    WHERE id = NEW.id AND updated_at != strftime('%Y-%m-%dT%H:%M:%S', 'now');
END;
```

Column rationale:

- `title` — the decision statement itself (declarative; verb-first validation skipped per E-931).
- `description` — short rationale / context.
- `text` — long-form body for heavier decisions (e.g. E-1199 today has ~3 paragraphs).
- `status` — `proposed` | `accepted` | `rejected`; defaults to `proposed` so Claude-initiated creation paths require explicit user acceptance. Validation enforced in Python and Go (no CHECK constraints; schema.sql line 11-13 forbids them).
- `origin_task_id` — the task being worked on when the decision was recorded.
- `origin_session_id` — audit trail; which session created this record. New pattern (tasks don't have this column today); justified by the events-as-WAL posture from E-1327.
- `notes` — escape hatch for agent-readable extra context that doesn't fit the other fields.
- `rejection_reason` — populated only when status transitions to `rejected`.

Explicitly NOT included: `phase`, `parent_id`, `prompt`, `source_file`, `sort_order`, `tier`, `focus_id`, `outcome`, `analysis`, `completed_at` — all task-shaped concerns.

### `decision_relations` table (added to `schema.sql`)

```sql
CREATE TABLE IF NOT EXISTS decision_relations (
    id                 INTEGER PRIMARY KEY,
    source_decision_id INTEGER NOT NULL,
    target_kind        TEXT NOT NULL,
    target_id          INTEGER NOT NULL,
    relation_type      TEXT NOT NULL,
    created_at         TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%S', 'now')),
    UNIQUE (source_decision_id, target_kind, target_id, relation_type),
    FOREIGN KEY (source_decision_id) REFERENCES decisions(id) ON DELETE CASCADE
);
CREATE INDEX IF NOT EXISTS idx_decision_relations_target
    ON decision_relations (target_kind, target_id);
```

Source-table mapping: rows where the **source is a decision** live here. Task-sourced rows (task→task, task→decision) stay in `task_deps`. E-1389 will later rename `task_deps` → `task_relations` and reshape its columns.

No auto-creation of inverse rows. Reciprocal backfill (a future watcher) is out of scope.

## ID namespace: ED-NNN (local-only)

Per the two-letter prefix convention established by E-1031 (`EF-NNN` for focus) and anticipated by E-1032 (`ET-NNN` for tasks) and E-1261 (`ES-NNN` for sessions), decisions adopt **`ED-NNN`** as their local-ID prefix. Local-only per E-823's paired identifier scheme; global content-hash IDs come from E-816 when that lands.

- `decisions.id` is its own auto-increment column. Integer collisions with `tasks.id` are possible and harmless because the display prefix disambiguates.
- Existing decision IDs are preserved during the data move (no renumber). The decisions table's auto-increment seeds at `MAX(existing decision id) + 1`.
- `internal/eventcmd` emit prints `ED-<n>` for `decision.created` results so the Python CLI gets the prefixed ID to display.

## How this lands — schema.sql + change file (per E-1472)

E-1472 settled the schema-evolution mechanism: `schema.sql` is the single source of truth for declarative shape (idempotent CREATE IF NOT EXISTS on every DB connect); destructive one-off changes live in `internal/schema/changes/<e-NNN-slug>.{sql,go}`, applied once at land via `endless db apply-change`, keyed by basename in `_schema_version(name, applied_at)`. The Justfile `land` recipe (~lines 105-148) is the hook: it diffs `main...HEAD` for added files under `internal/schema/changes/`, runs `endless db backup`, applies each, then runs `endless worktree land`.

### `schema.sql` — declarative additions

Adds the two `CREATE TABLE IF NOT EXISTS` blocks above plus the `decisions_updated_at` trigger and the `idx_decision_relations_target` index. Fresh installs and the `schema.sql` baseline pass on every DB connect both pick them up.

### `internal/schema/changes/e-1378-extract-decisions.sql` — one-shot data move

Runs once at land, wrapped in `BEGIN IMMEDIATE` by the change-file dispatcher, marker recorded after success. Operations:

1. Copy existing `type='decision'` rows from `tasks` into `decisions`, with status mapping:
   - `confirmed` | `completed` | `assumed` → `accepted`
   - `needs_plan` | `ready` → `proposed`
   - Anything else → `accepted` (best-effort fallback)
2. Move decision-sourced rows from `task_deps` to `decision_relations`. Source-decision identification: `EXISTS (SELECT 1 FROM tasks WHERE id = source_id AND type = 'decision')`. `target_kind` derived from the target row's `tasks.type`.
3. Delete the just-moved rows from `task_deps`.
4. Update remaining `task_deps` rows where target is now a decision: set `target_type='decision'` (today it's hardcoded `'task'`).
5. Delete the migrated `type='decision'` rows from `tasks`.

Idempotency: not required on its own (the marker gates re-runs). Tests build from `schema.sql` + can apply the change file against a populated fixture.

### Events package — executor + projector

`schema.sql` and the change file fix the *state* of the live DB at land. The events-as-WAL design demands the *ledger* also represents this state correctly so `rebuild-db` (which runs on every clone) produces the same projection. Two pieces:

- **Executor** (`internal/events/decision.go`): new event kinds — `decision.created`, `decision.fields_updated`, `decision.accepted`, `decision.rejected`, `decision.deleted`, `decision_relation.created`, `decision_relation.deleted`. Each exec function writes to `decisions` or `decision_relations` directly. Status set `{proposed, accepted, rejected}` and target_kind set `{task, decision}` validated in Go.
- **Projector** (`internal/events/projector.go` + `decision.go`):
  - Replay functions for the 7 decision kinds, mirroring the executor logic.
  - **Legacy routing:** `replayTaskCreated` routes payloads with `type='decision'` to `replayLegacyDecisionCreated`, applying the same status mapping the change file uses. `replayTaskStatusChanged`, `replayTaskFieldsUpdated`, `replayTaskDeleted` consult `isDecisionID` and update `decisions` instead of `tasks` when the entity is a legacy decision. Only the fields that exist on decisions are applied; task-only fields drop.
- **`runRebuildDB --confirm`** (`internal/eventcmd/event.go`): replaces `decisions` and `decision_relations` from the projection alongside `tasks`. FK cascade clears `decision_relations` when `decisions` rows are deleted; INSERT brings projected rows back.

Net: `rebuild-db` is safe to run post-land. Replay produces the same shape the change file establishes, and `runRebuildDB` syncs the live DB with that shape.

## Open questions / deferred

- **Reciprocal backfill watcher.** Not in this task. Recording `task X implements decision Y` does not auto-create the reciprocal `decision Y implemented_by task X`; querying "all relations on entity X" requires a union across `task_deps` and `decision_relations`. File a follow-on if usage warrants.
- **`accepted_at` / `rejected_at` columns.** Skipped for now; `status` + `updated_at` cover it.
