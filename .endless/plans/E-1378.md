# E-1378: Extract decisions from tasks table into a dedicated decisions table

## Context

Decisions are stored as `tasks` rows with `type='decision'`. The shared table means task-only operations get misapplied to decisions: worktree auto-create on `task update --text`, `task confirm` / `task assume` on items where verify-then-confirm has no meaning, lifecycle states (`needs_plan`, `verify`) that don't map to decisions at all. Observed 2026-05-16 with E-1373 (worktree created just to hold a plan file; `confirmed` used where the intent was `completed`), prompting band-aid gates E-1375 and E-1376 (now obsolete, superseded by this task).

The endless guide already pre-announces this migration in `docs/guide/decisions.md`: *"decisions are currently stored in the `tasks` table with `type=decision` and will move to their own table in a future release. The CLI surface (`endless decision ...`) is stable; the underlying storage is implementation detail you can ignore today."* This task delivers on that pre-announcement.

## Scope

**In scope:**

- New `decisions` table with decision-shaped schema (no phase, no worktree, no verify state).
- New `decision_relations` table for decision-sourced edges.
- New decision lifecycle: `proposed` → `accepted` | `rejected` (3-state set; `proposed` initial, the other two terminal).
- New CLI verbs: `decision accept`, `decision reject`.
- Refit existing `decision *` commands to new storage.
- Refit `task link` and `decision link` as target-kind-agnostic dispatchers.
- Refit `task add --decision` and `task update --decision` to use new storage with `status=proposed`.
- Schema additions to `schema.sql` (declarative) + one-shot data movement via change file at land.
- Updated executor and projector so writes route to new tables AND rebuild-from-ledger reproduces the projection.
- ED-NNN local-ID convention (per E-1031/E-1032/E-1261 two-letter prefix scheme).
- Echo-format output for link/unlink (`• Linked: Decision ED-42 documents Task E-1199`).

**Out of scope (filed as separate tasks):**

- E-1383 — Team-decisioning state model (multi-participant flow).
- E-1389 — Rename `task_deps` → `task_relations` with `target_kind` column refactor.
- E-1400 — Allow `task spawn` from the session currently claiming the task (transfer binding).
- E-816 — Paired local/global content-hash identifiers (decisions get the global half when this lands).
- E-1157 — `clarifies` / `reaffirms` decision-link types (still deferred per YAGNI).

## Schema

### `decisions` table (added to `schema.sql`)

```sql
CREATE TABLE IF NOT EXISTS decisions (
    id                INTEGER PRIMARY KEY,
    project_id        INTEGER NOT NULL,
    title             TEXT NOT NULL,
    description       TEXT,
    text              TEXT,
    status            TEXT NOT NULL DEFAULT 'proposed',
    origin_task_id    INTEGER,
    origin_session_id INTEGER,
    notes             TEXT,
    rejection_reason  TEXT,
    created_at        TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%S', 'now')),
    updated_at        TEXT NOT NULL DEFAULT '',
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
    FOREIGN KEY (origin_task_id) REFERENCES tasks(id) ON DELETE SET NULL,
    FOREIGN KEY (origin_session_id) REFERENCES sessions(id) ON DELETE SET NULL
);

CREATE TRIGGER IF NOT EXISTS decisions_updated_at AFTER UPDATE ON decisions
BEGIN
    UPDATE decisions SET updated_at = strftime('%Y-%m-%dT%H:%M:%S', 'now')
    WHERE id = NEW.id AND updated_at != strftime('%Y-%m-%dT%H:%M:%S', 'now');
END;
```

Column rationale:

- `title` — the decision statement itself (declarative; verb-first validation skipped per E-931).
- `description` — short rationale / context (2–3 sentence pitch).
- `text` — long-form body for heavier decisions (e.g. E-1199 today has ~3 paragraphs).
- `status` — `proposed` | `accepted` | `rejected`; defaults to `proposed` so Claude-initiated creation paths require explicit user acceptance. Validation enforced in Python and Go (no CHECK constraints; schema.sql line 3 forbids them).
- `origin_task_id` — the task being worked on when the decision was recorded.
- `origin_session_id` — audit trail; which session created this record.
- `notes` — escape hatch for agent-readable extra context that doesn't fit the other fields.
- `rejection_reason` — populated only when status transitions to `rejected`.

Explicitly NOT included: `phase`, `parent_id`, `prompt`, `source_file`, `sort_order`, `tier`, `focus_id`, `outcome`, `analysis`, `completed_at` — all task-shaped concerns.

### `decision_relations` table (added to `schema.sql`)

```sql
CREATE TABLE IF NOT EXISTS decision_relations (
    id                 INTEGER PRIMARY KEY,
    source_decision_id INTEGER NOT NULL,
    target_kind        TEXT NOT NULL,
    target_id          INTEGER NOT NULL,
    relation_type      TEXT NOT NULL,
    created_at         TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%S', 'now')),
    UNIQUE (source_decision_id, target_kind, target_id, relation_type),
    FOREIGN KEY (source_decision_id) REFERENCES decisions(id) ON DELETE CASCADE
);
CREATE INDEX IF NOT EXISTS idx_decision_relations_target
    ON decision_relations (target_kind, target_id);
```

Source-table mapping: rows where the **source is a decision** live here. Task-sourced rows (task→task, task→decision) stay in `task_deps`. E-1389 will later rename `task_deps` → `task_relations` and reshape its columns.

**No auto-creation of inverse rows.** Recording `task E-908 implements decision ED-42` creates one row in `task_deps`; the reciprocal `decision ED-42 implemented_by task E-908` exists only if separately recorded. A future watcher (no task filed; out of scope) may backfill reciprocals.

### Relation-type vocabulary by pair

| Pair                | Legal `relation_type` values                                                |
|---------------------|-----------------------------------------------------------------------------|
| decision → task     | `documents`, `cleans_up_by`, `implemented_by`, `relates_to`                 |
| decision → decision | `reverses`, `modifies`, `documents`, `relates_to` (+ future `clarifies`, `reaffirms`) |
| task → decision     | `implements`, `cleans_up`, `documents`, `relates_to`                        |
| task → task         | `blocks`, `implements`, `replaces`, `documents`, `cleans_up`, `relates_to` (unchanged) |

Each CLI dispatcher validates `relation_type` against the detected pair and errors with the legal set when an illegal type is used.

## ID namespace: ED-NNN (local-only)

Per the two-letter prefix convention established by E-1031 (`EF-NNN` for focus) and anticipated by E-1032 (`ET-NNN` for tasks) and E-1261 (`ES-NNN` for sessions), decisions adopt **`ED-NNN`** as their local-ID prefix. Local-only per E-823's paired identifier scheme; global content-hash IDs come from E-816 when that lands.

Implementation:

- `decisions.id` is its own auto-increment column. Integer collisions with `tasks.id` are possible and harmless because the display prefix disambiguates.
- Existing decision IDs are preserved during the data move (no renumber). The decisions table's auto-increment seeds at `MAX(existing decision id) + 1`, so the next NEW decision continues just past the last migrated one.
- A `DECISION_ID` Click type accepts `ED-42` or `42` as input, parallel to the existing `TASK_ID` type.
- Display formatting in CLI and web uses `ED-{id}` whenever a decision is shown.
- References to old `E-NNN` decision IDs in plan files, commit messages, and the JSONL ledger are NOT rewritten — cost outweighs benefit for cosmetic continuity.

## How this lands — schema.sql + change file (per E-1472)

E-1472 settled the schema-evolution mechanism: `schema.sql` is the single source of truth for declarative shape (idempotent CREATE IF NOT EXISTS on every DB connect); destructive one-off changes live in `internal/schema/changes/<e-NNN-slug>.{sql,go}`, applied once at land via `endless db apply-change`, keyed by basename in `_schema_version(name, applied_at)`. The Justfile `land` recipe (~lines 132-148) is the hook: it diffs `main...HEAD` for added files under `internal/schema/changes/`, runs `endless db backup`, applies each, then runs `endless worktree land`.

Three coordinated edits land together:

### 1. `schema.sql` — declarative additions

Add the two `CREATE TABLE IF NOT EXISTS` blocks above. Fresh installs and the `schema.sql` baseline pass on every DB connect both pick them up.

### 2. `internal/schema/changes/e-1378-extract-decisions.sql` — one-shot data move

Runs once at land, wrapped in `BEGIN IMMEDIATE` by the change-file dispatcher, marker recorded after success. Operations:

1. Copy existing `type='decision'` rows from `tasks` into `decisions`, with status mapping:
   - `confirmed` | `completed` | `assumed` → `accepted`
   - `needs_plan` | `ready` → `proposed`
   - Anything else → `accepted` (best-effort; audited against the real DB at implementation time).
2. Move decision-sourced rows from `task_deps` into `decision_relations`. Source-decision identification: join on `tasks WHERE type='decision'` (or on `decisions` if the change file has already run partially). `target_kind` derived from the target row's `tasks.type`.
3. Update `task_deps` rows where target is a decision: set `target_type='decision'` so the column reflects reality going forward (today it's hardcoded `'task'`).
4. Delete the migrated `type='decision'` rows from `tasks`.

Idempotency: not required on its own (the marker gates re-runs). Tests build from `schema.sql` + may apply the change file against a populated fixture.

### 3. Events package — executor + projector updates

`schema.sql` and the change file fix the *state* of the live DB at land. The events-as-WAL design demands the *ledger* also represents this state correctly so rebuild-from-ledger (which runs on every clone) produces the same projection. Two pieces:

- **Executor** (`internal/events/executor.go`): new writes use new event kinds — `decision.created`, `decision.fields_updated`, `decision.accepted`, `decision.rejected`, `decision.deleted`, `decision_relation.created`, `decision_relation.deleted`. Existing `task.created` with `type='decision'` is deprecated for new writes; the Python CLI emits the decision-kind events instead via `event_bridge.emit_event`.
- **Projector** (`internal/events/projector.go`): replay recognizes both the new `decision.*` kinds AND legacy `task.created` events whose payload contains `type='decision'`. Legacy events route to the `decisions` table; the existing `task_dep` events are routed to `decision_relations` when the source row's type was `decision`. This keeps clones consistent with the live DB without rewriting the ledger.

Both pieces ship in the same branch as the schema/change-file edits so a clone never sees a half-state.

## CLI surface — per-verb audit

### `endless decision *`

| Verb              | Current                                                                      | Change                                                                                                                       |
|-------------------|------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------|
| `decision list`   | Queries `tasks WHERE type='decision'` via `list_decisions()` in task_cmd.py  | Query `decisions` table directly. Same renderer shape (id, title, status, created).                                          |
| `decision add`    | Calls `add_item(..., task_type='decision', status='confirmed')`             | Emit `decision.created`; status default `'proposed'`. `--about` writes `documents` row to `decision_relations`. `--decides` writes `implements` row to `task_deps` (source=task). |
| `decision show`   | Delegates to `detail_item()` (task renderer)                                | New decision-shaped renderer: title, status, description, text, origin task, relations, created/updated. No phase, no worktree, no verify state. |
| `decision link`   | dec↔dec only via `require_decision_pair`                                    | Dispatcher: target can be task or decision. Validates `relation_type` is legal for the detected pair. Writes to `decision_relations`. |
| `decision unlink` | Mirror                                                                       | Mirror of new `decision link`.                                                                                              |
| `decision accept` | **NEW**                                                                      | Flips `proposed` → `accepted`. Errors clearly if status is already `accepted` or `rejected`.                                 |
| `decision reject` | **NEW**                                                                      | Flips `proposed` → `rejected` with required `--reason "..."`. Stores reason in `rejection_reason`.                          |

### `endless task *` changes

| Verb                            | Current                                                              | Change                                                                                                                       |
|---------------------------------|----------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------|
| `task add --decision "..."`     | Creates paired `type=decision` task; links via `documents`           | Create row in `decisions` with `status='proposed'`. Link from new decision to the parent task via `decision_relations` row, `relation_type='documents'`. |
| `task update --decision "..."`  | Same                                                                  | Same as above (task already exists; only the decision is created).                                                          |
| `task link`                     | Writes to `task_deps`; both ends assumed tasks                       | Dispatcher: target can be task or decision. For task→decision, write to `task_deps` with `target_type='decision'`. For task→task, unchanged. Validates type vocabulary per pair. |
| `task unlink`                   | Mirror                                                               | Mirror.                                                                                                                     |
| `task confirm` / `task assume`  | Sets status                                                          | No change at the verb level. Decisions are no longer in `tasks`, so `task confirm ED-42` resolves to "no task with id 42" (via the `DECISION_ID` parser routing). The error message redirects to `decision accept`. |

The five `WHERE t.type != 'decision'` filters in `src/endless/task_cmd.py` (lines ~904, 1066, 1278, 1367, 3246) and the one `WHERE t.type = 'decision'` for `list_decisions` (line ~1454) become no-ops after the data move and should be removed. The Go web queries similarly.

### Echo-format output (link/unlink)

```
• Linked: Decision ED-42 documents Task E-1199
• Linked: Task E-908 implements Decision ED-42
• Linked: Decision ED-42 reverses Decision ED-89
• Linked: Task E-908 blocks Task E-1199            (unchanged for task↔task)
• Unlinked: Decision ED-42 documents Decision ED-89
```

Bullet prefix retained for visual parity with existing CLI; kind labels (`Task` / `Decision`) make direction self-documenting on read-back. Symmetric for `task link`, `task unlink`, `decision link`, `decision unlink`.

## Verification

1. `endless decision list` returns the same decisions as before, with `ED-NNN` IDs.
2. `endless task list`, `task next`, `task recent`, `task active` no longer show any decisions (and the `type != 'decision'` filters are gone from the queries).
3. `endless decision add "Test decision"` creates with `status='proposed'`. `endless decision accept ED-<id>` flips to `accepted`. `endless decision reject ED-<id> --reason "..."` flips to `rejected` and stores the reason.
4. `endless decision link ED-42 --to E-1199 --type documents` writes a row to `decision_relations` and prints `• Linked: Decision ED-42 documents Task E-1199`.
5. `endless task link E-908 --to ED-42 --type implements` writes a row to `task_deps` with `target_type='decision'` and prints `• Linked: Task E-908 implements Decision ED-42`.
6. `endless task add "Some task" --decision "Why we chose X"` creates the task in `tasks` AND a paired row in `decisions` with `status='proposed'`, linked via a `documents` row in `decision_relations`.
7. **Land hook applies the change file:** `just land E-1378` runs `endless db backup`, then `endless db apply-change <wt>/internal/schema/changes/e-1378-extract-decisions.sql`, then proceeds with the worktree land. Verify the change file marker lands in `_schema_version`.
8. **Rebuild-from-ledger parity:** `endless-event rebuild-db /tmp/scratch-rebuild.db` produces a projection whose `decisions` and `decision_relations` counts match the live DB after the change file ran. This is the critical proof that the executor + projector updates keep clones consistent.
9. Type validation works: `endless decision link ED-42 --to E-1199 --type blocks` errors with "`blocks` is not legal for decision→task; legal types: documents, cleans_up_by, implemented_by, relates_to."
10. `endless task confirm ED-42` errors with a clear redirect: `Decisions don't have a confirm verb; use 'decision accept ED-42'.`
