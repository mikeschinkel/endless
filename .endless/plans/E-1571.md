# E-1571 — Add sessions.active_epic_id column and kind discriminator for bg-agent rows

Lands after E-1565 / E-1566 (per-type templates, not directly dependent) and **before** E-1568 (--bg dispatch writes these columns) and E-1570 (attach verbs may read kind to discriminate fg vs bg).

## Decisions

> **Decision update (2026-06-16, Mike):** `kind` is stored as an **int FK enum**
> mirroring the TaskType pattern (E-1538 / ED-1506), NOT a free TEXT column. The
> original draft's `kind TEXT` is superseded. Column is `kind_id INTEGER
> REFERENCES session_kinds(id) NOT NULL DEFAULT 1`; a seeded `session_kinds`
> mirror table lives in `schema.sql`; a new `internal/sessionkind` package holds
> the Go enum (`SessionKind` int) with `String/Label/Parse/Validate/All/VerifyIntegrity`,
> and `VerifyIntegrity` is wired into `monitor.DB()` startup alongside the
> task_types check. This keeps enum storage consistent with the house rule
> ("FK to a values table seeded by schema.sql, no CHECK").

### Schema additions
Two new columns on `sessions`:

- `active_epic_id INTEGER` — nullable FK to `tasks(id)`. Semantics:
  - **Coordinator (epic) fg row:** stable = the epic's task id; `active_task_id` mutates as user switches view to children.
  - **Bg-agent row:** `active_task_id` = the child task id; `active_epic_id` = parent epic id (resolved at spawn time by Python).
  - **Non-epic fg row:** NULL.
- `kind_id INTEGER NOT NULL DEFAULT 1 REFERENCES session_kinds(id)` — discriminator: `tmux` (1) or `background` (2). Int FK enum (see decision update above). Backfill: all existing rows take the default `1` (tmux).

`process` column (the tmux pane id) is already nullable in the current schema. No structural change there — semantics shift: `kind='background'` rows legitimately leave it NULL. Document the invariant in a comment.

### Window-name / status-line rendering
Status-line `[E-NNN]` prefix in `internal/tmuxcmd/status_line.go::format()` (line 69) becomes:

- `active_epic_id IS NULL` → `[E-NNNN]` (today's behavior, unchanged)
- `active_epic_id IS NOT NULL AND active_epic_id = active_task_id` → `[E-EEEE]` (viewing the epic itself)
- `active_epic_id IS NOT NULL AND active_epic_id != active_task_id` → `[E-EEEE:E-CCCC]` (viewing a child of the epic)

Same logic applied to `internal/tmuxcmd/show_menu.go::format()` (line 75) for the popup header.

`monitor.ActiveTaskInfo` (the struct the format functions read) gains `ActiveEpicID *int64`. The query that populates it joins `sessions` (it already does — that's where `active_task_id` comes from) and selects `active_epic_id`.

### Coordinator row vs child-view: how active_task_id moves
Out of scope here — this child only adds the columns. The mechanism for "user switches focus to a child" is part of E-1570 (attach verbs) and the day-to-day `endless task claim` flow. This child does NOT implement focus-switching; it only ensures the data shape exists.

### "Focused bg agent" derivation
Per E-1552 outcome ("derived from the coordinator row, not stored"), no column added. Queries that need it compute it: `SELECT * FROM sessions WHERE kind_id = <background> AND active_task_id = <coordinator's active_task_id>` — i.e. the bg agent whose child task matches what the coordinator is viewing. This child adds a Go helper `monitor.FocusedBgAgent(coordinatorSessionID) (*SessionInfo, error)` so callers don't reinvent the SQL.

### Validation
`internal/sessionkind.Validate(slug)` rejects any kind outside the enum, mirroring `tasktype.Validate` (E-1538). The DB-level backstop is the `kind_id` FK to `session_kinds`. No SQL CHECK. `sessionkind.VerifyIntegrity(db)` runs at `monitor.DB()` startup (guarded by `hasTable(db, "session_kinds")`, like the task_types check) and fails closed on drift between the Go enum and the seeded table.

Startup integrity check (drift *warning*, separate from the fail-closed enum check) is **deferred** — no `kind='background'` rows exist until E-1568 writes them, so a `process IS NOT NULL` vs `kind_id` consistency probe has nothing to inspect yet. Folded into E-1568 where the write path that can violate it is introduced.

  ```sql
  CREATE TABLE IF NOT EXISTS session_kinds (
      id INTEGER PRIMARY KEY, slug TEXT UNIQUE NOT NULL, label TEXT NOT NULL);
  INSERT OR IGNORE INTO session_kinds (id, slug, label) VALUES
      (1, 'tmux', 'Tmux'), (2, 'background', 'Background');
  ALTER TABLE sessions ADD COLUMN active_epic_id INTEGER REFERENCES tasks(id);
  ALTER TABLE sessions ADD COLUMN kind_id INTEGER NOT NULL DEFAULT 1 REFERENCES session_kinds(id);
  -- No backfill UPDATE needed: existing rows take the default 1 (tmux).
  ```
