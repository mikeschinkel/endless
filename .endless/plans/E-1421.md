# Curated per-project "next" list with lanes, locks, and consistency hooks

## Motivation

Across many parallel sessions, the working "what's next for this project" decision keeps getting re-derived from scratch — each fresh session re-reads tasks, re-ranks, re-explains. Mike asks "what's next" repeatedly because the answer isn't persisted. The current `endless task next` is heuristic and not authoritative; it cannot capture the parallelization analysis a curating session produces.

This task adds a **curated, persistent, per-project "next" list** as a first-class artifact. Sessions revise it deliberately; Mike reads it via `endless task next`; consistency updates (auto-add on urgent, render-side terminal-status filter) keep it honest without manual maintenance.

## Confirmed design decisions

- **Authoritative, not heuristic.** Empty list = empty list, not "fall back to heuristic." The heuristic view becomes a separate command/flag.
- **Per-project.** Schema keyed by project; UX scoped to one project at a time. Cross-project ranking is out of scope.
- **Soft cap 5-10 items**, hard cap 25. Warning on revise if >10, refusal at >25.
- **Lanes, not strict 1-2-3 ranking.** Items grouped into lanes; within-lane order is sequential, across-lane is parallel-safe. Lane carries its own rationale.
- **Pending-triage bucket.** Auto-added urgent tasks land here unranked. Curation moves them into ranked lanes during the next revise.
- **Render-side filter for terminal status.** Storage holds `[task_id, lane, reason]` triples; reading joins against `tasks` and hides items with terminal status. Race-free by construction. Each revise also compacts the storage for hygiene.
- **Auto-lock around revise**, no explicit lock command. Lock auto-acquired at revise start, auto-released at end, with TTL (10 min) for crash recovery.
- **Only user has `unlock`.** Claude has no way to release a lock it didn't create; the auto-release on revise completion is the only Claude-driven unlock.
- **JSON, array-of-structs (no maps).** Order intrinsic in arrays; deterministic by structure, not by helper type.
- **Per-task `reason` (~60-80 chars)**, per-lane `rationale` (longer). Both required.
- **Briefing command outputs text by default, JSON via `--llm`.**

## JSON schema (grouped-by-lane)

```json
{
  "project": "endless",
  "last_revised": "2026-05-18T14:30:00Z",
  "revised_by_session": 468,
  "lanes": [
    {
      "id": "resolver",
      "rationale": "Stop silent session-id misattribution; foundational",
      "items": [
        {"task_id": "E-1415", "reason": "DB existence check; loud failure when companion stale"}
      ]
    },
    {
      "id": "worktree-land-serial",
      "rationale": "All touch worktree land code; serialize to avoid merge conflicts",
      "items": [
        {"task_id": "E-1337", "reason": "Archive-and-reap; deepest fix for worktree-deleted-under-session"},
        {"task_id": "E-1416", "reason": "Pre-rebase dirty-tree check; safety net"}
      ]
    }
  ],
  "pending_triage": [
    {"task_id": "E-1404", "reason": "auto-added: phase=urgent", "added_at": "2026-05-17T10:00:00Z"}
  ]
}
```

## CLI surface

**Mike-facing (read):**
- `endless task next` — render the curated list, grouped by lane, with `last_revised` timestamp prominently displayed. Pending-triage section rendered after ranked lanes.
- `endless task next --heuristic` — the old heuristic view (today's `task next` behavior).
- `endless task next unlock` — force-release any lock. User-only.

**Claude-facing (write):**
- `endless task next briefing` — sitrep for revision decisions. Text default, `--llm` adds JSON. Fields:
  - Current list (with lanes, last_revised, drift indicators)
  - Recently changed tasks (last 24h)
  - In-progress tasks across sessions + claiming session
  - Tasks in `verify` (candidates to confirm-and-drop)
  - Blockers that just resolved (candidates to add)
  - Worktree states (dirty trees, branches behind main)
  - Drift: list items whose status changed since last revision
- `endless task next revise --file <json>` — full rewrite. Auto-acquires lock, validates schema, compacts terminal-status items, writes, releases lock. Prints "list last revised Xh ago by session N" before applying so collisions are visible.
- `endless task next move <task_id> --to <lane-id-or-position>` — single-item nudge. Auto-locks for the duration of the call.

## Lock model

| Operation | Lock behavior |
|---|---|
| `revise` | Auto-acquire at start, auto-release at end. TTL 10 min for crash recovery. |
| `move` | Auto-acquire for duration of call. |
| `briefing` | No lock (read-only). |
| `task next` (read) | No lock (read-only). |
| Auto-add on urgent | Bypasses lock (consistency). |
| Auto-drop (compaction) | Bypasses lock (consistency). |
| `unlock` (user) | Force-releases any lock. Claude has no equivalent. |

## Hook integration

Three event subscribers maintain consistency:

1. **`task.created` with `phase=urgent`** → append `{task_id, reason: "auto-added: phase=urgent", added_at}` to `pending_triage`. Idempotent (no-op if already on list).
2. **`task.phase_changed` to `urgent`** → same as above.
3. **`task.status_changed` to terminal** → no subscriber needed. Render-side filter handles visibility; compaction in `revise` handles storage.

All three bypass the lock. Triggered from the task-mutation code path, not from external hooks (single point of truth).

## Open questions for Mike

1. **Storage location**: DB table `project_next_lists(project, json_blob, last_revised, locked_by_session, locked_until)` vs file at `<main>/.endless/next-list.json`. DB has the better story for concurrent access and joins for render-side filter. Proposal: DB.
2. **Lane id format**: free-text slug (e.g. `worktree-land-serial`) vs typed enum. Free-text is more flexible; typed reduces drift. Proposal: free-text, but `briefing` highlights when a lane appears for the first time so naming drift is visible.
3. **`move --to <position>` semantics**: within current lane only, or cross-lane? If cross-lane, how is the target lane named? Proposal: `move --to <lane-id>:<position>` or `--to <lane-id>` (appends to lane).
4. **What happens to `pending_triage` items when Mike confirms a curated revision that didn't move them out?** Proposal: stay where they are; only revise can remove them from pending.
