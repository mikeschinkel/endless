# Curated per-project "next" list with lanes and consistency hooks

## Motivation

Across many parallel sessions, the working "what's next for this project" decision keeps getting re-derived from scratch — each fresh session re-reads tasks, re-ranks, re-explains. Mike asks "what's next" repeatedly because the answer isn't persisted. The current `endless task next` is heuristic and not authoritative; it cannot capture the parallelization analysis a curating session produces.

This task adds a **curated, persistent, per-project "next" list** as a first-class artifact. Sessions revise it deliberately; Mike reads it via `endless task next`; consistency updates (auto-add on urgent, render-side terminal-status filter) keep it honest without manual maintenance.

## Confirmed design decisions

- **Authoritative, not heuristic.** Empty list = empty list, not "fall back to heuristic." The heuristic view is exposed via `--inferred`.
- **Per-project.** Schema keyed by project; UX scoped to one project at a time. Cross-project ranking is out of scope.
- **Soft cap 5-10 items**, hard cap 25. Warning on revise if >10, refusal at >25.
- **Two-axis model: priority + lane.** Each lane carries an integer `priority` (lower = sooner) AND a free-text `id`. Multiple lanes may share a priority — that means "parallel-safe at this tier." Within a lane: items are sequential. Across lanes at the same priority: items are parallel-safe. Lane carries its own rationale.
- **Pending-triage bucket.** Auto-added urgent tasks land here unranked. Curation moves them into ranked lanes during the next revise.
- **Render-side filter for terminal status.** Render joins against `tasks` and hides items with terminal status. Race-free by construction. Storage compaction is deferred (YAGNI).
- **Concurrency via `BEGIN IMMEDIATE TRANSACTION`.** `revise` and `move` wrap their multi-row work in a SQLite immediate transaction; the loser of a concurrent attempt gets `SQLITE_BUSY` and can surface the conflict. No explicit lock columns, no TTL, no `unlock` command. Auto-add and reads are unconstrained.
- **JSON, array-of-structs (no maps).** Order intrinsic in arrays.
- **Per-task `reason` (~60-80 chars)**, per-lane `rationale` (longer). Both required.
- **Output convention follows existing endless commands**: default = human-readable text; `--llm` = terse agent-friendly text; `--json` = structured JSON.

## SQL schema (5 tables)

Modeled storage, not JSON blob. Per house rules: every table starts with `id INTEGER PRIMARY KEY`; FK columns use the `_id` suffix; multi-segment names are avoided by table-name disambiguation.

```sql
-- One row per project (header + FK target)
CREATE TABLE project_next (
  id INTEGER PRIMARY KEY,
  project TEXT NOT NULL UNIQUE
);

-- Lanes within a project's curated list
CREATE TABLE project_next_lanes (
  id INTEGER PRIMARY KEY,
  project_next_id INTEGER NOT NULL REFERENCES project_next(id) ON DELETE CASCADE,
  lane_id TEXT NOT NULL,            -- free-text slug
  priority INTEGER NOT NULL,
  rationale TEXT NOT NULL,
  UNIQUE(project_next_id, lane_id)
);

-- Items within a lane, ordered by position
CREATE TABLE project_next_items (
  id INTEGER PRIMARY KEY,
  project_next_lane_id INTEGER NOT NULL REFERENCES project_next_lanes(id) ON DELETE CASCADE,
  task_id TEXT NOT NULL,
  reason TEXT NOT NULL,
  position INTEGER NOT NULL,
  UNIQUE(project_next_lane_id, task_id),
  UNIQUE(project_next_lane_id, position)
);

-- Auto-added urgent tasks awaiting curation
CREATE TABLE project_next_pending (
  id INTEGER PRIMARY KEY,
  project_next_id INTEGER NOT NULL REFERENCES project_next(id) ON DELETE CASCADE,
  task_id TEXT NOT NULL,
  reason TEXT NOT NULL,
  added_at TIMESTAMP NOT NULL,
  UNIQUE(project_next_id, task_id)
);

-- Audit trail of revisions (in-scope per design decision)
CREATE TABLE project_next_revisions (
  id INTEGER PRIMARY KEY,
  project_next_id INTEGER NOT NULL REFERENCES project_next(id),
  session_id INTEGER NOT NULL REFERENCES sessions(id),
  revised_at TIMESTAMP NOT NULL,
  change_kind TEXT NOT NULL,        -- 'revise' | 'move' | 'auto-add'
  json_snapshot TEXT
);
```

"Who revised last" and "when" derive from `project_next_revisions` (MAX `revised_at`, ORDER BY), not denormalized columns.

## CLI surface

**Mike-facing (read):**
- `endless task next` — render the curated list, grouped by lane, with `last_revised` displayed. Pending-triage section after ranked lanes.
- `endless task next --inferred` — the heuristic view (today's `task next` behavior). Non-authoritative; derived from task state.

**Claude-facing (write):**
- `endless task next briefing` — sitrep for revision decisions. Text default (verbose dashboard), `--llm` for terse agent text, `--json` for structured JSON. Fields:
  - Current list (with lanes, last_revised, drift indicators)
  - Recently changed tasks (last 24h)
  - In-progress tasks across sessions + claiming session
  - Tasks in `verify` (candidates to confirm-and-drop)
  - Blockers that just resolved (candidates to add)
  - Worktree states (dirty trees, branches behind main)
  - Drift: list items whose status changed since last revision
- `endless task next revise --file <json>` — full rewrite. Wraps the multi-row replace in `BEGIN IMMEDIATE TRANSACTION`. Validates schema (lanes, priorities, items, soft-cap warning at >10, refusal at >25). Inserts a row in `project_next_revisions` with `change_kind='revise'`. Reads `last_revised` + last revisor before BEGIN and prints "list last revised Xh ago by session N" so cross-session collisions are visible.
- `endless task next move <task_id> --to <lane-id>[:<position>]` — single-item nudge. `--to <lane-id>:<position>` places at a specific index; `--to <lane-id>` alone appends to the lane. Cross-lane supported (destination's priority inherited). Wraps the position updates in `BEGIN IMMEDIATE TRANSACTION`. Inserts a `change_kind='move'` audit row.

## Concurrency model

| Operation | SQL pattern | Concurrency |
|---|---|---|
| `revise` | `BEGIN IMMEDIATE`; delete lanes+items (CASCADE), insert new ones; insert audit row; `COMMIT` | RESERVED lock for the duration; loser gets `SQLITE_BUSY` |
| `move` | `BEGIN IMMEDIATE`; update item positions (and lane FK if cross-lane); insert audit row; `COMMIT` | Same as above |
| Auto-add on urgent | `INSERT INTO project_next_pending ... ON CONFLICT DO NOTHING` | No transaction needed |
| Reads (render, briefing) | `SELECT` joining `tasks` with terminal-status filter | No blocking |

Crash mid-transaction → SQLite rolls back automatically. No stuck state, no TTL, no `unlock` command needed. If experience shows we need explicit lock state (e.g. for human-visible diagnostics across processes), add `project_next_lock` table later.

## Hook integration

Two event subscribers maintain consistency:

1. **`task.created` with `phase=urgent`** → `INSERT INTO project_next_pending(...) ON CONFLICT DO NOTHING`. Idempotent.
2. **`task.phase_changed` to `urgent`** → same as above.

Terminal-status changes need no subscriber — render-side filter handles visibility.

Triggered from the task-mutation code path, not from external hooks (single point of truth).

## Resolved (formerly open questions)

1. **Storage**: modeled (5 tables), not blob. Per the schema above. Concurrency via `BEGIN IMMEDIATE TRANSACTION` rather than explicit lock columns.
2. **Lane id format**: free-text slug. `briefing` highlights first-time-seen slugs so naming drift is visible.
3. **`move --to` syntax**: `--to <lane-id>:<position>` or `--to <lane-id>` (append). Cross-lane supported.
4. **`pending_triage` lifecycle after revise**: items stay until explicitly moved or until their task hits terminal status (render-side filter hides them). No special handling needed.
5. **Compaction of storage**: deferred (YAGNI). Render-side filter keeps reads correct.
6. **Explicit lock columns / `unlock` command / TTL**: dropped in favor of `BEGIN IMMEDIATE TRANSACTION`. Revisit if a real edge case needs visible lock state.
