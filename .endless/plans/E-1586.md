# E-1586 — PreToolUse cwd gate: block edits outside a claimed worktree

## Problem

`endless task claim <id>` creates and binds a fresh, live worktree, but Claude
Code's **harness cwd** still starts at the project root. The shell-level moves
the guide documents (`cd "$(endless worktree for-task)"`, `esu`) only change a
Bash/interactive cwd — they don't move the harness cwd that `Write`/`Edit` and a
fresh `Bash` resolve against. So between `claim` and the agent running `/cd`,
every mutating tool use silently lands in **main's working tree** instead of the
isolated worktree. The agent must hold per-tool "directory discipline" to avoid
it. `/cd` (newly shipped) lets the agent move the harness cwd once — this gate
makes doing so non-optional.

## Why the gate lives in the hook, not in `claim`/`bind`

- A CLI subprocess can't move its parent harness cwd, so `claim` can only *emit*
  a `/cd` directive (a guardrail, ignorable) — see the claim-output nudge folded
  into E-1428.
- `bind` must **not** auto-`/cd`. Its canonical target is an already
  `assumed`/`confirmed`/`verify` task; any worktree it coincides with is
  **landed and reap-pending**, so `/cd`-ing in re-creates the stale-cwd footgun
  of E-1280/E-1337. bind is also used to *watch* tasks other sessions own —
  `/cd`-ing into a sibling's live worktree (two sessions, one worktree) is worse.
- The hook is the only layer with the runtime context to decide: it sees the
  payload `cwd`, and can resolve worktree liveness + ownership.

## Gate condition (the one unambiguous signal is `claim`-ownership)

Block the tool when **all** hold:

1. The session **owns a live, claimed worktree** (claim-ownership, not mere
   display-bind; the worktree dir exists and is not landed/reap-pending).
2. The harness `cwd` (from the hook payload) is **outside** that worktree.
3. The tool is **mutating**: `Write`, `Edit`, `NotebookEdit`, or a `Bash`
   command that writes (conservatively: any `Bash`, or detect write intent —
   decide during impl; start with Write/Edit/NotebookEdit, evaluate Bash).

Block message (no internal task IDs per house rule):

> You've claimed this task's worktree but your working directory is still
> `<displayPath(cwd)>`. Run `/cd <displayPath(worktree)>` first so your edits
> land in the worktree, not main.

Use the `displayPath` helper (cwd-relative or `~/…`, never raw absolute).

## Explicitly NOT gated (carve-outs)

- **Reads anywhere** (`Read`/`Grep`/`Glob`) — never blocked; cross-checkout
  reading (inspecting main, comparing) is legitimate.
- **Display-only / done-ish / others' / worktree-less binds** — no claim
  ownership of a live worktree → condition (1) is false → never trips.
- **Legitimate project-root work while unclaimed** — `esp`, `git pull`,
  coordinator sessions. No claimed live worktree → no gate.
- **The `claim` call itself and the moment right after** — claim runs from the
  project root; the gate only fires on the *first mutating tool*, which is
  exactly when `/cd` should already have happened. No special grace logic
  needed.

## Relationship to existing gates

Sibling of E-1012 (block git commits on `main`). E-1012 stops the catastrophic
outcome (a commit to main); E-1586 closes the narrower residual — uncommitted
edits to main's working tree before the agent `/cd`s.
