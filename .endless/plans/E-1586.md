# E-1586 — PreToolUse cwd gate: force /cd into your worktree before acting

## Problem

`endless task claim <id>` (and, once redesigned, `bind`) gives the session a
live, owned worktree, but Claude Code's **harness cwd** still starts at the
project root. The shell moves the guide documents (`cd "$(endless worktree
for-task)"`, `esu`/`eswt`) only change a Bash/interactive cwd — not the harness
cwd that `Write`/`Edit` and a fresh `Bash` resolve against. So between claim and
the agent running `/cd`, tool uses land in **main's working tree**. `/cd` (newly
shipped) lets the agent move the harness cwd once; this gate makes doing so
non-optional.

## Posture: default-deny, then allowlist (decided with Mike)

Do NOT start by gating only Write/Edit and hoping we spot the leaks. **Lock down
everything** while the session owns a live worktree but cwd is outside it, then
**allowlist** what's proven safe. This way every tool is consciously classified,
not assumed.

### Gate condition

Deny the tool when **all** hold:

1. The session owns a **live worktree** — resolved from its `active_task_id`
   (set by `claim`, and by `bind` once the bind redesign lands); the worktree
   dir exists and is not landed/reap-pending.
2. The harness `cwd` (from the hook payload) is **outside** that worktree.
3. The tool is **not** in the read allowlist.

### Read allowlist (seed)

`Read`, `Grep`, `Glob` (+ `LS`/`NotebookRead` if present). Cross-checkout
*reading* (inspecting main, comparing) stays legitimate. Everything else —
`Write`, `Edit`, `NotebookEdit`, **and `Bash`** — is denied until `/cd`. Bash is
denied by default on purpose; carve out specific read-only Bash patterns later,
consciously, if friction warrants.

### Block message (no internal task IDs per house rule)

> You've entered this task's worktree but your working directory is still
> `<displayPath(cwd)>`. Run `/cd <displayPath(worktree)>` first so your work
> lands in the worktree, not main.

Use the `displayPath` helper (cwd-relative or `~/…`, never raw absolute).

### Why a hook, not the `claim`/`bind` commands

A CLI subprocess can't move its parent harness cwd — it can only *emit* a `/cd`
directive (the claim-output nudge). The hook is the only layer that sees the
runtime `cwd` + worktree liveness/ownership and can actually gate.

## Staleness is a `bind`-time concern, NOT a reason to skip `/cd`

(Resolved with Mike — supersedes the earlier "bind worktrees are reap-pending"
argument, which was wrong.) The just-landed-then-found-a-bug case is bind's
reason to exist: retain the worktree, bind back, `/cd` in, fix.

- No new commits in main beyond the worktree → not stale.
- New commits in main → **fast-forward the worktree branch from main**. Post-land
  the worktree branch is an ancestor of main, so FF is clean and always applies →
  refreshed, not stale.
- FF only fails if the worktree already has commits main lacks (you started the
  fix before refreshing) → real divergence, rebase; should not happen in the
  normal post-land/pre-work state.

So staleness is handled by FF-or-recreate **at bind time**; it never argues
against `/cd`.

## Adjacent work — `bind` redesign (SEPARATE task, see Open question)

Out of scope for E-1586's hook gate, but the gate keys on "owns a live worktree"
specifically so it covers `bind` automatically once bind:
1. ensures a worktree exists after binding (creating it if absent),
2. fast-forwards an existing worktree from main (recreates if FF fails),
3. **fails if a sibling session has the task bound/claimed**,
4. emits the same `/cd` directive claim does.

bind stays a *status-preserving* claim (does not demote
confirmed/assumed/verify → in_progress, which `claim --force` would).
