# E-1309: Route ledger auto-commits to project main checkout, not caller cwd

## Problem

Ledger auto-commits ("Endless: record ledger entry") periodically land on a worktree's task branch instead of on `main`, accumulating commits that conflict on rebase during `endless worktree land`. Two confirmed instances on 2026-05-14:

- **E-1329** (worktree session, sibling sessions emitting concurrently). Diagnostic: `/tmp/e-1309-e-1329.md`.
- **E-1333** (this session). Bug fired during `endless task add E-1333` + `endless task claim E-1333`, both run from `cwd=/Users/mikeschinkel/Projects/endless` (main) with `GIT_DIR` and related git-locating env vars unset in the parent shell. Rogue commit `bd9e796` grouped 3 events from session 426 (mine) and landed on `task/1333-...`.

The first instance suggested the parent shell's `GIT_DIR` was leaking into the git subprocess. The second instance disproves "shell env leak" as the sole mechanism — the bug also fires from a clean main shell with no relevant env vars set.

## Mechanism analysis

What we know:

- `event_bridge.emit_event` correctly resolves `project_root` to the projects-table `path` (main checkout). Verified for both instances.
- `endless-event` is invoked with `--project-root <main>`; the JSONL file is written into main's working tree at the correct path.
- `internal/events/commit.go::runGit` builds `exec.Command("git", ...)` with no explicit `cmd.Env`, so the subprocess inherits whatever env the parent has.
- `git -C <projectRoot>` is overridable by `GIT_DIR`/`GIT_WORK_TREE`/`GIT_INDEX_FILE`/`GIT_OBJECT_DIRECTORY`/`GIT_COMMON_DIR`/`GIT_NAMESPACE`/`GIT_ALTERNATE_OBJECT_DIRECTORIES`.
- In `task_cmd.py::_perform_claim_work` (lines 2097-2164), event emission happens **before** `create_task_worktree`. So at emit-time, the per-task worktree doesn't exist yet — there's no obvious worktree-gitdir for `GIT_DIR` to point at, even hypothetically.
- The same vulnerability (subprocess.run without env=) exists in Python `commit_project_verb` (`src/endless/matchers.py:913-950`) for `verbs.jsonl` auto-commits.

What we don't know:

- The actual env that `endless-event`'s git subprocess sees in the failing case. We've inspected the parent Python's shell env (clean) but not the env at the `exec.Command` site inside the Go binary.
- Whether some Python or Go code in the call chain mutates `os.environ` / sets env vars on the subprocess.
- Whether the E-1281 sandbox setup (newly enabled on this project via `worktree_sandbox: true`) interacts with the auto-commit path in a way that misdirects git.

The fix below works regardless of which mechanism is responsible: stripping git-locating env vars closes the GIT_DIR vector; the worktree-detection guard refuses any commit that would land outside the main checkout regardless of how git was misdirected; the debug log surfaces the actual env+args so the next occurrence reveals the source directly.

## Fix

### 1. Sanitize git env in `internal/events/commit.go::runGit` (and `runGitOutput`)

Build `cmd.Env` from `os.Environ()` with the following variables stripped before invoking git:

- `GIT_DIR`
- `GIT_WORK_TREE`
- `GIT_INDEX_FILE`
- `GIT_OBJECT_DIRECTORY`
- `GIT_COMMON_DIR`
- `GIT_NAMESPACE`
- `GIT_ALTERNATE_OBJECT_DIRECTORIES`

After this, `git -C <projectRoot>` becomes authoritative against env-based redirection. Fixes both `CommitLedgerSegment` and `CommitSnapshotPair` since they share `runGit`.

### 2. Add a worktree-detection guard to `commitPaths`

Before staging, run `git -C projectRoot rev-parse --git-dir --git-common-dir` with the sanitized env. If the two values differ, the resolved git repo is a linked worktree, not main. Error loudly with a message naming the projectRoot, the resolved git-dir, the resolved common-dir, and which paths were about to be committed.

This is the actual catch-all: regardless of how git got steered toward a worktree (env, cwd, internal state, anything), the guard refuses to commit. The env sanitization in #1 makes it a clean miss most of the time; the guard makes it observable when something else is in play.

### 3. Optional debug logging

Behind `ENDLESS_DEBUG_GIT=1` env var, log to stderr:

- The full args passed to git.
- The subset of env vars set: `cwd`, `GIT_*`, `XDG_*`, `PATH` (just first entry).
- The git-dir and common-dir resolved at runtime.

Off by default. When the next "Endless: record ledger entry" rogue commit happens, Mike can rerun the failing operation with `ENDLESS_DEBUG_GIT=1` and the source becomes visible.

### 4. Mirror the env fix in Python `commit_project_verb` (`src/endless/matchers.py`)

`commit_project_verb` shells to git via `subprocess.run`. Pass an explicit `env=` with the same `GIT_*` variables stripped. Same vulnerability, same fix.

### 5. Tests

- **Go unit:** in a test that sets `GIT_DIR=<bogus-path>` in the test process env, call `runGit(<real-repo>, "rev-parse", "--git-dir")` and assert the result is `<real-repo>/.git`, not the bogus path.
- **Go integration:** create a temp repo, `git worktree add` a linked worktree, set `GIT_DIR=<worktree-gitdir>` in env, call `CommitLedgerSegment(<main-root>, "<file>")`, assert the commit landed on main's branch (`git -C <main> log -1 --format=%H`) and not on the worktree's branch.
- **Go unit (guard):** call `commitPaths` with `projectRoot = <a worktree path>` and assert it returns the worktree-detection error without committing.
- **Python:** parallel unit test for `commit_project_verb`.

## Loose ends / follow-ups

- **Identify the actual mechanism.** Step 3's debug log is built to make the next occurrence self-explanatory. File a follow-up task (`--cleans-up E-1309`) once we have one captured debug log to read.
- **E-1329 worktree cleanup.** Already documented at `/tmp/e-1329-fix.md`. Awaits this fix landing.
- **`emit_event` cwd fallback.** `event_bridge.emit_event` falls back to `Path.cwd()` if the project isn't in the projects table. Defensible but masks bugs. File a follow-up to remove it; not load-bearing for this fix.
- **E-1281 sandbox interaction.** When sandboxing is active for an endless self-dev worktree, ledger writes are routed to the sandbox DB and don't touch main's tree at all — so the auto-commit path doesn't fire for them. The env-sanitization in #1 is still correct in that world; the guard in #2 must not trip on the legitimate sandbox path. Verify behavior in both modes during testing; add a code comment noting the sandbox-DB case is structurally separate.
