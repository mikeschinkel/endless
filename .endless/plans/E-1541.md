# E-1541 — Implement auto-derivation of epic status from child states

## 1. The derivation rule

Epic status is a pure function of the children's statuses, except when the epic itself is in a sticky-override status. Computed top-to-bottom:

| Children's state | Epic status |
|---|---|
| Any child has `status = in_progress` | `in_progress` |
| No `in_progress` but any child has `status = ready` | `ready` |
| No `in_progress` or `ready` and any child has `status = needs_plan` | `needs_plan` |
| All children are terminal (`completed`/`confirmed`/`assumed`/`declined`/`obsolete`) AND at least one child exists | `completed` |
| Epic has zero children | unchanged (no derivation) |

**Sticky-override statuses** block derivation: `revisit`, `declined`, `obsolete`, `blocked`. When the epic is in one of these, derivation reads the current state and does nothing. The override must be cleared (manually set the epic to a derivable status) before derivation resumes.

The "add a non-terminal child to a completed epic reverts it" case from E-1537 §4 is no longer a special case — it falls out naturally: adding a `needs_plan` child to a completed epic makes the epic `needs_plan` again per the rule.

E-1537's §4 will be updated to use this rule.

## 2. Where derivation fires

The recompute fires inline, inside the Go SQL transaction that handles the triggering child mutation. Callers:

- `execTaskStatusChanged` — after the SQL update, before commit
- `execTaskFieldsUpdated` — only when `Fields["status"]` or `Fields["parent_id"]` is present
- `execTaskMoved` — both the old and new parent chains need recompute
- `execTaskCreated` / `execTaskImported` — only when `ParentID != nil`
- `execTaskDeleted` — recompute the deleted task's parent chain

Same calls in the projector counterparts (`replayTaskStatusChanged`, etc.) so `rebuild-db` reproduces derivation.

## 3. Where the code lives

New file `internal/events/epic_derivation.go` exporting:

```go
func recomputeEpicStatus(db dbQuerier, evt *Event, changedTaskID int64) error
```

Algorithm:

1. Walk ancestors of `changedTaskID` via `WITH RECURSIVE ancestors`, filtering to rows where `type_id = TaskTypeEpic`. Depth-cap at 32.
2. For each epic in order:
   - Read current `status` and direct children's `status`.
   - If current status is in the sticky-override set → skip.
   - Compute the target status per §1.
   - If `target != current`, emit a derived `epic.status_derived` event with actor `{Kind: ActorSystem, ID: "epic-derivation"}` and execute the SQL update via the same transaction. The derived event then triggers recompute for the next-level epic ancestor.

The derivation event is written through the executor's existing event writer (segment + projector), threaded as a `derivedEmitter func(Event) error` parameter into `recomputeEpicStatus`. The projector passes a no-op emitter (segments already exist; replay only updates the DB).

A `seenEpicIDs map[int64]bool` is threaded through the recursion as a belt-and-suspenders cycle guard.

## 4. New event kind

`epic.status_derived` joins the closed event-kind vocabulary in `internal/events/event.go`. Schema mirrors `task.status_changed`: `task_id`, `old_status`, `new_status`, `actor`. The projector dispatcher gets a new `replayEpicStatusDerived` that applies the SQL update without re-running derivation (since the live path already emitted any cascading events).

Decision rationale: separate event kind, not reusing `task.status_changed`. Filterability is the main benefit (audit log of "what derivation did" stays distinct from human-triggered status changes). Projector complexity is bounded — one new dispatch case.

## 5. Idempotency

- Equality short-circuit: if `target == current`, emit nothing.
- Replay safety: a duplicate derivation event from the live path is re-applied by `replayEpicStatusDerived` as the same SQL update — idempotent at the DB level.
- Single transaction: derivation runs inside the trigger's `BEGIN IMMEDIATE`. A retry of the whole transaction (SQLite busy) recomputes from scratch.
- The `OldStatus` in the derived event is the epic's status at the moment of recompute, not at the moment of the trigger.

## 6. Cycle detection

Two layers:
- The `WITH RECURSIVE` ancestor walk has a depth cap of 32. `parent_id` cycles are already prevented at write time by `execTaskMoved`'s ancestor-loop check, but the cap is cheap insurance.
- The `seenEpicIDs` set prevents re-visiting an epic within a single derivation pass.

## 7. Tests

New `internal/events/epic_derivation_test.go`:

- Table-driven cases for §1's rule:
  - All needs_plan children → epic needs_plan
  - Mix of needs_plan + ready → epic ready
  - Any in_progress → epic in_progress
  - All terminal → epic completed
  - Mix of ready + terminal → epic ready
  - Mix of needs_plan + terminal → epic needs_plan
- Reopen case: adding a needs_plan child to a completed epic → epic flips back to needs_plan
- Sticky-override states (revisit/declined/obsolete/blocked) on the epic block derivation
- Nested: grand-child status flip propagates up two levels via derived events
- Cycle: synthetic parent-loop hits depth cap without panic
- Idempotency: replaying a derivation event yields the same final state
- Actor attribution: derived events have `actor.kind = system`, `actor.id = epic-derivation`, no `session_id`
- End-to-end via Python `endless task confirm <child>`: assert parent epic's status flipped

## 8. Implementation sequence

1. Add `epic.status_derived` to the closed event-kind vocabulary (`internal/events/event.go`).
2. Add `replayEpicStatusDerived` to the projector dispatcher (`internal/events/projector.go`).
3. Create `internal/events/epic_derivation.go` with `recomputeEpicStatus`.
4. Extend `events.Execute` to thread the `derivedEmitter` from `eventcmd.run`.
5. Wire calls into the five executor entry points (one line + an if-guard each).
6. Wire equivalent (no-op emitter) calls into the projector counterparts.
7. Tests per §7.
8. Smoke-test against the actual endless DB by creating a synthetic epic + children in a `--db sandbox` and walking through state transitions.

No Python changes. No schema migration (uses existing `tasks.type_id` from E-1538). No new tables.
