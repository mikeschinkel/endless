# Implement `endless session next` (the read command)

Build the real Go command behind the throwaway bash/sqlite prototype at `~/.config/endless/session-next.sql` + `session-next.sh`. **That prototype is the authoritative spec** for the query, columns, sorting, and rendering — read it first. The Go command supersedes and deletes both.

## Architecture
- `endless session next` lives in the Python `session` command group, delegating the read to **Go (endless-go)** per the reads-in-Go rule (E-1486): the Python verb shells out to an endless-go subcommand and prints its output. (Confirm the existing Python→endless-go delegation pattern, e.g. `session-query`.)
- **Focal-task resolution** — do NOT re-derive pane→session logic; compose the existing Go resolvers, matching claude.go's documented priority (*session active task > `@endless_task_id`*):
  1. `monitor.GetActiveTaskForPane($TMUX_PANE)` (ED-1523) — live session's `active_task` via process match (state != 'ended').
  2. **Fallback** when that returns nothing — which is COMMON: active windows are often state `needs_input` with NULL `active_task`, and bound sessions can be `ended` with empty `process`, so the process-based resolver finds neither. Fall back to the window's `@endless_task_id` via `monitor.tmuxTaskID` (claude.go:972; reads it with `tmux display-message -p "#{@endless_task_id}"`).
  3. Global most-recent only as a last resort.
  The bash prototype gets this wrong precisely because it can't call these Go functions — the real command must. (This extends ED-1523, which named `GetActiveTaskForPane` as canonical, with the `@endless_task_id` fallback for null-active windows.)

## Row set (per the prototype)
For focal task T (= the window session's `active_task_id` via GetActiveTaskForPane):
- `session_tasks` for ALL sessions whose `active_task_id = T` (cross-project; dup-session-proof — matches `WHERE active_task_id=T`)
- ∪ the focal task T
- ∪ the parent session's focal task (parent = this window's `@endless_spawned_by` source pane → its session)

Omit done-work (terminal status) EXCEPT the focal and parent rows, which always show. `--all` includes done-work.

## Columns (left→right), single-spaced
`<action-icon> <type-letter> <E-id> <phase-char> <block> <title-fills-width>`
- **action icon**: `●` this · `↑` parent · `⟳` doing (another live session on it) · `▶` do (ready + has text) · `✎` plan (unplanned/needs_plan) · `☑` verify · `◷` orphan (in_progress, no live session) · `·` other. (No `⏸` as an action.)
- **type letter**: E/T/B/R/Z from `task_types.slug`
- **phase char**: `!` urgent, `1` now, `2` next, `3` later, `?` maybe; `✓` when the task is terminal/done (focal/parent only)
- **block column** (auto-width 0/1/2): `⊗` blocked-by (any open blocker), `⏸` blocks (this blocks others). Width 0 if nothing blocked; 1 if no single task needs both; 2 only when some task is both.
- **title**: fills remaining terminal width (detect term size like `task list` does); truncate.

Use **display-width-aware rendering** (runewidth) — the prototype fights byte-vs-display width in `printf`; Go must pad by display width. Phase-by-intensity color via **lipgloss** is fine in Go (the prototype reverted raw ANSI as too fragile for bash).

## Sort
action-rank (this, parent, doing, do, plan, verify, orphan, other), then phase-rank (urgent<now<next<later<maybe), then id.

## Status model
`ready` = approved-to-spawn, plan optional (ED-1522); `do` (`▶`) surfaces ready tasks regardless of plan. Canonicalize status in-query (unplanned/needs_plan→plan; verify/unverified→verify; underway/in_progress→working; terminal = confirmed/assumed/declined/obsolete/completed).

## Out of scope (separate tasks)
- `--watch` / TUI — E-859 / E-1622 (lipgloss/bubbletea).
- `--tree` order view (E-1684) + session ordering (E-1683).
- goto/back navigation (E-1681).
- Duplicate session rows (E-1640) — resolver is dup-proof regardless.
